## Problems Hit

- **Rowan + Pratt parser conflict** ‚Äî Rowan's push model (`start_node` ‚Üí tokens ‚Üí `finish_node`) fundamentally conflicts with Pratt parsing, which needs to wrap already-emitted nodes in parent nodes (e.g., emitting `1`, then wrapping it in `BinaryExpr` when it sees `+`). The standard solution (used by rust-analyzer) is a two-phase event-based approach: collect parsing events into a `Vec<Event>`, then do a single pass to build the GreenNode. This is a significant architectural piece.

  **Resolution:** Deferred the Rowan-first parser to v0.3.0. For v0.2.0, we keep the existing direct-AST parser and add an AST ‚Üí Rowan round-trip (`ast_to_cst`) that proves the Rowan infrastructure works end-to-end. This unblocks LSP preparation without blocking v0.2.0's user-facing features.

- **String representation in codegen** ‚Äî Strings as `{ i8*, i64 }` fat pointers are 128 bits, but our uniform `i64` calling convention can't return them. We can't just change the convention because all existing integer code depends on it.

  **Resolution:** String codegen returns a pointer-to-data cast to `i64` as a placeholder in generic expression context. The real fat pointer is only constructed when calling `println`/`assert` shims, where we have specialized handling that knows to extract the struct fields. This is a hack but isolates the complexity to exactly the call sites that need it.

- **`lower()` didn't have access to the interner** ‚Äî The HIR lowerer needed to check "is this Ident symbol named 'println'?" but `lower()` only received `&Ast`, not `&Interner`. Symbols are opaque `u32` handles without the interner.

  **Resolution:** Changed `lower()` signature to `lower(ast: &Ast, interner: &Interner) -> Hir`. This is a breaking change that rippled through ~15 call sites (tests + `compile_to_ir`), but it's the correct design ‚Äî any pass that needs to interpret Symbol values needs the interner.

- **Runtime shim for integer-to-string conversion** ‚Äî `glyim_println_int` needs to convert `i64` to a decimal string using only LLVM IR (no libc `snprintf` because we said no FFI beyond builtins). This is ~60 instructions of `udiv`/`srem` loops, negative number handling, and buffer management.

  **Resolution:** Wrote it manually. It's tedious but correct: handles zero, positive numbers, and negatives (negate first, then convert). Uses a stack-allocated 21-byte buffer (enough for `i64::MIN` = "-9223372036854775808" which is 20 chars + null).

## Decisions Made

- [ADR-010] Strings are borrowed fat pointers `{ i8*, i64 }` ‚Äî aligns with Rust's `&str` mental model, no null terminator needed, length is O(1). [ADR-012] `println`/`assert` are compiler built-ins, not `@rust()` FFI ‚Äî 10x faster to implement, zero FFI complexity, runtime shims use `write`/`abort` as external POSIX functions only.

- [ADR-012] `glyim run` switches to JIT via Inkwell's `ExecutionEngine` ‚Äî removes `cc`/`gcc` dependency for the common case, enables future REPL, faster dev cycle. AOT path (`glyim build`) remains for production.

- [ADR-013] HIR gains `HirStmt` enum (`Let`, `Assign`, `Expr`) ‚Äî blocks change from `Vec<HirExpr>` to `Vec<HirStmt>`. The last `Expr` statement's value becomes the block's value. Matches Rust semantics exactly.

- [ADR-014] Rowan-first parser deferred to v0.3.0 ‚Äî the event-based two-phase architecture needed for Pratt + Rowan is too large for v0.2.0's scope. AST ‚Üí Rowan round-trip proves the infrastructure.

- [ADR-015] UI tests use `.g` + `.g.stderr` file pairs with `insta` for initial generation ‚Äî every diagnostic is a tested contract. Same pattern as rustc's `ui` test suite.

- [ADR-016] Error recovery uses sync points (`fn`, `let`, `}`, `EOF`) ‚Äî on error, emit diagnostic, insert error node, skip tokens, resume. Reports multiple errors per compile.
