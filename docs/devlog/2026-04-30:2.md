# 2026-04-30: Testing Infrastructure Overhaul & Regression Prevention

## What I Did

- **Comprehensive regression test expansion** — Added 66 new tests across three previously zero-coverage crates that form the core compiler pipeline. Wrote 7 unit tests for `glyim-diag` covering `Span` construction, validation, length, emptiness, and `into_source_span` conversion. Added ~33 unit tests for `glyim-typeck` exercising scope mechanics, struct/enum registration, expression inference (literals, identifiers, binary/unary/block/if), type error detection (unknown fields, missing fields, non-exhaustive matches, invalid casts, return type mismatches), and multiple-error accumulation. Added ~26 tests for `glyim-hir/lower` validating expression, statement, control flow, item, and sugar lowering, plus `ExprId` monotonicity. All tests use a `typecheck()` helper that runs parse→lower→typecheck end-to-end.

- **UI test harness repair** — Replaced the broken harness in `crates/glyim-cli/tests/ui.rs` that only ran parsing and often left snapshots empty. Now the harness executes the full pipeline: parse, lower to HIR, type check, and optionally IR generation, collecting errors from every phase. Filled all 6 previously empty snapshot files with real, color-stripped error output, ensuring that UI snapshot tests now catch regressions in error reporting across the entire front-end.

- **LLVM IR snapshot testing** — Added three `insta` snapshot tests (`return_42`, `let_and_add`, `if_else`) for the codegen crate. Each test compiles a small Glyim program to LLVM IR and snapshots the output. This provides automatic regression detection for future codegen changes without requiring full E2E pipeline tests.

- **Package manager edge‑case tests** — Created a new test module `manifest_edge_tests.rs` with 6 tests covering valid minimal manifests, dependency parsing, workspace-only manifests, and error cases (missing package name, invalid TOML, empty manifest). Extended existing resolver tests with conflicting constraint resolution and cyclic dependency handling. Extended lockfile tests with multiple-version roundtrips, sorting stability, and missing type field parsing.

- **Fuzzing infrastructure** — Added `libfuzzer` targets for the lexer and parser in a `fuzz/` directory (detached from the main workspace). The lexer fuzzer feeds arbitrary byte sequences to the tokenizer, asserting span validity and catching panics. The parser fuzzer ensures the parser never crashes on any input. Both targets smoke-tested with 1000 iterations without failures. Corresponding `justfile` recipes (`fuzz-lexer`, `fuzz-parser`, plus short smoke-test variants) were added.

- **Justfile modernization & advanced demo** — Rewrote `justfile` with correct indentation, integrated fuzz recipes, restored tier/DAG checks, and added a `demo-advanced` recipe that exercises structs, `Option`, `match`, and generic identity, demonstrating the current state of the compiler. The demo script was moved to a standalone `scripts/demo-advanced.sh` to avoid justfile heredoc indentation issues.

- **Pipeline fixes** — Removed a duplicate `tracing_subscriber::init()` call in the CLI that caused a "global default trace dispatcher has already been set" panic when invoking `glyim`. Fixed corrupted `Cargo.toml` files (duplicate dev-dependencies) and detached the fuzz crate from the workspace. Added `glyim-diag` as a dev‑dependency where needed and resolved multiple borrow/lifetime conflicts in the new test code.

## Problems Hit

- **Snapshot format mismatch** — The old UI harness produced ANSI-colored error output via `miette`. The new harness, running through a different code path, produced plain text. This caused all UI snapshot tests to fail with subtle differences.  
  **Resolution:** Ran `INSTA_UPDATE=always` to regenerate all snapshots at once, then verified no ANSI codes remained with `grep`.

- **Repeated duplicate definitions in test files** — Due to multiple rounds of appending code to existing test files, I accidentally re-added imports like `HashMap`, `LockSource`, and full helper functions such as `make_req`/`make_available` multiple times. This caused Rust's `E0252` and `E0428` errors. Multiple test functions were also duplicated.  
  **Resolution:** Fully overwrote the affected files (`resolver_tests.rs`, `lockfile_tests.rs`, `glyim-hir/lower/tests.rs`, `glyim-typeck/tests.rs`) with cleaned versions containing only the correct set of imports and tests.

- **Borrow conflict in test helpers** — The `get_main_body` function took the interner by immutable reference and then caller tried to call `intern()` on it while the body was still borrowed. Also, `interner.resolve(interner.intern("test"))` produced an immutable-then-mutable conflict.  
  **Resolution:** Added a lifetime parameter `'a` to `get_main_body` and restructured callers to intern symbols first, then call `get_main_body`, avoiding the conflict. Switched `lower_int_lit` to a two-step approach: intern first, then obtain the body.

- **`f64` prevents `Eq` derive on `HirType`** — While not a new problem, it surfaced again when writing tests that compared `HirExpr` values with `PartialEq`. The existing codebase already handles this correctly, but test assertions had to use `matches!` patterns instead of `assert_eq!` for types containing `f64`.

- **Inkwell type method resolution** — `SourceSpan` from `miette` does not have `start()` and `end()` methods as I initially assumed. I incorrectly wrote `ss.start()` and `ss.end()`.  
  **Resolution:** Switched to `ss.offset()` and `ss.len()` which are the correct public API for `miette::SourceSpan`.

- **Fuzz crate location conflict** — `cargo fuzz` expects a `fuzz/` directory at repository root. Placing it inside `crates/glyim-fuzz` caused "could not read the manifest file" errors even with `--manifest-path`.  
  **Resolution:** Moved the crate to `fuzz/` at the repo root and gave it its own `[workspace]` table to detach from the main workspace, avoiding workspace member resolution conflicts.

## Decisions Made

- **[ADR‑030] Full‑pipeline UI test harness** — UI tests now run parse, lower, and type check, not just parsing. This catches regressions in type error messages, exhaustiveness checks, and lowering, not just syntax errors. The cost is slightly slower tests, but they remain well under 100ms.

- **[ADR‑031] `insta` for LLVM IR snapshots** — Chose `insta` over manual file comparison for codegen IR tests to leverage auto‑updating snapshots and diff visualization. Each snapshot test lives alongside its `.g` source file for easy maintenance.

- **[ADR‑032] Fuzzing with `libfuzzer` and `arbitrary`** — Targets take arbitrary byte slices (`Vec<u8>`) and convert to strings lossily, ensuring the fuzzer explores a wide range of input including invalid UTF‑8. Both targets check for panics only, not correctness, which is appropriate for a first pass.

- **[ADR‑033] Smoke‑test fuzz targets in development** — Added `fuzz-*-short` justfile recipes with 1000 iterations for developer use. Full fuzz runs remain available via the unfiltered recipes for CI or dedicated sessions.

- **[ADR‑034] Standalone advanced demo script** — To work around justfile's strict indentation rules for heredocs, the advanced demo source was moved to `scripts/demo-advanced.sh`. The justfile recipe simply invokes the script. This keeps the justfile clean and avoids syntax errors.
