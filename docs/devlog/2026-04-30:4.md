# 2026-04-30: v0.5.0 Language Feature Completion – Monomorphization, Method Calls, While Loops, As Casts, &self, VecI64, and For‑In

## What I Did

- **Monomorphization Pipeline** — Completed the monomorphization pass for generic functions and structs. Fixed five distinct bugs: missing work queue seeding, positional type‑param binding, missing body substitution in `specialize_fn`, key collision in `fn_mangle_map`, and nested generic calls not being queued. Implemented a worklist algorithm with structural param‑to‑type‑argument mapping, deduplication (`fn_queued`), and proper mangling via `format_type_short`. Integrated monomorphization into the compiler pipeline after type checking, merging type overrides into the expression type table before codegen.

- **Three‑Pass Codegen** — Refactored `Codegen::generate` and `generate_for_tests` into three passes: (1) register struct/enums/externs, (2) forward‑declare all functions via `declare_fn`, (3) emit function bodies. This ensures that monomorphized functions are visible to callers even when they appear later in the HIR item list. Also removed a duplicate `add_function` call in `codegen_fn` that was creating conflicting function declarations.

- **Language Features** —
  - **`as` Cast Codegen** — Replaced the stub implementation with proper LLVM cast instructions (`sitofp`, `fptosi`, `inttoptr`, `ptrtoint`). Added `TypeExpr::Float`, `TypeExpr::Int`, etc. to the symbol mapping so that `42 as f64` produces the correct `HirType::Float` in lowering.
  - **`while` Loops** — Added lexing, parsing, AST/HIR nodes (`While`), lowering, and LLVM codegen using a standard three‑block pattern (`while.cond`, `while.body`, `while.end`). Leverages existing alloca‑based mutable variables, avoiding the need for PHI nodes.
  - **Method Call Desugaring** — The parser already recognized `obj.method(args)` and produced `ExprKind::MethodCall`. Lowering now correctly maps it to `HirExpr::MethodCall`. In codegen, the receiver type is used to construct a mangled function name (`TypeName_methodName`), and the method is called. The type checker now also checks impl methods by iterating `HirItem::Impl` and pushing individual methods into `fns`.
  - **`&self` / `&mut self` Parameters** — In impl blocks, `&self` and `&mut self` are now parsed by consuming the `&` token and an optional `mut`. The impl target type is propagated into the parameter list, so `&self` desugars to `self: TargetType` with the type provided by the impl block. The lexer was fixed to recognise `&` as `SyntaxKind::Amp` (previously it was lexed as `Error`).

- **VecI64 Standard Library Type** — Added a concrete `VecI64` struct with `data: *mut u8`, `len`, and `cap` fields. The impl block provides `new()` and `len(&self)`. An integration test (`e2e_veci64_impl`) verifies that the whole pipeline (parse, lower, typecheck, monomorphize, codegen, run) succeeds.

- **Integration Tests** — Added numerous tests to validate the new features: `e2e_generic_identity_call`, `e2e_mono_*` family, `e2e_while_loop`, `e2e_veci64_impl`. Un‑ignored and fixed `e2e_impl_method` (struct‑associated function call). Fixed `e2e_cast_int_to_float` after the parser change to `parse_type_expr`.

- **Bug Fixes & Polish** — Corrected the `SyntaxKind::COUNT` mismatch. Removed unused `resolve_type_name` function, unused imports, and debug prints. Fixed the `veci64_push_get` test to embed the stdlib source inline to avoid path issues.

## Problems Hit

- **Borrow checker and compilation errors in monomorphize** — The `MonoContext` had multiple borrow conflicts when building the mangle maps and iterating over `fn_specs`/`struct_specs`. Repeatedly re‑structured the `build_result` method to pre‑compute lookups and clone data before mutable operations.

- **`HirType` missing `Hash` and `Eq`** — Using `Vec<HirType>` as a map key required those derives, which were missing. Added them to the derive macro.

- **Three‑pass codegen not taking effect** — The `generate` method in `mod.rs` was still a single pass after multiple attempts. The Python replacement code didn't match the exact text in the file. Finally overwritten manually.

- **Lexer missing `&` token** — `&` was not listed in the single‑character token match, causing the parser to see `Error` tokens. Added `'&' => SyntaxKind::Amp`.

- **`&self` parsing broken** — The initial implementation tried to handle `&self` in `parse_fn_def_with_attrs` but never received the target type from `parse_impl_block`. Passed an optional `self_type` parameter through the call chain, enabling proper desugaring.

- **Integration test placement errors** — The `e2e_veci64_impl` test was accidentally inserted inside another function due to brace mismatch. Required a complete rewrite of the integration test file to fix.

- **UI test snapshots had ANSI codes** — Added `std::env::set_var("NO_COLOR", "1")` in the UI test harness to disable color output for consistent snapshot comparison.

- **COUNT constant out of sync** — Adding new `SyntaxKind` variants (e.g., `Amp`, `KwFor`, `KwIn`, `WhileExpr`, `ForInExpr`) required updating the hardcoded `COUNT` and the corresponding test assertion.

## Decisions Made

- **Work‑queue seeding from both `call_type_args` and body scanning** — The monomorphization queue is seeded from explicit type arguments (provided by the type checker) and also by scanning function bodies for generic calls. This ensures that even when the type checker doesn't populate `call_type_args`, generic instantiations are discovered.

- **Structural type‑param inference** — Instead of assuming param *i* maps to type‑param *i*, inference walks the function's parameter types and matches any `HirType::Named` equal to a type parameter symbol, using the corresponding argument's inferred type. This correctly handles `fn wrap<T>(label: i64, value: T)`.

- **`fn_mangle_map` keyed by `(Symbol, Vec<HirType>)`** — To avoid collisions when the same generic function is instantiated with different type arguments, the mangle map uses the full key, and `rewrite_expr` infers type arguments at call sites from argument expression types.

- **`self: Type` preferred over `&self` for now** — The `&self` sugar works but relies on the impl target type being propagated. For robustness, the stdlib initially uses `self: VecI64` explicitly.

- **No for‑in desugaring yet** — The for‑in loop was partially implemented (AST, HIR, lowering as a desugaring to `while` + `match`), but the desugaring contained type mismatches and was reverted. The syntax tokens and HIR variant remain, ready for a future implementation.
