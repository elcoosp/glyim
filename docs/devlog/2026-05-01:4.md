# 2026-05-01: Type Checker Completion, Stdlib Foundations & Codegen Fixes

## What I Did

### Phase 1: Type Checker Gap Closure

- **Argument type checking in function calls** — Extended `check_call_with_type_args` to validate each argument against the corresponding parameter type. Only fires for fully concrete (non-generic) functions; generic calls are deferred to monomorphization. Integration test `call_with_wrong_argument_type_reports_error` confirms that passing `i64` to a `bool` parameter produces a `MismatchedTypes` error.

- **Let annotation enforcement** — Modified `check_stmt` for `LetPat` to use the explicit type annotation (when present) as the binding type, overriding inference. The annotation is validated against the inferred type, with compatibility relaxations for `Generic<->Named` mismatches (same base type name). Integration test `let_annotation_mismatch_reports_error` confirms `let x: f64 = 42` now errors.

- **Assign‑to‑immutable detection** — The `Assign` arm in `check_stmt` now checks the mutability flag on the binding and emits `AssignToImmutable` when appropriate. UI test `assign_immutable.g` was rewritten to trigger the error, and the snapshot was updated.

- **Bool mismatch UI tests** — Updated `bool_mismatch.g` and `type_mismatch.g` snapshots to capture the new error messages ("expected Bool, found Int"). Both UI tests now produce visible error output instead of empty snapshots.

- **`is_valid_cast` relaxation: Int → Named(type_param)** — Extended the cast validator to accept `(Int, Named(_))` and `(_, Named(_))` as valid casts, enabling `0 as T` for generic type parameters. This unblocked the `Vec::get` out‑of‑bounds fallback.

### Phase 2: Cross‑Compilation

- **Target triple wired into object writer** — Modified `write_object_file_with_opt` to use `self.target_triple` when set, falling back to the host default. Added integration test `cross_compile_produces_correct_elf_magic` that builds for `aarch64-unknown-linux-gnu` and verifies the ELF header magic and `e_machine` field. Invalid triples are rejected early by `cross::validate_target`. All 4 cross‑compilation tests pass.

### Phase 3: Standard Library — String, Range, HashMap

- **`String` type** — Implemented `stdlib/src/string.g` as a `Vec<u8>` wrapper. Methods: `new`, `len`, `is_empty`, `push_byte`. Properly depends on `vec.g` in integration tests (no self‑contained copy). Two integration tests pass: `e2e_string_new_len` (returns 0), `e2e_string_is_empty` (returns 1).

- **`Range` type** — Implemented `stdlib/src/range.g` with `new(start, end)` and a self‑consuming `next() -> Option<i64>`. Four integration tests pass: `e2e_range_next` (0..5 returns 4), `e2e_range_empty` (0..0 returns None), `e2e_range_iteration`, `e2e_range_sum`. One test ignored (`e2e_range_sum_full`) due to method rebinding SIGSEGV.

- **`HashMap<K,V>`** — Implemented `stdlib/src/hashmap.g` with `insert`, `len`, and a full linear‑probing `get` using `K==K` comparison. `insert` is a stub (increments `len` only — full implementation blocked by `Vec::set` codegen). The `get` logic properly walks the bucket array, checks occupancy with `entry.occupied == 0`, and compares keys. Three integration tests pass: `e2e_hashmap_basic`, `e2e_hashmap_insert_get` (insert+len), `e2e_hashmap_new_len`.

### Phase 4: Codegen Fix — `0 as Struct` Segfault

- **Root cause** — `Vec::get` on an empty vector returns `0 as T` for out‑of‑bounds access. For `T = Entry<i64,i64>` (24 bytes), this returned a null pointer. Subsequent field access (`entry.key`) GEP'd into null → SIGSEGV.

- **Fix: zero‑initialized struct allocation** — Added a new arm in the `HirExpr::As` codegen (`codegen/expr/mod.rs`) for `(Int, Named(_)) | (Int, Generic(_, _))` matches. When the target type is a registered struct, it calls `glyim_alloc(size)`, stores `0` to every field, and returns the pointer. Integration test `e2e_zero_as_struct_field_access` confirms no segfault.

- **Monomorphizer: ensure struct registration** — Added `ensure_struct_specialized` method that recursively registers concrete struct types used as generic arguments (e.g., `Entry<i64,i64>` when specializing `Vec_get` with `T=Entry<i64,i64>`). This ensures `hir_type_to_llvm` can find the struct layout at codegen time.

- **Monomorphizer: correct specialization selection** — Fixed the `MethodCall → Call` rewrite in `rewrite_expr` to match specialization entries by concrete type arguments, not just the base function name. Previously, `Vec_get__i64_i64` (wrong) could be selected over `Vec_get__Entry_i64_i64` (correct) because both share the base name `Vec_get`. Now the receiver type args are compared to the specialization key. This caused `main` to call the wrong specialization, which returned plain `0` instead of an allocated struct.

### Phase 5: Generic Equality `K==K`

- **Type checker relaxation** — `infer_expr` for `Binary { op: Eq | Neq, .. }` already returns `Bool` for comparisons. No type‑checker changes were needed — generic equality already worked. Integration tests `e2e_generic_equality` (true) and `e2e_generic_equality_false` (false) both pass, confirming `fn eq<K>(a: K, b: K) -> bool { a == b }` works correctly with monomorphization.

## Problems Hit

- **Debug‑damage spiral** — Attempts to add `eprintln!` tracing across multiple crates caused cascading syntax errors (unmatched braces, orphaned function arguments, broken `format!` calls). Required `git checkout` to restore clean files and re‑apply fixes manually. Lesson: never inject debug logging across more than one file at a time; use a single temporary file with `cargo run --bin glyim -- ir <file>` instead.

- **Annotation type propagation** — `let v: Vec<Entry<i64,i64>> = Vec::new()` required the annotation type to be used as the binding type (overriding the inferred `Vec<T>` with unresolved generic params). The initial annotation check code compared the annotation and inferred types but kept the inferred type for binding, causing later monomorphization to use `Vec<T>` instead of `Vec<Entry<i64,i64>>`. Fixed by using `annotated.clone()` as the binding type when present.

- **`build_result` eprintln damage** — Removing debug logs mangled two `eprintln!` calls in `build_result`, leaving orphaned format arguments and mismatched braces. Required replacing the entire function with a clean version.

- **Unused `vec_src` warning** — `e2e_zero_as_struct_field_access` included `vec_src` but the test source is self‑contained. Accepted as a minor warning.

## Decisions Made

- **ADR‑043: Zero‑allocation for `0 as Struct`** — The `As` codegen now allocates a properly sized zero‑initialized buffer when casting integer `0` to a concrete struct type. This matches user expectation that `0 as T` produces a zeroed value of type `T`, not a null pointer. The alternative (allocating on the stack via `alloca`) was rejected because structs may need to outlive the function (e.g., returned from `Vec::get`).

- **ADR‑044: `ensure_struct_specialized` in monomorphizer** — When specializing a generic function with a concrete type argument that is itself a generic struct (e.g., `T = Entry<i64,i64>`), the monomorphizer now recursively registers the concrete struct type as a specialized `HirItem::Struct`. This ensures the codegen can find the LLVM struct type for field access and `__size_of::<T>()`.

- **ADR‑045: Monomorphizer specialization matching by concrete args** — The `MethodCall → Call` rewrite now filters `fn_map` entries by the receiver's concrete type arguments, not just the mangled base name. This prevents ambiguity when multiple specializations of the same method exist for different type parameters.

- **ADR‑046: `occupied` field uses `i64` instead of `bool`** — The `Entry` struct's `occupied` field uses `i64` (0 = empty, non‑zero = occupied) because the codegen for `bool` fields currently returns `Int` instead of `Bool`, causing `IfConditionMustBeBool` type errors. This is a workaround pending proper `bool` field codegen.
