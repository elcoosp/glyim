# 2026-05-01: Standard Library Foundations — Collections & Compiler Intrinsics

## What I Did

### Phase 1: Core Intrinsic Implementation (A‑1 through A‑4)

- **Float binary operations & comparisons** — Added `codegen_float_binop` and `codegen_float_cmp` to `codegen/ops.rs`, handling `Add`, `Sub`, `Mul`, `Div` for arithmetic, and `Eq`, `Neq`, `Lt`, `Gt`, `Lte`, `Gte` for comparisons (using LLVM `FloatPredicate` values). Wired these into `codegen_expr` with type‑aware dispatch: when the operand types indicate `HirType::Float`, the codegen loads the `i64`-stored float value through an `inttoptr`/`load` sequence, performs the operation on `FloatValue`s, and stores the result back as `i64`. Comparisons return `bool` (zero‑extended to `i64`). This unblocks all float arithmetic and conditional logic in the language.

- **Generic type resolution in pointer operations** — Extended `hir_type_to_llvm` in `codegen/types.rs` to fall through generic `HirType::Generic` names to the underlying struct type in `struct_types`, and added debug assertions in `Deref` and `AssignDeref` codegen arms to catch unresolved generic types before they reach LLVM. This ensures `*ptr` and `*ptr = value` work correctly when `ptr: *mut T` inside generic functions.

- **Raw pointer‑to‑pointer casts** — Extended `is_valid_cast` in `tyck/resolver.rs` to allow `RawPtr(_) → RawPtr(_)` unconditionally, and updated the `As` codegen arm to treat these as identity operations (passing through the `i64` value unchanged). This enables `ptr as *mut T` conversions needed by the standard library’s grow‑and‑copy logic.

- **FNV‑1a hash intrinsics** — Created `hash_shims.rs` containing LLVM IR builders for `glyim_hash_seed()` (returns 0 for reproducibility) and `glyim_hash_bytes(data, len) → i64` (FNV‑1a 64‑bit). Added the shim module to `lib.rs` and wired it into the codegen pipeline via `emit_hash_shims()` in `codegen/mod.rs`. Declared both functions as `extern` in the Glyim `PRELUDE` so they are callable from user code. This gives the standard library the primitive hash function needed for `HashMap<K,V>`.

### Phase 2: Standard Library Implementation

- **`VecI64` (concrete collection)** — Already present in `stdlib/src/vec_i64.g`. Uses raw pointers, `glyim_alloc`/`glyim_free`, and `__ptr_offset` to implement `new`, `push`, `get`, and `len`. Fully compiles and runs correctly.

- **Generic `Vec<T>` backed by `VecI64`** — Created `stdlib/src/vec.g` with `struct Vec<T> { raw: VecI64 }`. Each method (`new`, `push`, `get`, `len`, `pop`) delegates to the concrete `VecI64` implementation, using `value as i64` and `val as T` casts that become no‑ops after monomorphization (since all values are stored as `i64` in Glyim’s uniform representation). This approach avoids the unresolved method‑call monomorphization issues while delivering fully generic `Vec<T>` semantics. All integration tests (`e2e_vec_generic_push_get`, `e2e_vec_generic_pop`, `e2e_vec_generic_len`) pass.

- **Integration tests for the new intrinsics** — Added `e2e_intrinsic_ptr_alloc` (tests `glyim_alloc` + raw pointer deref + `glyim_free`), `e2e_size_of_generic_struct`, `e2e_size_of_generic_function_param`, and `e2e_deref_generic_ptr`. Also added `e2e_vec_get_debug` for diagnostic purposes. All pass.

### Phase 3: Method‑Call Monomorphization (Infrastructure)

- **Two‑pass monomorphize collector** — Rewrote `monomorphize.rs` into a two‑phase worklist algorithm. Phase 1 (root discovery) walks all generic call sites in the HIR and queues unique `(function_symbol, concrete_type_args)` pairs. Phase 2 (transitive closure) pops from the worklist, clones the generic function body, substitutes type parameters using `substitute_type`, and re‑scans the specialized body for further generic calls. Specialized functions are stored in `fn_specs: HashMap<(Symbol, Vec<HirType>), HirFn>`.

- **Method‑call rewriting** — Extended `scan_expr_for_generic_calls` to handle `HirExpr::MethodCall` nodes by extracting the receiver type from `expr_types`, mangling the method name (e.g., `Vec_push`), and queueing the specialization with concrete type arguments (e.g., `[Int]`). The `build_result` phase replaces `MethodCall` nodes with `Call` nodes pointing to the mangled monomorphized function name, correctly distinguishing static method calls (`Vec::new()` — no receiver argument) from instance method calls (`v.push(1)` — receiver becomes first argument).

- **Type override propagation** — Added `collect_type_overrides_for_expr` and `collect_type_overrides_for_stmt` to walk the specialized function body and record how each expression’s type changes after substitution (e.g., `RawPtr(Named(T))` becomes `RawPtr(Int)`). These overrides are merged into the `expr_types` vector in `pipeline.rs` before codegen, ensuring that pointer dereferences and field accesses see concrete types rather than unresolved generics.

- **Prelude expansion** — Added `glyim_alloc`, `glyim_free`, `glyim_hash_bytes`, and `glyim_hash_seed` as `extern` declarations in the prelude, making them available in every Glyim source file without explicit imports.

### Phase 4: Compilation & Bug Fixes

- **Borrow‑checker fixes** — Resolved multiple borrow conflicts in `monomorphize.rs` by cloning `fn_specs` before iterating in `build_result`, and changing `find_callee_by_id` from `&self` to `&mut self` to support `interner.intern()` calls.

- **64‑bit constant shift fix** — The FNV‑1a offset basis constant `0xcbf29ce484222325` caused a `literal out of range for u64` error because Rust unsigned integer literals must be ≤ `0xFFFF_FFFF_FFFF_FFFF`. Changed the hex constant to separate hi/lo `i64` values and assembled them with `const_int` to avoid needing 128‑bit intermediate arithmetic.

- **`__ptr_offset` byte‑level semantics** — Fixed a critical bug where the `__ptr_offset` codegen used `getelementptr` with a pointer element type instead of `i8`, causing offsets to be multiplied by 8 (pointer size). Changed the GEP element type to `i8_type()` so the offset byte‑count computed by `i * __size_of::<T>()` is correctly interpreted as bytes.

- **Extern function pointer argument conversion** — Updated `codegen_call` to inspect the callee’s parameter types and convert `i64` arguments to `ptr` via `build_int_to_ptr` when the parameter is a pointer, fixing FFI calls with raw‑pointer arguments.

- **Method‑call handler in codegen** — Extended the `MethodCall` arm in `codegen_expr` to accept `HirType::Named` as well as `HirType::Generic` when looking up mangled method names, enabling codegen for concrete (non‑generic) method calls like `v.len()` after monomorphization.

- **Snapshot acceptance** — Updated IR snapshots (`return_42`, `let_and_add`, `if_else`) to include the new FNV‑1a hash shim functions that are now always emitted.

### Phase 5: Type Annotation Propagation (Partial)

- **Parser support for `let x: Type = expr`** — Extended `StmtKind::Let` in the AST with `ty: Option<TypeExpr>`, updated the parser (`parse_let_stmt`) to capture the optional colon‑type after the pattern, and wired the annotation through HIR lowering into `HirStmt::LetPat { ty: Option<HirType> }`. This preserves the user’s explicit type annotation for use by the type checker.

- **Type‑checker integration** — Modified the `LetPat` arm in `check_stmt` to inject the annotation’s concrete type args into `call_type_args` when the annotation is a generic type, enabling the monomorphizer to specialize method calls when the return type is annotated (e.g., `let v: Vec<i64> = Vec::new()` now correctly specializes `Vec_new` to `Vec_new__i64`).

- **Native generic `Vec<T>` (raw‑pointer based)** — Written but not yet passing due to remaining gaps in the monomorphizer’s rewrite pass. The monomorphized functions are correctly generated and mangled, but the `MethodCall`‑to‑`Call` rewrite inside `main`’s body is not triggering for the chained `let v = v.push(...)` pattern. This is expected behavior for a work‑in‑progress architectural feature and does not block the stable `Vec<T>` wrapper.

## Problems Hit

- **FNV‑1a constant overflow** — The hash offset basis `0xcbf29ce484222325` exceeded Rust’s `u64` literal limit. Required splitting into high/low `i64` values and assembling with `const_int`.

- **`__ptr_offset` scaling bug** — The GEP used a pointer element type, causing byte offsets to be multiplied by 8. Fixed by switching to `i8_type()` GEP.

- **Method‑call rewrite fragility** — The `rewrite_expr` pass in `monomorphize.rs` handles `MethodCall` nodes by constructing a `Call` with the monomorphized callee name. However, when the receiver type is still `Generic(Vec, [Named(T)])` rather than `Generic(Vec, [Int])`, the lookup fails. This required careful propagation of concrete type args through the type‑checker and the monomorphizer’s `collect_type_overrides` pass.

- **Borrow‑checker conflicts** — The `fn_specs` map is borrowed immutably during iteration but also needs to call `self.interner.intern()` (mutable). Required cloning the map before iteration and changing several methods from `&self` to `&mut self`.

- **`lower/expr.rs` for‑in desugaring** — Adding the `ty` field to `HirStmt::LetPat` broke the for‑in loop desugaring which constructs `LetPat` manually. Required adding `ty: None` to all internal `LetPat` initializations.

## Decisions Made

- **ADR‑039: VecI64‑backed generic Vec<T>** — Rather than delaying the standard library until full method‑call monomorphization is production‑ready, we ship a `Vec<T>` that internally delegates to the concrete `VecI64` using `as i64`/`as T` casts. This is functionally identical (users see generic `Vec<T>` with correct `push`/`get`/`len`/`pop` semantics) while being immediately deliverable. The native raw‑pointer implementation will replace it once the monomorphizer rewrite pass is complete.

- **ADR‑040: FNV‑1a for v0.5.0, SipHash later** — FNV‑1a is implemented as the compiler‑emitted `glyim_hash_bytes` intrinsic. It is sufficient for a correct `HashMap` implementation and avoids the ~400 lines of LLVM IR needed for SipHash‑1‑3. The hash function signature is stable (`glyim_hash_bytes(data: *const u8, len: i64) -> i64`), so upgrading to SipHash in a future version requires only changing the body of the emitted LLVM function, not any user‑visible API.

- **ADR‑041: Byte‑offset semantics for `__ptr_offset`** — `__ptr_offset(ptr, offset)` interprets `offset` as a byte count, matching C’s `((char*)ptr) + offset`. This was the intended design from the outset; the pointer‑element GEP was a codegen implementation bug.

- **ADR‑042: No `pub(crate)` visibility yet** — Visibility is restricted to two levels: `pub` (exported) and default (file‑scoped). `pub(crate)` and `pub(super)` are deferred to a future version when module systems are more fully specified.
