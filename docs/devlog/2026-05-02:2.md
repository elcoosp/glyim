# 2026-05-02: Runtime Shims, Enum Representation, I/O, and Iterators

## What I Did

- **Renamed all compiler-emitted runtime shims** — Prefixed every runtime function with `__glyim_` (`__glyim_alloc`, `__glyim_free`, `__glyim_println_int`, `__glyim_println_str`, `__glyim_assert_fail`, `__glyim_hash_seed`, `__glyim_hash_bytes`). The C standard library functions (`write`, `abort`, `printf`, `malloc`, `free`, `open`, `close`) keep their original names and are JIT-mapped to libc. This eliminates symbol collisions when user code declares `extern { fn write(...) }`, the primary blocker for I/O support.

  **Scope:** Modified `runtime_shims.rs`, `alloc.rs`, `hash_shims.rs`, `codegen/string.rs`, `codegen/expr/mod.rs`, `codegen/expr/data.rs`, `pipeline.rs` (prelude), all stdlib source files, and all integration tests. Updated 3 insta snapshot files to match the new IR output.

- **Implemented i64 → i32 truncation for extern calls** — Added automatic `build_int_truncate` in `codegen_call` when a Glyim `i64` argument is passed to an extern function expecting `i32`. Also patched extern function registration in `codegen/mod.rs` to use correct LLVM types for `RawPtr` parameters (emitting `ptr` instead of `i64`), and added `get_function` guards to prevent duplicate runtime function declarations.

- **Fixed enum variant representation (the 3-day bug)** — The compiler was stack-allocating enum variants (`alloca {i32, {i64,i64,i64}}`) and returning dangling pointers. Rewrote `codegen_enum_variant` in `data.rs` to heap-allocate via `__glyim_alloc` with a uniform `{i32, i64}` layout: tag (4 bytes) + payload pointer (8 bytes). Struct values are stored by pointer, not by deep copy. Updated `codegen_match` in `control.rs` to perform proper two-arm tagged dispatch: load the tag, compare against 0, branch to `some`/`none` blocks, and use phi nodes to merge results. This fixed the 3-week-old SIGSEGV in all hashmap tests.

  **Impact:** 8 hashmap-related integration tests went from SIGSEGV to passing.

- **Added automatic string fat pointer extraction** — When a Glyim `Str`-typed expression is passed to an extern function expecting `*const u8`, `codegen_call` now extracts the thin data pointer from the `{i8*, i64}` fat pointer. This enables `write(1, "hello\n", 6)` without manual `as` casts.

- **Built the complete I/O standard library** — `stdlib/src/io.g` now provides `Stdin`, `Stdout`, `Stderr`, and `File` structs with `read`/`write`/`close` methods backed by `extern { fn write/read/open/close }`. Added `stdin()`, `stdout()`, `stderr()`, and `file_open()` constructors. Wrote 8 integration tests covering all I/O paths.

- **Implemented `for...in` loop support end-to-end** — Added `parse_for_in` to the parser (the AST node and HIR lowering already existed), fixed the desugaring to remove an extra `__iter` argument to `next()` that caused LLVM verification failures, and changed the loop condition from `while !__done` (which doesn't work for i64 booleans) to `while __done == 0`. Added `VecIter<T>` and `iter()` method to `Vec<T>` in `vec.g`.

- **Added comprehensive integration tests** — `e2e_for_in_range` (sums 1+2+3+4 = 10), `e2e_for_in_vec` (iterates Vec<i64>), `e2e_extern_write_i32_fd`, `e2e_write_string_literal`, `e2e_io_write_stdout_compile`, `e2e_io_write_stderr_compile`, `e2e_io_method_stdout_compile`, `e2e_io_method_stderr_compile`, `e2e_io_stdin_read_compile`.

- **Fixed the type checker's exhaustiveness for generic enums** — `get_enum_variants` in `match_check.rs` now handles `HirType::Generic(name, _)` by extracting the base name, which was preventing `Option<T>` matches from being recognized as exhaustive.

## Problems Hit

- **`build_bit_cast` returns `BasicValueEnum`, not `PointerValue`** — Attempting to use `build_bit_cast` to convert a fat pointer for string extraction failed because the Inkwell API requires a `PointerValue` for `build_load`. Resolution: Loaded the first `i64` field directly from the struct pointer using `build_int_to_ptr` + `build_load(i64)`, which works because the data pointer is stored at offset 0.

- **`for...in` desugaring produced broken IR** — The HIR lowering was passing `__iter` as an extra argument to `next()`, generating `call i64 @Range_next(i64 %__iter2, i64 %__iter3)` which failed LLVM verification. Resolution: Changed `args: vec![HirExpr::Ident { ... }]` to `args: vec![]` in the desugaring. Additionally, the loop condition `while !__done` used `HirUnOp::Not` on an `i64`, which doesn't work for boolean negation. Changed to `while __done == 0` using `HirBinOp::Eq`.

- **Duplicate runtime function declarations** — When user code declared `extern { fn write(...) }` and the compiler also emitted `write` as a runtime shim, LLVM raised "functions with that name already exist" errors. Resolution: Added `if module.get_function(name).is_none()` guards before every `module.add_function()` call in both `runtime_shims.rs` and `codegen/mod.rs`.

- **`Vec::get` only loaded 8 bytes from 24-byte structs** — When `Vec<Entry<K,V>>` called `*ptr` through `Deref`, the codegen loaded only an `i64` (the first field) instead of the full 24-byte struct. Later field access on `entry.key` read garbage memory. Resolution: Patched `codegen_expr`'s `Deref` handler to detect struct types and deep-copy from the source buffer into a fresh heap allocation via `build_load(struct_type)` + `build_store`.

- **Integration test source concatenation bugs** — Several hashmap tests included `hashmap.g` without `vec.g`, causing `Vec<T>` to be undefined. Each test source is a string concatenation, and missing dependencies silently fail at codegen time. Resolution: Added `vec_src` to the format strings for all hashmap tests.

- **Snapshot tests broke on function name changes** — Renaming runtime shims changed the LLVM IR output, causing 3 insta snapshot tests to fail with diff output. Resolution: Updated the snapshot files with the new `__glyim_` prefixed function names.

## Decisions Made

- **Enum representation is permanently `{i32, i64}`** — All enum variants (Option, Result, user-defined) use the same heap-allocated 16-byte layout. Tag meaning: 0 = Some/Ok/first variant, 1 = None/Err/second variant. This is a simplification that avoids type-dependent payload sizing and makes match codegen uniform. The tradeoff is that small enums carrying an `i64` payload use 16 bytes instead of 12, but the codegen simplicity is worth it.

- **`for...in` desugars to `while __done == 0` with flag-based iteration** — The desugaring creates mutable `__iter` and `__done` variables and uses a `match __iter.next()` in the loop body. This works because any struct with `next() -> Option<T>` is iterable without needing a `trait Iterator` keyword. The `trait` abstraction is deferred until the compiler can parse and check trait declarations.

- **Extern-backed methods use `extern_methods` registry** — Added `extern_methods: HashMap<Symbol, Symbol>` to `Codegen` and populate it during Pass 1 type registration. When codegen encounters a `MethodCall`, it checks this map first before falling through to mangled-name lookup. This is a lightweight dispatch mechanism that doesn't require HIR-level changes.

- **Runtime shim prefix `__glyim_` is reserved** — User code cannot define functions starting with `__glyim_`. This is enforced by convention only (no compiler check), matching the C/C++ pattern of reserved identifiers with double underscores.

## Next

- The iterator standard library (`iterator.g`) can be written without the `trait` keyword — just document the `next() -> Option<T>` convention. Range, VecIter, and any user type following this pattern work with `for...in`.
- File I/O works through the `extern` functions but needs proper error handling (return codes, errno). Currently `write`/`read` return values are ignored.
- Memory management for heap-allocated enums: they are never freed. A simple bump allocator or GC would be needed for production use.
- The `trait` keyword remains unimplemented but is now unblocked by all the infrastructure work (generic type checking, method dispatch, match exhaustiveness on generics).
