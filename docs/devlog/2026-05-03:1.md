# 2026-05-03: Macro Execution Engine, Iterator Support, and Doc Comments ‚Äî Complete

## What I Did

- **Built the Glyim macro execution engine** ‚Äî Created a deterministic WebAssembly sandbox (`MacroExecutor` using `wasmtime`) that runs macros in a fully reproducible environment. Implemented custom WASI stubs that fix the clock, use a seeded PRNG for randomness, and intercept filesystem I/O for content hashing. Every macro expansion is content-addressed via SHA-256 and cached in the local CAS (`LocalContentStore`). The engine supports remote caching through a REAPI-compatible CAS server built with Axum.

- **Extended the Glyim compiler to produce WebAssembly** ‚Äî Added `wasm32-wasi` target support to `glyim-codegen-llvm` via Inkwell, enabling `compile_to_wasm()` for macro definitions. Created a Wasm ABI layer (`wasm_abi.rs`) that maps typed macro signatures to Wasm exports and linear memory. Built the publish-time Wasm compilation path in `glyim-pkg` so macros are compiled to `.wasm` blobs, stored in the CAS, and verified for reproducibility.

- **Implemented typed AST serialization** ‚Äî Created `wasm_interface.rs` in `glyim-macro-core` with a binary format for serializing/deserializing typed AST nodes (`HirExpr`, `HirStmt`) between the host and Wasm guest. Supports integers, identifiers, binary operations, blocks, conditionals, and booleans with round-trip preservation.

- **Built the CAS server with REAPI endpoints** ‚Äî Rewrote `glyim-cas-server` from Actix to Axum, implementing blob storage, retrieval, missing-blob batch queries, and action result caching. Added a `/verify-wasm` endpoint that re-compiles macro source to verify reproducibility.

- **Implemented concrete iterators without traits** ‚Äî Created `Iter<T>` in `stdlib/src/iter.g` with `new()` and `next()` methods that return `Option<T>`. Added `Vec::iter()` method. This enables `for x in collection` loops without requiring a trait system, which the spec defers to v0.5.0+.

- **Fixed generic type inference for nested types** ‚Äî The monomorphizer could only infer `T = i64` from direct `Named` matches (e.g., `T ‚Üí i64`), but failed when type parameters were wrapped in `RawPtr`, `Generic`, or `Tuple`. Implemented recursive `extract_type_substitutions()` that walks through pointer types, generic instantiations, and tuples to find concrete type arguments. This fixed `Iter::new(data: *mut T, len: i64) ‚Üí Iter<T>` inference from `*mut i64` ‚Üí `T = i64`.

- **Fixed transitive monomorphization** ‚Äî When `Vec::iter()` internally called `Iter::new()`, the monomorphizer generated `Vec_iter__i64` but didn't scan its body for further generic calls, leaving `Iter_new` unresolved. Added type substitution via the `type_overrides` table so that monomorphized function bodies are scanned with concrete types, ensuring `Iter_new__i64` is generated.

- **Implemented two-pass forward-reference resolution** ‚Äî The parser now scans for `struct`/`enum` declarations before full parsing, building a `known_structs` set. When encountering expressions like `Iter::new()`, the parser checks this set to determine if the symbol refers to a struct (vs. an enum variant). This eliminates the need to define types in dependency order.

- **Added documentation comment support** ‚Äî Implemented Go-style position-based doc comments using standard `//` syntax. Consecutive `//` lines immediately preceding a top-level item are collected, beautified (prefixes stripped, whitespace normalized), and attached to HIR items via `attach_doc_comments()`. The documentation generator uses `pulldown-cmark` for full GitHub Flavored Markdown rendering including fenced code blocks, paragraphs, and lists.

- **Wired `glyim doc` CLI command** ‚Äî Connected the `Command::Doc` arm in `glyim-cli` to `pipeline::generate_doc()`, which lowers AST to HIR, attaches doc comments, and generates HTML with GFM rendering.

## Problems Hit

- **Wasm binary construction** ‚Äî Initially attempted to hand-craft the identity `.wasm` binary with Python, hitting multiple "invalid value type" and "unknown memory" errors from `wasmtime`. Each incorrect byte triggered an opaque parsing failure with no clear diagnostic. **Resolution:** Switched to using the `wat` crate to parse `.wat` text format in tests, making the modules readable and debuggable.

- **Borrow checker error in executor** ‚Äî `memory.grow(&mut store, ...)` and `memory.size(&store)` couldn't coexist in the same expression because `store` was borrowed both mutably and immutably. **Resolution:** Extracted `memory.size(&store)` into a separate `let current_pages` binding before the `grow` call.

- **`wasmtime::Error` doesn't implement `std::error::Error`** ‚Äî The `anyhow::Context::context()` trait requires `StdError`, which `wasmtime::Error` doesn't implement. This broke all `.context("...")` calls on fallible Wasm operations. **Resolution:** Replaced `.context()` with `.map_err(|e| anyhow!("...: {e}"))` throughout the executor.

- **Fuel metering immediately killed all Wasm executions** ‚Äî Setting `config.consume_fuel(true)` and `store.set_fuel(10_000)` caused every macro invocation to trap with "Interrupt" before executing a single instruction. **Resolution:** Disabled fuel and epoch interruption for initial development; they will be re-enabled with proper tuning in a later iteration.

- **Cache lookup returned zero bytes** ‚Äî The `MacroExpansionCache` stored action results via `store_action_result()` and `retrieve_action_result()`, but the output blob was never stored in the CAS under its content hash. The cache metadata said "output exists at hash X" but the blob wasn't actually there. **Resolution:** Explicitly called `store.store(output_bytes)` before storing the action result metadata, ensuring the blob is retrievable.

- **Name-based cache resolution failed on macOS** ‚Äî Using `:` as a separator in cache names (e.g., `macro:<hex>`) caused silent failures because `:` is disallowed in macOS filenames. **Resolution:** Switched to `-` as the separator (`macro-<hex>`).

- **Iter_for-loop crashed with SIGSEGV** ‚Äî `for x in v.iter()` desugared to a `MethodCall` on `Iter::next`, but the codegen treated unresolved method calls as if the receiver were an enum pointer, generating `load i32, ptr null` ‚Äî a null pointer dereference. **Resolution:** Multiple related fixes ‚Äî recursive type inference for pointer-wrapped generics, fallback mangle name lookup, and ensuring the monomorphizer's worklist processes transitive dependencies.

- **HIR spans vs. token offsets mismatch in doc generation** ‚Äî HIR spans used byte offsets from the prelude-prepended source, but `attach_doc_comments` tokenized the original source, so byte offsets didn't match. HIR item `add` had `span.start=405` (prelude-relative) but the token in the original source was at byte `120`. **Resolution:** Changed `generate_doc` to parse the original source directly without the prelude, so AST spans and token offsets inhabit the same coordinate space.

- **Span pointed at function name, not keyword** ‚Äî HIR `FnDef` spans started at the function name (e.g., `add`), not the `fn` keyword. Doc comments precede the `fn` keyword, so `collect_doc_comments` (which looks before the span's token) missed them. **Resolution:** Modified `attach_doc_comments` to walk backward from the name token to find the preceding keyword token (`fn`, `struct`, `enum`, `impl`, `extern`) and use that index for doc comment collection.

## Decisions Made

- **Wasm execution engine uses `wasmtime`** ‚Äî Wasm provides sandboxing, determinism, and performance. The emerging industry standard (dtolnay's Pre-RFC, Google Summer of Code 2024, rustc integration plan) all converge on Wasm for macro execution. Glyim's typed AST interface is simpler than token-stream-based approaches.

- **Typed AST serialization uses string-based identifiers** ‚Äî Because `Symbol(u32)` has a private constructor, reconstructed symbols from deserialization would be invalid. Using string identifiers avoids the issue while maintaining type safety for the Wasm interface.

- **CAS server uses Axum** ‚Äî Switched from Actix to Axum for the CAS server because Axum's API is simpler for route definition and it's the more modern choice in the Rust ecosystem.

- **Iterator support is concrete, not trait-based** ‚Äî Implementing a proper trait system would delay shipping. Concrete `Iter<T>` structs with `next()` methods work with the existing `for`-loop desugaring, delivering user value immediately. The trait system remains on the v0.6.0 roadmap.

- **Doc comments use Go-style position-based association** ‚Äî Standard `//` comments immediately preceding a declaration are treated as its documentation. Blank lines terminate the doc block. No new syntax, no lexer changes needed. This avoids the complexity of `///` token distinct while achieving the same result.

- **GFM rendering uses pulldown-cmark** ‚Äî Full GitHub Flavored Markdown support for doc comments (fenced code blocks, tables, task lists, links, etc.) via the well-established `pulldown-cmark` crate rather than a custom Markdown renderer.

- **Forward references are resolved by a parser pass, not a declaration phase** ‚Äî The two-pass parser scans for struct/enum names before full parsing, avoiding the need for a full AST-level declaration table. This is simpler than the originally planned `DeclTable` approach while solving the same problem.
