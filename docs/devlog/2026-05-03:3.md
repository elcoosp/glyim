# 2026-05-03: Fixing HashMap E2E: Pointer Representation, Monomorphization, and Type Inference

## What I Did

- **Diagnosed and fixed a critical representation mismatch in `Vec<Entry>`** — The compiler stored `Entry` structs as heap-allocated pointers, but the generic `Vec` implementation used `__size_of<T>()` returning the full struct size (24 bytes) for element offsets, causing access to read/write beyond allocated slots. Fixed by making `__size_of` return 8 for any struct type, aligning the array stride with stored pointer handles.

- **Corrected `Deref` and `AssignDeref` for struct types** — Previously, these operations loaded/stored the full 24‑byte struct from the pointer slot, reading garbage and corrupting memory (leading to stack overflows). Changed them to load/store only the 8‑byte pointer handle, treating struct values as references throughout the runtime.

- **Strengthened monomorphization for generic containers** — Added scanning of `SizeOf` expressions to ensure specialized struct types are emitted even if only used in `__size_of`. This fixed `e2e_size_of_generic_struct` (returning 0 instead of 8). Also improved the monomorphizer's ability to discover required specializations by tracking `call_type_args` from method calls with generic receivers (`Vec::new()`, `HashMap::get()`, etc.) and recording them so specialized functions like `HashMap_new__i64_i64` are generated.

- **Enhanced type checking for struct field access** — `check_field_access` now returns actual field types instead of always `HirType::Int`. This allows downstream passes (like monomorphization) to infer concrete type parameters from the accessed field’s type, which was essential for `Vec_len__u8` to be generated when called via `String::len()`.

- **Fixed backward type‑arg inference in the type checker** — When a `let` binding has a type annotation (e.g., `let m: HashMap<i64,i64> = HashMap::new();`), the checker now looks up the callee both in `self.fns` and `self.impl_methods`, then records the concrete type args so the monomorphizer later generates the proper specialization.

- **Improved the monomorphizer’s `Call` scanning logic** — Added a fallback that examines later method calls on the same variable within the enclosing block to infer type arguments for zero‑argument generic constructors (like `HashMap::new()` followed by `m.insert(…)`). Although not all cases were resolved this way, combining with type‑checker inference resolved the majority of concrete instantiation required by tests.

- **Re‑enabled and fixed several integration tests** — `e2e_hashmap_full_get`, `e2e_hashmap_insert_get`, `e2e_string_generic_len`, `e2e_size_of_generic_struct`, and others that were failing due to the above infrastructure issues now pass or only fail due to missing monomorphization of `unknown_len`/`unknown_push` (a separate desugaring bug where method names are not fully mangled for generic `self` receivers).

## Problems Hit

- **Stack overflow after initial `SizeOf` fix** — Changing `SizeOf` to 8 fixed array indexing, but the deep‑copy in `Vec_get` and `Vec_push` still tried to `load { i64, i64, i64 }` from the 8‑byte pointer slot, reading 16 bytes of garbage and corrupting memory. This manifested as infinite recursion → stack overflow. Fixed by treating struct `Deref`/`AssignDeref` as simple pointer loads/stores.

- **"function 'HashMap_new' not found" persisted** — Despite adding type annotations and recording call‑type args, the monomorphizer still didn’t generate the specialized constructor because the type checker did not search `impl_methods` when resolving backward inference. Adding `or_else()` to look into `impl_methods` solved it.

- **`unknown_len`/`unknown_push` functions not found** — The desugaring pass `desugar_method_calls` sometimes produces mangled names that don't match any declared function because the method is called on a variable whose type has already been resolved to a named base type (e.g., `String`), but the type arguments are lost. Fixing `check_field_access` to return actual field types and adding reverse‑lookup in the monomorphizer (`concretize_type`) helped but did not completely eliminate all occurrences; the root cause is better addressed in a separate desugaring improvement.

- **Iteration of several patches** — Multiple cycles of adding, testing, and reverting patches were required to isolate the correct set of changes. The interdependencies between SizeOf, Deref, monotomorphization, and type inference meant a single fix often exposed the next underlying issue, requiring a systematic approach.

## Decisions Made

- [ADR-017] Struct values continue to be heap‑allocated and referenced via 8‑byte pointers. The `SizeOf` intrinsic returns the pointer size for struct types, not the actual struct size, to maintain compatibility with the generic `Vec` implementation.

- [ADR-018] Dereferencing a struct pointer returns the handle directly, not a deep copy. Code that needs a copy must explicitly use `*ptr` and then re‑store.

- [ADR-019] Type inference for generic method calls records call‑type args in the type checker even when substitution is empty, using the receiver’s concrete type arguments to guide later monomorphization.

- [ADR-020] Struct field access types are propagated from the struct definition, enabling monomorphization to infer type arguments through chains of field accesses.
