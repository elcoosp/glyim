# 2026-05-04: Bug Fixes — Monomorphizer Pipeline, Generic Specialization, and Struct `As` Codegen

## What I Did

- **Fixed two critical failing tests** that blocked the v0.5.0 pipeline: `e2e_vec_generic_push` and `e2e_zero_as_struct_field_access`. These revealed deep architectural issues in the monomorphizer's handling of already-specialized generic functions, the type-checker's `call_type_args` inference for mangled method names, and the codegen's struct type resolution for `As` expressions on generic types.

- **Root Cause — `As` Handler Struct Lookup** — The `As` expression codegen for `0 as Entry<i64,i64>` failed because the handler directly looked up `cg.struct_types.borrow().get(&sym)` using the unmangled symbol (`Symbol("Entry")`), while structs are stored with their mangled names (`Symbol("Entry__i64_i64")`). This caused `0 as T` in `Vec::get()`'s out-of-bounds branch to return literal `0_i64` instead of a heap-allocated zeroed struct, producing a SIGSEGV when `main` dereferenced it as a pointer.

  **Resolution:** Changed the `As` handler to call `cg.resolve_struct_type(target_type)` which properly mangles generic types to their concrete names before the struct table lookup. This single-line fix resolved the SIGSEGV and made `e2e_zero_as_struct_field_access` pass.

- **Root Cause — Monomorphizer Skipping Mangled Calls** — `e2e_vec_generic_push` failed because the monomorphizer never produced specializations for `Vec_new` and `Vec_inc_len`. The type-checker recorded `call_type_args` as `[Int, Int]` (Entry's type params K and V) for the desugared `Vec_get__Entry_i64_i64` call. When the monomorphizer's `collect_and_specialize` first loop processed this entry, it queued `Vec_get__Entry_i64_i64` with `[Int, Int]`, which `find_fn` resolved back to the generic `Vec_get` template. The `specialize_fn` then mapped `T → Int` (the first element of `[Int, Int]`), causing `0 as T` to become `0 as Int` instead of `0 as Entry<i64,i64>`.

  **Resolution:** Added logic in `scan_expr_for_generic_calls` to detect mangled callee names (containing `__`), extract the base function name and the receiver's concrete type args (e.g., `Generic(Entry, [Int, Int])` from `Vec<Entry<i64,i64>>`), and queue a specialization of the base function with those correct type args. Also added a skip in the first loop of `collect_and_specialize` to avoid processing mangled callee entries with incorrect `call_type_args`.

- **Root Cause — Codegen Missing Mangled Functions** — After fixing the monomorphizer skip, the codegen couldn't find `Vec_get__Entry_i64_i64` because the call site used the intermediate mangled name, while the module only contained the fully-specialized `Vec_get__Entry_i64_i64__i64_i64`. The prefix-based fallback in `codegen_call` (in `string.rs`) needed to handle both `_` and `__` separator patterns.

  **Resolution:** Extended the prefix search in `codegen_call` to match any function name starting with the requested name followed by `_` (including `__`). This allows calls to partially-specialized mangled names to resolve to their fully-specialized implementations.

- **Root Cause — Interner Cloning** — Discovered that the type-checker received `interner.clone()` while the monomorphizer and codegen used the original interner. This created inconsistent Symbol values across phases: symbols interned by the type-checker had different `u32` values than the same strings in the HIR interner. The `find_fn` method in the monomorphizer failed to match impl methods because the type-checker's `intern("Vec_get")` produced a different Symbol than the HIR's `Symbol(37)` stored in the impl block. Multiple critical lookups (method resolution in `typeck`, struct type registration in `codegen`, and `resolve_struct_type` lookups) all depended on Symbol identity which broke across interner instances.

  **Resolution:** Removed `.clone()` calls when passing the interner to `Codegen::new()` in the pipeline functions. The interner now flows from parsing → type-checking → monomorphizer → codegen as a single instance, ensuring Symbol identities remain consistent throughout the entire compilation pipeline. Removed 8 `interner.clone()` calls across `run`, `run_with_mode`, `build_with_mode`, `run_tests`, and `run_jit`.

- **Debugging Infrastructure** — Added extensive targeted debug logging throughout the monomorphizer (`collect_and_specialize`, `scan_expr_for_generic_calls`, work queue processing), the codegen (`resolve_struct_type`), and the type-checker (`MethodCall` handling) to trace the flow of type arguments and specialization decisions. Used `eprintln!` with `[mono scan]`, `[mono queue]`, `[specialize_fn]`, `[force_sub]`, and `[resolve_struct_type]` prefixes for grep-based analysis.

## Problems Hit

- **Symbol Identity Across Interner Clones** — The core issue was that `Interner` uses `HashMap<String, u32>` and `Vec<String>` internally. When `clone()` is called, the new Interner has the same strings at the same indices initially, but any new `intern()` call creates a different mapping. The type-checker interns `"Vec_get"` as `Symbol(0)` in its fresh clone, while the HIR's interner has `"Vec_get"` as `Symbol(37)`. Rust's `HashMap<Symbol, ...>` uses `Symbol(u32)` as the key — so `HashMap::get(&Symbol(0))` never matches `Symbol(37)` even though they represent the same string.

  This affected:
  - `find_fn` in the monomorphizer — couldn't find impl methods by name
  - `struct_types` in codegen — stored with HIR interner symbols but looked up with codegen interner symbols
  - `resolve_struct_type` — resolved the mangled name string correctly but couldn't find the resulting Symbol in struct_types

  **Resolution:** Eliminated the clone entirely by passing the same Interner instance through the pipeline. This required changing the ownership pattern: instead of `TypeChecker::new(interner.clone())` which created independent copies, all phases now share one Interner. The monomorphizer takes `&mut Interner` (to add new mangled symbols), and the codegen takes ownership of it at the end of the pipeline.

- **Borrow Checker in Monomorphizer Override Processing** — When iterating `self.call_type_args_overrides` and simultaneously calling `self.queue_fn_specialization()` and `self.specialize_fn()`, the borrow checker rejected the code because the immutable iteration borrowed `self` while the mutable calls also borrowed `self`.

  **Resolution:** Cloned the overrides HashMap into a local `Vec<(ExprId, Vec<HirType>)>` before iterating. The clone is cheap because `ExprId` is `Copy` and the type args vectors are small (1–2 elements).

- **Scoping Issue with `fn_def` in SAFE FALLBACK** — When moving the SAFE FALLBACK logic outside the `if let Some(type_args) = ...` block, the `fn_def` variable went out of scope. The fallback code referenced `fn_def` which was only available inside the `if let` match arm.

  **Resolution:** Extracted `fn_def` into a separate `fn_def_opt` binding before the `if let`, then used `if let Some(ref fn_def) = fn_def_opt` in the SAFE FALLBACK code path. This required restructuring the control flow to avoid the `if let ... && ...` pattern in favor of nested `if let` blocks.

- **Private Field Access for Debug Logging** — `Symbol` has a private `u32` field, preventing direct access in debug format strings. Multiple debug logging attempts failed with `error[E0616]: field '0' of struct 'Symbol' is private`.

  **Resolution:** Used `interner.resolve(sym)` to get the string representation instead of printing the raw u32 value. Changed debug messages to use string-based identifiers throughout.

- **Unused Variable Warnings During Iterative Debugging** — Adding and removing debug logging created several unused variable warnings: `id` in `HirExpr::Call` destructuring, `mangled_sym` in `rewrite.rs`, `first_sym` in `specialize.rs`, `base_sym` in `collect.rs`. These accumulated as the code was modified through multiple patches.

  **Resolution:** Fixed all warnings by prefixing unused bindings with `_` or removing them entirely. Cleaned up after the final fix was applied.

## Decisions Made

- **[ADR-030] Single Interner Throughout Pipeline** — The Interner must be a single shared instance from parsing through codegen. Cloning creates inconsistent Symbol identities that break all Symbol-keyed HashMaps (struct_types, impl_methods, fns, call_type_args). The monomorphizer receives `&mut Interner` to add newly-generated mangled names. The codegen takes ownership of the final Interner at pipeline end. Type-checker operations that need to intern new strings should work on the shared instance.

- **[ADR-031] Mangled Name Skip in Monomorphizer First Loop** — When `collect_and_specialize` iterates `call_type_args` entries and the callee's name contains `__` (indicating it's already been specialized during desugaring), the entry should be skipped. The mangled name itself encodes the concrete types (e.g., `Vec_get__Entry_i64_i64` tells us T = `Entry<i64,i64>`). The `call_type_args` for these entries contain the inner type's params (Entry's K and V), not the outer function's type params (Vec's T).

- **[ADR-032] Base Function Specialization from Mangled Calls** — When `scan_expr_for_generic_calls` encounters a call with a mangled callee name, it should extract the base function name (before `__`), get the receiver's concrete type, and queue a specialization of the base function with those type args. This avoids re-interpreting the inner type's params as the outer function's params.

- **[ADR-033] Prefix-Based Function Resolution in Codegen** — The `codegen_call` function should search for any function starting with the requested name followed by `_` when an exact match fails. This handles the case where call sites use partially-specialized mangled names (like `Vec_get__Entry_i64_i64`) while the module contains fully-specialized versions (`Vec_get__Entry_i64_i64__i64_i64`). The prefix search checks both single and double underscore separators.

- **[ADR-034] `resolve_struct_type` for Generic Struct Lookup** — All codegen paths that need to look up a struct type from a `HirType` should use `resolve_struct_type()`, not direct `struct_types.borrow().get()` calls. The `resolve_struct_type` method handles mangling `Generic(sym, args)` to the concrete struct name (`"Entry__i64_i64"`) and checking existence in the interner. Direct lookups fail for generics because the struct is stored by mangled name, not by the generic base name.
