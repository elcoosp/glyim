# 2026-05-04: Monomorphization & Codegen Hardening Sprint

## What I Did

- **Built a deterministic name‑mangling table (`MangleTable`)** — Ensures every `(base, concrete_args)` pair always produces the identical mangled symbol across different phases of the compiler, preventing `Interner` desynchronization bugs that had caused mysterious segmentation faults.

- **Split monomorphization internally while keeping the public API stable** — Introduced `MonoDiscovery` struct to capture all state (`fn_specs`, `struct_specs`, `enum_specs`, `type_overrides`, `call_type_args_overrides`, `mangle_table`, `interner`) during the collection phase, then forward it to the rewriting phase. This eliminates the subtle state‑loss regression that had previously broken 28 integration tests.

- **Implemented recursive type‑specialization for generics** — Added a `type_work_queue` that processes concrete struct and enum instantiations (like `Vec<Int>`, `Option<Option<Int>>`) until a fixpoint. This allows the compiler to emit specialized `Enum` and `Struct` items in the monomorphized HIR.

- **Centralized type unification** — Created a pure `unify` function in `glyim-typeck` that returns a substitution map or a clean error, then threaded it into argument checking, return‑type validation, and method‑call inference. Replaced ad‑hoc unification scattered across multiple files.

- **Added safe codegen helpers** — `struct_field_ptr`, `tuple_field_ptr`, `build_zeroed_alloca`, and `zero_init_struct_on_heap` now wrap raw GEP and allocation with bounds checks and zero‑initialization, drastically reducing the risk of segmentation faults from uninitialized memory or incorrect field offsets.

- **Removed the brittle `__T/__K/__V` guard in codegen** — The monomorphizer now guarantees full concretization, so the hack that skipped functions with those substrings is no longer needed. This unblocked ~15 integration tests that had previously been failing or segfaulting.

- **Built post‑specialization validation** — A `no_type_params` pass walks the final HIR and panics if any unresolved type parameters remain, catching incomplete specialization before codegen can produce invalid IR.

- **Wrote a suite of hardening tests** — Property‑based arithmetic tests (`a + b`, `a - b`, `a * b`, `a / b` via JIT), stress programs for nested generics (currently `#[ignore]` with clear documentation), and a fuzzer target for type‑check + codegen pipeline.

- **Fixed enum specialization rewriting** — Replaced the old `enum_map: HashMap<Symbol, Symbol>` with `enum_spec_map: HashMap<(Symbol, Vec<HirType>), Symbol>`, so that both `Option<i64>` and `Option<Option<i64>>` can coexist and `EnumVariant`/`Match` nodes are rewritten to the correct specialized name.

## Problems Hit

- **Two‑phase monomorphization lost internal state** — When `discover_instantiations` built a fresh `MonoContext` and discarded it, the `Interner` mutations and `type_overrides` were lost, causing the rewriting pass to miss mangled names.  
  **Fix:** Created a `MonoDiscovery` struct that captures all mutable state from the discovery context and is passed to `apply_specializations`.

- **`concretize_type` ignored enums** — The method only checked `struct_specs`, so `Generic(Option, [Int])` remained generic even after the enum was specialized.  
  **Fix:** Extended `concretize_type` to also check `enum_specs` and always mangle fully‑concrete types.

- **Rewriting happened before type‑specialization was complete** — The rewriting pass ran after a single `process_type_specializations` call, but new specializations could still be discovered during that pass.  
  **Fix:** Wrapped `process_type_specializations` in a `while !self.type_work_queue.is_empty()` loop to reach a fixpoint before building maps and rewriting.

- **`LetPat` type annotations were never concretized** — `rewrite_stmt` cloned the type annotation without running it through `substitute_type` or `concretize_type`, so `let x: Option<Option<i64>>` stayed generic.  
  **Fix:** Added `self.concretize_type(&crate::types::substitute_type(t, type_sub))` in `rewrite_stmt`.

- **Borrow checker and duplicate method conflicts** — Adding `concretize_type` methods across `collect.rs` and `context.rs` (both `impl MonoContext`) caused duplicate method errors and borrow issues.  
  **Fix:** Made the method `pub(crate)` in `collect.rs`, removed the duplicate from `context.rs`, and collected `type_overrides` values before iteration to avoid simultaneous borrows.

- **Nested generics stress tests still fail** — Despite all the above fixes, `stress_nest_option` and `stress_nest_vec` return incorrect results (SIGSEGV or 0 instead of 42/length). The root cause is that type annotations in the user‑written `main` function never undergo the full concretization pass (they aren't inside a generic function, so `type_sub` is empty and `substitute_stmt_types` is never called). A full HIR‑wide concretization pass is required but is a larger architectural change.  
  **Decision:** Re‑ignored the stress tests with descriptive `#[ignore]` messages and documented the remaining work.

## Decisions Made

- **Keep single‑pass `monomorphize` entry point with internal two‑phase design.** The external API stays unchanged, preventing pipeline regressions, while internal code can safely separate discovery from rewriting.

- **`MangleTable` is the single source of truth for name mangling.** All mangled names flow through this table, guaranteeing determinism.

- **Run type queue to fixpoint before codegen.** Prevents “late‑discovered” specializations from being missing in the rewrite pass.

- **Replace `__T/__K/__V` guard with reliance on monomorphizer + `no_type_params` validation.** Less hacky, more reliable.

- **Nested generics stress tests remain `#[ignore]` with explicit comments.** The infrastructure is in place (enum specialization, pattern rewriting, fixpoint queue), but the final concretization pass for HIR type annotations is deferred to a follow‑up sprint.
