# 2026-05-05: Nested Enum Specialization & Monomorphization Fix

## What I Did

- **Root‑caused the three persistent test failures** (`enum_specialization_for_nested_option`, `nested_option_enum_variant_rewritten`, `cli_cache_roundtrip_works`) that survived multiple attempted fixes. The core issue was in the HIR lowering of enum variant fields: for both `Unnamed` and `Named` variants, the field’s type was unconditionally set to `HirType::Int`, ignoring the type annotations parsed from the source. When nested generic types like `Option<Option<i64>>` were used, the monomorphization pass never saw the inner `Option<i64>` as a type argument, so it never specialized it. The type checker then failed to resolve the inner variant’s name (`Symbol(8)`) because the corresponding specialized enum was missing, producing `UnresolvedName` errors.

- **Fixed enum variant lowering in `glyim-hir/src/lower/item.rs`.** Replaced the hard‑coded `HirType::Int` with proper type expression lowering. For `Unnamed` variants (e.g., `Some(T)`), the field’s symbol is the type expression itself; for `Named` variants (e.g., `Rect { w, h }` with optional type annotations), the annotation, if present, is lowered, falling back to `Int`. This ensures the HIR carries accurate type information for subsequent monomorphization.

- **Enhanced monomorphization to fully specialize nested types** by patching `collect.rs`:
  - Extended `enqueue_type_if_generic` to explicitly enqueue `Option` and `Result` types for specialization, ensuring that `Option<i64>` is created before `Option<Option<i64>>`.
  - Modified `concretize_type` to resolve `Option` and `Result` into their mangled `Named` forms when the inner types are fully concrete, allowing the later passes to treat them as ordinary named types.
  - Added a fix‑point loop in `process_type_specializations` after the initial queue processing, so that newly enqueued types generated during specialization are processed immediately, guaranteeing that all nested types are fully specialized before the rewrite phase.

- **Fixed pattern rewriting in `rewrite.rs` for match arms.** Previously, enum variant patterns always kept the original generic enum name, even after specialization. Now the match expression’s scrutinee type is computed and used to determine the fully concrete, mangled enum name for each pattern’s `EnumVariant`, preventing wild goose chases in the type checker.

- **Debugged an unrelated compile error** (`CodegenBuilder not found`) that was introduced by an earlier, incomplete change; this was not part of the final fix and was a red herring. All compilation errors were resolved by the final patches.

- **Ran all 691 tests** after the complete fix. All previously failing tests now pass, including the three hard failures and the snapshot regression for `invalid_cast_to_str` (which had changed due to a new `span` field on `MismatchedTypes`—updated with `cargo insta accept`).

## Problems Hit

- **Multiple incremental fixes that didn’t stick** — Early attempts (adding recursive loops in `process_type_specializations`, scanning `LetPat` annotations, patching `rewrite.rs` with different approaches) failed because the root cause (incorrect variant field types) was not addressed. Each time, the same `UnresolvedName` error would resurface, leading to a cycle of patching, testing, and frustration.

- **Difficult‑to‑interpret error messages** — The test output showed `Symbol(8)` was unresolved, but it was not obvious that `Symbol(8)` was the inner `Some` variant of `Option<i64>`. The debug log from `dbg.txt` earlier was instrumental in pinpointing that monomorphization only created `Option__Option_i64` but not `Option__i64`, confirming the missing specialization step.

- **Snapshot test interference** — The initial fix changed `MismatchedTypes` to include a `span` field, causing a snapshot test failure. This was expected and resolved by accepting the new snapshot.

## Decisions Made

- **[ADR‑035] Full fidelity in HIR for generic enums** — The lowering phase must preserve all type information from the source. Hard‑coded fallbacks like `HirType::Int` are unacceptable for generics. Every type annotation must be lowered correctly so that later passes (type check, monomorphization) have accurate data.

- **[ADR‑036] Fix‑point iteration for type specialization** — The specialization queue must be drained completely, including items enqueued during processing. A second `while` loop at the end of `process_type_specializations` guarantees that nested types are fully specialized before the rewrite phase begins.

- **[ADR‑037] Monomorphization of built‑in generic types** — `Option` and `Result` are now treated as first‑class generic types and are explicitly enqueued by `enqueue_type_if_generic`. Their concrete forms are resolved by `concretize_type` when all type arguments are concrete, converting them to `Named(mangled)` early so the rewrite can replace patterns correctly.
