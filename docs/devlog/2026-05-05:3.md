# 2026-05-06: Test Suite Architecture – Integration Tests to Unit Tests

## What I Did

- **Collapsed 22 integration test binaries into a handful of `--lib` unit test modules across six crates** (`glyim-hir`, `glyim-typeck`, `glyim-parse`, `glyim-codegen-llvm`, `glyim-macro-core`, `glyim-lex`).  Every `tests/*.rs` file that didn't require a subprocess or external files was moved into a `src/tests/` directory, declared via a `mod.rs`, and connected to the crate with `#[cfg(test)] mod tests;`.

- **Resolved a crate dependency version mismatch** that prevented `glyim-hir` unit tests from calling `glyim-typeck`.  Two test files (`enum_specialization_test` and `nested_enum_rewrite`) relied on types from both crates and were relocated into `glyim-typeck`’s own test module where the dependency graph is clean.

- **Eliminated 16 compilation warnings** that had accumulated across test support files.  Removed unused imports (`glyim_interner::Interner`, `glyim_cli::pipeline::BuildMode`, `proptest::prelude::*`), dead extern declarations, unreferenced statics, and obsolete helper functions.

- **Adjusted snapshot paths** for `glyim-codegen-llvm` and `glyim-typeck` tests that used `insta` so they continue to read and write snapshot files relative to their new location.

- **Ported all migration and fix operations into self‑contained `bash` scripts** using Python for non‑trivial transformations, heredocs for file writing, and `git checkout` safety nets on failure.

- **Ran the full suite** – all unit tests (`cargo nextest run --workspace --lib`) and the remaining integration tests (`cargo nextest run --workspace --test integration`) pass without warnings.

## Problems Hit

- **“multiple different versions of crate `glyim_hir`”** – When tests inside `glyim-hir` called `typeck.check(&hir)`, the compiler treated `crate::Hir` and `glyim_hir::Hir` as different types because `glyim-typeck` depended on its own copy of `glyim-hir`.  The root cause was that `glyim-hir` itself was a transitive dependency of its own dev‑dependencies.  The only clean fix was to move those tests into `glyim-typeck`.

- **Stale `src/tests.rs` file shadowing `src/tests/mod.rs`** – After earlier botched attempts, a leftover `src/tests.rs` prevented Rust from discovering the directory‑based module.  Rust’s file‑vs‑directory resolution picks the file first, so the test modules didn’t appear.  Deleting the file and forcing a clean `mod.rs` solved it.

- **macOS `sed` incompatibilities** – The `c\` and `i\` commands require a literal newline after the backslash on BSD `sed`, while the scripts were written for GNU `sed`.  Switched to `perl -pi -e` in all places where the replacement was a single pattern, and to Python for multi‑line edits.

- **Duplicate function definitions after append operations** – When appending tests to existing files, some functions were added twice, causing `E0428` errors.  The surgical fix rewrote entire files from known‑good content rather than trying to surgically remove duplicates.

- **Snapshot path drift** – Tests that used `concat!(env!("CARGO_MANIFEST_DIR"), "/tests/snapshots")` had to be adjusted to `"../../tests/snapshots"` after the files moved from `tests/` to `src/tests/`.  Running `INSTA_UPDATE=always` once regenerated the snapshots correctly.

## Decisions Made

- **[ADR‑044] Integration tests become unit tests wherever possible** – Crates that don’t require a compiled binary, external subprocess, or test‑only data files should host their tests as `#[cfg(test)]` modules inside `src/`.  This dramatically reduces the number of test binaries and speeds up incremental compilation.

- **[ADR‑045] Dependency‑crossing tests belong in the depended‑upon crate** – When a test requires types from both `glyim-hir` and `glyim-typeck`, it should live in `glyim-typeck` (the “higher” crate) where both are ordinary external dependencies.  This avoids the “two versions of the same crate” problem.

- **[ADR‑046] Scripts commit immediately on success, roll back on failure** – Every migration script uses `git checkout HEAD -- <crate>` to undo partial damage if compilation or tests fail.  This makes the process idempotent and safe to re‑run.

- **[ADR‑047] Use `perl` for single‑line replacements, Python for multi‑line edits** – Avoids the portability issues of `sed` while keeping the patching logic simple and auditable.
