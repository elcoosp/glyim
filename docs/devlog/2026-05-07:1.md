## What I Did

- **Created the `glyim-orchestrator` crate** (Tier 5) ŌĆö the central nervous system for cross-package compilation. Contains 7 modules:
  - `graph.rs` ŌĆö Package graph discovery via workspace manifests, topological sorting with petgraph, cycle detection, transitive dependents. Supports both single-package and multi-member workspaces.
  - `symbols.rs` ŌĆö `PackageSymbolTable` mapping `(symbol_name) ŌåÆ (package_name, artifact_hash)`. Serializes via postcard for CAS storage. Merge support for combining dependency tables.
  - `artifacts.rs` ŌĆö `ArtifactManager` wrapping CAS + Merkle store. Stores/retrieves `PackageArtifact` (object code, symbol table, metadata, per-function objects). Named artifact resolution for cache lookups.
  - `linker.rs` ŌĆö Multi-object linker replacing the old single-file `cc` invocation. `LinkConfig` supports custom linker, library search paths, target triple, sysroot, LTO flags.
  - `incremental.rs` ŌĆö Cross-package incremental state (`CrossPackageIncremental`). Tracks per-package Merkle roots, dependency fingerprints, and determines which packages need rebuilding when source or dependency changes.
  - `interface.rs` ŌĆö `DependencyInterface` extracted from type-checked HIR. Contains public function signatures, struct/field types, enum variants. Serialized to CAS for downstream packages to type-check cross-package calls without needing dependency source code.
  - `orchestrator.rs` ŌĆö `PackageGraphOrchestrator` with `build()`, `check()`, `run()` entry points. Compiles packages in topological order, caches artifacts in local CAS, optionally pushes/pulls to remote CAS, links final binary from all object files.

- **Integrated workspace-aware CLI commands** ŌĆö `glyim build` and `glyim run` now detect workspace manifests via `find_package_root()`. When a workspace is detected, they delegate to `PackageGraphOrchestrator`. Added `--remote-cache <URL>` flag to `build`, `run`, and `test` commands. Enhanced `glyim fetch` to report remote cache status and lockfile contents.

- **Added `library_mode` to codegen** ŌĆö Non-root packages compile in library mode: skip `main` function, skip runtime shims (`__glyim_alloc`, `__glyim_println_int`, `__glyim_hash_bytes`, etc.). This eliminates duplicate symbol errors when linking multiple packages. The `CodegenBuilder` gains a `with_library_mode()` method; `Codegen::generate()` checks `self.library_mode` before emitting shims and requiring `main`.

- **Enhanced lockfile format** ŌĆö `LockedPackage` now carries `artifact_hash`, `interface_hash`, and `target_triple` fields. These enable `glyim fetch` to download pre-compiled artifacts from a remote CAS without recompilation.

- **Added serde support to core types** ŌĆö `Symbol` in `glyim-interner` and `HirType` in `glyim-hir` now derive `Serialize`/`Deserialize`. Required for `DependencyInterface` and `PackageSymbolTable` serialization.

- **Wired up `glyim-pkg` for cross-crate use** ŌĆö Made `resolve_member_globs` public in `workspace.rs` so the orchestrator can discover package members from glob patterns.

## Problems Hit

- **TempDir dropped before linking** ŌĆö `compile_single_package()` wrote object files to a `tempfile::tempdir()`. When the function returned, the `TempDir` was dropped, deleting the `.o` files. The linker then failed with "no such file or directory."

  **Resolution:** Added a `temp_dirs: Vec<TempDir>` field to `PackageGraphOrchestrator`. Object file temp dirs are pushed into this vector and kept alive until after `link_multi_object()` completes, then cleared with `self.temp_dirs.clear()`.

- **Duplicate symbols from runtime shims** ŌĆö When linking two packages together, both emitted `__glyim_alloc`, `__glyim_println_int`, `main`, and other runtime shims, causing linker errors.

  **Resolution:** Introduced `library_mode` in `Codegen`. Libraries skip shim emission and the `main` function. The orchestrator detects the root package and compiles it normally; all other packages get `builder.with_library_mode()`.

- **Cache lookup used wrong hash** ŌĆö The orchestrator stored artifacts under `ContentHash::of(source)` but later tried to look them up using that same hash, assuming `retrieve_object_code(hash)` would find them. However, artifacts were stored as blobs with their own content hash (different from the source hash), so lookups always returned `None`.

  **Resolution:** After storing a `PackageArtifact`, register a named mapping: `self.artifact_mgr.register_name(&format!("artifact:{}", source_hash), artifact_hash)`. Cache lookups now resolve the name to get the actual artifact hash before retrieval.

- **Borrow checker conflict in build loop** ŌĆö Computing `current_source_hash` required reading the package node from the graph, but the same node was later borrowed for `compile_single_package`. The immutable borrow of `self.graph.get()` conflicted with the mutable borrow of `self.compile_single_package()`.

  **Resolution:** Clone the `PackageNode` before the mutable call: `let pkg_node_clone = pkg_node.clone(); self.compile_single_package(&pkg_node_clone)`.

- **Dependency rebuilds weren't triggered** ŌĆö When package A's source changed, the orchestrator correctly identified A as changed, but package B (which depends on A) was marked as cached because B's own source hadn't changed. The cross-package incremental state wasn't consulted.

  **Resolution:** Added `deps_changed` computation in the build loop: for each package, check `self.cross_state.did_dependency_change()` for every declared dependency. The `should_skip` condition now requires `!source_changed && !deps_changed`.

## Decisions Made

- **Library mode is a codegen flag, not a separate target** ŌĆö Rather than creating a separate "library" compilation target, non-root packages compile through the same `Codegen` pipeline with a `library_mode: bool` flag. This avoids duplicating the entire codegen path and keeps the change minimal (~30 lines in mod.rs).

- **Package-level Merkle roots used for cross-package cache keys** ŌĆö Each package's source hash (SHA-256 of `main.g` content) serves as the Merkle root for that package's artifact. This root is stored in `CrossPackageIncremental` and used to detect when a dependency has changed.

- **Named CAS entries bridge source hashes to artifact hashes** ŌĆö Instead of changing the artifact storage model to use source hashes as keys, we use the CAS name registry (`register_name`/`resolve_name`) to map `artifact:<source_hash>` ŌåÆ `<artifact_hash>`. This keeps the CAS content-addressable model pure while enabling lookup by source identity.

- **The orchestrator clones package nodes before mutable calls** ŌĆö To avoid the borrow checker issues inherent in a graph traversal that both reads and mutates, `PackageNode` derives `Clone`. Cloning a node is cheap (it's just strings, paths, and a manifest). This is a pragmatic choice; a future optimization could use `Rc` or interior mutability.

- **Multi-object linker reuses the existing `link_object` infrastructure** ŌĆö `link_multi_object` invokes `cc` with all object file paths and library flags. It doesn't introduce a new linker or format; it just extends the single-object case to multiple inputs. This maintains compatibility with cross-compilation sysroots and custom linker configurations.
