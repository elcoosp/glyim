# 2026-05-07: Source‑Based Coverage Instrumentation

## What I Did

Built a complete source‑based code coverage pipeline for Glyim from scratch, covering
JIT execution, AOT binaries, branch instrumentation, and CLI reporting.

### Architecture Decisions

**Custom coverage instead of LLVM profraw.** Profraw requires linking
`libclang_rt.profile_*.a` into every binary and doesn't understand Glyim's HIR,
macros, or incremental compilation. Our hand‑rolled approach gives us:

- Macro‑aware source mapping (future)
- Live‑editor coverage (future)
- Incremental persistence across builds (future)
- Zero external runtime dependencies for JIT

**Two‑path runtime.** The flush function exists in two forms:
1. **JIT path** – `glyim_cov_flush_impl` in `runtime_shims.rs`, mapped into the
   JIT engine just like `println`/`assert`.
2. **AOT path** – `glyim-coverage-rt` static library (zero dependencies), linked
   via `rustc` instead of `cc` so the Rust runtime is initialised.

### Files Created

- `crates/glyim-coverage/` – Data types (`CoverageDump`, `SourceLocation`),
  text report generator, HTML report generator. Tests live in `src/tests.rs`.
- `crates/glyim-coverage-rt/` – Zero‑dep `staticlib` containing only
  `glyim_cov_flush_impl`. Uses `core::slice`, `std::fs::File`, and manual
  JSON formatting.
- `crates/glyim-cli/src/commands/cmd_coverage.rs` – `glyim coverage report`
  and `glyim coverage html` subcommands.
- `crates/glyim-cli-tests-full/tests/integration/coverage_e2e.rs` – JIT e2e
- `crates/glyim-cli-tests-full/tests/integration/coverage_aot_e2e.rs` – AOT e2e

### Key Fixes During Development

- Wrapped `CoverageInstrumenter` in `RefCell<Option<…>>` so it can be
  borrowed mutably through `&Codegen` in expression codegen.
- Overallocated the counter array (`num_fns * 50`) to accommodate branch
  probes added after the initial `emit_coverage_globals` call.
- Switched AOT linker from `cc` to `rustc` when coverage‑rt is present;
  `cc` cannot initialise the Rust runtime required by the static lib.

### What We Didn't Do (Yet)

- **Macro‑aware source mapping** – Instrumenter doesn't track macro call
  sites; all counters are attributed to the expanded source.
- **Incremental coverage** – Counters are not persisted across builds.
- **Live LSP coverage** – The flush function exists but isn't called
  mid‑execution.
- **LLVM‑IR flush for AOT** – The LLVM‑IR approach was abandoned because
  generating proper file I/O and JSON formatting in raw IR was impractical.
  The static‑library approach is cleaner and already works.
