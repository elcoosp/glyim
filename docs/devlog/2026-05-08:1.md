# 2026-05-08: Incremental Compilation Ramp-Up – Library Mode, Merkle Caching, and Incremental Test Selection

## What I Did

- **Implemented library compilation mode for AOT builds** — Added `--library` flag to `glyim build`, enabling compilation of Glyim source files to object files without requiring a `main` function. Wired through the CLI (`cmd_build.rs`), pipeline (`build_with_mode`), and codegen (`CodegenBuilder::with_library_mode`). The `Codegen::generate` method now skips the `main` requirement when `library_mode` is set.

- **Added no‑std prelude control** — Introduced `load_source_with_prelude_opt()` that conditionally skips prepending the `PRELUDE` constant based on `force_no_std` or the `no_std` keyword in source. Runtime shims (`emit_runtime_shims`, `emit_alloc_shims`, `emit_hash_shims`) respect `no_std` and are not emitted for `no_std` or library builds.

- **Fixed coverage runtime JSON format** — The original AOT coverage runtime wrote a simplified array-format JSON (`{"counters":[1,3,2],"version":1}`), but the test expected the full `CoverageDump` struct with a `counters` map and `metadata`. Updated `glyim-coverage-rt` to depend on `serde`/`serde_json`, parse the incoming `dump_json` blob, merge actual counter values into the `counters` map, and serialize the complete dump. Also fixed the linker invocation to use `cc` with `-Wl,-force_load` (macOS) / `-Wl,--whole-archive` for the coverage static library.

- **Added per‑function object code compilation support** — Created `Codegen::generate_for_items()` in `glyim-codegen-llvm` that compiles only a subset of HIR items (by index). Types and extern declarations are registered for all items (to resolve references), but only selected functions are forward‑declared and code‑generated. Also added `compile_items_to_objects()` that wraps each function in its own module, compiles to object blobs, and returns `Vec<(String, Vec<u8>)>`.

- **Built Merkle‑backed full‑module object caching** — Extended `QueryPipeline` with `prev_fingerprints` tracking and a `compile_incremental()` method that:
  1. Computes per‑function semantic fingerprints via `item_fingerprints`
  2. Diffs against previous fingerprints using `compute_item_diff` to find red (changed) and green (unchanged) items
  3. Loads cached per‑function object code for green items from the Merkle store
  4. Code‑gens only red items and stores new object code in the Merkle store
  5. Saves the fingerprint map as a Merkle node for the next run

  Updated `build_incremental()` to compute a Merkle root from function fingerprints, check for a cached full‑module object, and skip codegen entirely on cache hit. On miss, performs full codegen, then stores the object in the Merkle cache.

- **Added public name‑registry methods to `MerkleStore`** — `register_name()` and `resolve_name()` delegate to the backing CAS store, enabling human‑readable keys for cached artifacts (e.g., `obj:<function_name>`, `test-<test_name>`, `fingerprints_root`).

- **Implemented incremental test selection** — Added `--incremental` flag to `glyim test`. Created `IncrementalTestCache` in `glyim-testr` that stores/loads `TestResult`s as Merkle nodes keyed by `test-<name>` and qualified by source hash. On incremental runs, unchanged tests reuse cached results (skipping compilation and execution entirely); only tests whose source hash changed are re‑run. Results are stored back to the cache after each run.

- **Added serde derives to `glyim-testr` types** — `TestResult`, `TestOutcome`, and `TestDef` now derive `Serialize`/`Deserialize` for postcard serialization in the Merkle cache.

- **Wired `--incremental` through CLI** — Updated `main.rs` match arms, `cmd_test.rs` signature, and `TestConfig` struct to propagate the `incremental` flag from CLI to the test runner.

- **Added e2e smoke test** — Created `incremental_test_cache_e2e.rs` in `glyim-cli-tests-full` that verifies `glyim test --incremental` runs without crashing across consecutive invocations (with and without source modifications).

## Problems Hit

- **Writing the coverage runtime in Glyim hit language limitations** — Attempted to replace the Rust `glyim-coverage-rt` with a pure Glyim implementation. The Glyim compiler lacks arrays, address‑of (`&`), type‑annotated `let` bindings, and `else` in expression position — all needed for non‑trivial data manipulation.  
  **Resolution:** Reverted to the Rust `std` staticlib (which works and tests pass). Library mode and no‑std support were kept as valuable infrastructure for when Glyim matures.

- **Merkle store field access** — Early implementations tried to access `merkle.cas` directly, but `MerkleStore`'s `cas` field is private.  
  **Resolution:** Added `register_name()` and `resolve_name()` as public methods on `MerkleStore` that delegate to the backing CAS.

- **Missing `postcard` dependency** — `compile_incremental()` and the fingerprint storage code used `postcard::to_allocvec` / `postcard::from_bytes` but `glyim-compiler` didn't depend on `postcard`.  
  **Resolution:** Added `postcard = { version = "1", features = ["alloc"] }` to `glyim-compiler/Cargo.toml` and `glyim-testr/Cargo.toml`.

- **Multiple `build_with_mode` call sites needed updating** — Adding the `library: bool` parameter broke callers in `glyim-testr/src/compiler.rs`, `glyim-cli-tests-full/tests/integration/coverage_aot_e2e.rs`, and `glyim-cli/src/commands/cmd_build.rs`.  
  **Resolution:** Systematically found and updated all call sites via a series of Python scripts and sed patches.

- **`no_std` staticlib with stable Rust** — Attempted to compile the coverage runtime as a `#![no_std]` staticlib to avoid Rust runtime dependencies. Hit `unwinding panics are not supported without std` because the precompiled `core` library uses `panic=unwind`.  
  **Resolution:** Documented three solutions (keep `std`, nightly `-Zbuild-std`, or use `panic-halt` crate). Chose the pragmatic approach: keep `std`, link with `cc`, and revisit a pure Glyim implementation when the language supports it.

- **Brace mismatch from incremental logic insertion** — The first attempt at adding incremental test selection logic to `cmd_test.rs` introduced an unclosed delimiter.  
  **Resolution:** Replaced the entire post‑mutation‑mode section with a clean rewrite using well‑balanced braces.

- **Serde traits missing for cached types** — `TestResult` and `TestOutcome` didn't implement `Serialize`/`Deserialize`, causing `postcard::from_bytes` / `to_allocvec` to fail.  
  **Resolution:** Added `#[derive(serde::Serialize, serde::Deserialize)]` to both types.

## Decisions Made

- **Full‑module object caching over per‑function assembly** — Per‑function object caching would require merging multiple `.o` files (via `ld -r` or similar), adding complexity and external tool dependencies. Full‑module caching (one object per set of unchanged functions) is simpler, equally effective for the common case (most functions unchanged), and avoids the need for an assembler step. Per‑function storage is still done in the Merkle tree for future use, but the current pipeline caches and reuses the entire module object.

- **Incremental test caching lives in `glyim-testr`, not the CLI** — Moving `IncrementalTestCache` into `glyim-testr` keeps the CLI thin and puts the caching logic next to the test runner, where it has access to `TestResult` types and configuration. The CLI only passes the `--incremental` flag through `TestConfig`.

- **Merkle store as the unified caching layer** — Both compilation artifacts (full‑module objects, per‑function objects, fingerprints) and test results are stored in the Merkle store under the same `.glyim/incremental` directory. This gives us content‑addressed, deduplicated, verifiable caching with a single on‑disk format.
