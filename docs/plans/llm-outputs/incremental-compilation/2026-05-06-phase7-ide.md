**Date:** 2026-05-07

## 1. Executive Summary

Phase 7 transforms Glyim from a command-line-only compiler into a fully IDE-integrated development platform through a Language Server Protocol (LSP) implementation. Phases 0 through 6 built a comprehensive incremental compilation infrastructure — query-driven pipeline, per-function artifact caching, cross-module linking, CAS-backed dependency sharing, e-graph optimization, test-aware compilation, and mutation testing — but all of these capabilities are only accessible through the `glyim` CLI. The compiler has no LSP server, no symbol index, no source map infrastructure, and no mechanism to serve semantic information to editors and IDEs in real time. The closest existing feature is the `--json` flag in `glyim-cli`, which switches miette's diagnostic output to JSON format, but this is a post-hoc serialization of terminal output rather than a structured protocol for interactive tooling.

Phase 7 introduces a new `glyim-lsp` crate that implements the Language Server Protocol (LSP 3.17) and exposes the compiler's incremental pipeline as a persistent background service. The language server maintains an in-memory `AnalysisDatabase` — a symbol index, reference graph, type map, and documentation cache derived from the HIR — that is incrementally updated as source files change. When a file is edited, the server runs the query-driven pipeline from Phase 4 in a background thread, populates or updates the database, and pushes diagnostics, completions, hover information, and navigation data to the connected editor. The server leverages the existing `glyim-watch` file watcher for change detection, the `glyim-query` engine for cache-aware re-analysis, and the `glyim-syntax` CST (rowan-style `GreenNode`/`SyntaxNode`) for lossless syntax-tree access that supports formatting, folding, and syntax-aware editing.

The phase also introduces a `SourceMap` in `glyim-diag` that maps byte-offset `Span`s to `(FileId, Line, Column)` triples, a `SymbolIndex` that maps names to definition locations and type information, and a `ReferenceGraph` that supports find-all-references and rename operations. The `glyim-fmt` crate is upgraded from its current pass-through placeholder to a real AST-aware formatter using the CST. The `glyim-lint` crate is extended with a structured lint registry and integrated into the language server's diagnostic pipeline. Finally, the `glyim lsp` CLI command starts the language server, and editor configuration files for VS Code and Neovim are provided.

**Estimated effort:** 35–48 working days.

**Key deliverables:**
- `glyim-lsp` crate implementing LSP 3.17 (text document synchronization, diagnostics, completion, hover, goto definition, find references, rename, document symbols, folding range, formatting, code actions)
- `AnalysisDatabase` with `SymbolIndex`, `ReferenceGraph`, `TypeMap`, and `DocCache`
- `SourceMap` in `glyim-diag` for byte-offset-to-line/column mapping with multi-file support
- CST-aware formatter replacing `glyim-fmt` placeholder
- Structured lint registry in `glyim-lint` with configurable severity levels
- `glyim lsp` CLI command with `--port` and `--log-file` flags
- VS Code extension and Neovim LSP configuration
- Integration with Phase 4 incremental pipeline for sub-50ms diagnostics on edit
- Workspace-wide analysis using Phase 5 orchestrator for cross-module navigation

## 2. Current Codebase State Assessment

### 2.1 CLI-Only Interaction Model

The Glyim compiler is exclusively a CLI tool. Every interaction with the compiler requires invoking the `glyim` binary with a subcommand (`build`, `check`, `run`, `test`, `doc`, etc.). The CLI parses arguments via `clap`, runs the compilation pipeline, prints diagnostics to stderr, and exits. There is no persistent compiler process, no background analysis, no incremental state sharing across invocations, and no way for an editor to query the compiler without spawning a new process.

The `--json` global flag switches miette's `ReportHandler` to `JSONReportHandler`, which produces JSON-formatted diagnostic output. This is the closest the codebase comes to structured output, but it is designed for CI integration (parsing error messages from build logs), not for interactive tooling. The JSON output format is miette's internal representation, not LSP-compliant, and it does not include semantic information like types, definitions, or references.

## 3. Architecture Design

### 3.1 Language Server Architecture

The language server follows the standard LSP architecture: a persistent process that communicates with an editor over stdio (or TCP), maintains an in-memory analysis database, and responds to requests using the compiler's incremental pipeline. The server is organized into layers:

```
┌─────────────────────────────────────────────────────┐
│                    LSP Client (Editor)               │
│              VS Code / Neovim / Helix                │
└────────────────────┬────────────────────────────────┘
                     │ JSON-RPC 2.0 over stdio/TCP
┌────────────────────▼────────────────────────────────┐
│                  glyim-lsp Server                     │
│  ┌───────────────────────────────────────────────┐   │
│  │          LSP Protocol Handler                  │   │
│  │  (request routing, response serialization)     │   │
│  └───────────┬───────────────────────────────────┘   │
│  ┌───────────▼───────────────────────────────────┐   │
│  │          AnalysisDatabase                      │   │
│  │  SymbolIndex | ReferenceGraph | TypeMap       │   │
│  │  DocCache | SourceMap | Diagnostics           │   │
│  └───────────┬───────────────────────────────────┘   │
│  ┌───────────▼───────────────────────────────────┐   │
│  │       Analysis Driver (Background Thread)      │   │
│  │  QueryPipeline | FileWatcher | Orchestrator    │   │
│  └───────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────┘
```

The `AnalysisDatabase` is the central data structure that powers all LSP features. It is populated by the `AnalysisDriver`, which runs the incremental pipeline in a background thread whenever source files change, and queried by the `LspProtocolHandler` when the editor sends requests.

### 3.2 AnalysisDatabase

The `AnalysisDatabase` is a read-optimized data structure that indexes the compiler's analysis results for fast lookup by location, name, or type. It is designed for concurrent access: the background analysis thread writes to it, while LSP request handlers read from it.

```rust
// crates/glyim-lsp/src/database.rs

use glyim_diag::{FileId, SourceMap, Span};
use glyim_hir::{Hir, HirFn, HirType, HirItem};
use glyim_interner::Symbol;
use glyim_query::Fingerprint;
use std::collections::HashMap;
use std::sync::RwLock;

/// The central analysis database, optimized for LSP query patterns.
pub struct AnalysisDatabase {
    /// Maps file paths to FileIds.
    pub file_map: RwLock<FileMap>,

    /// Source maps for all open files, mapping byte offsets to line/column.
    pub source_maps: RwLock<HashMap<FileId, SourceMap>>,

    /// Symbol index: maps symbol names to definition locations.
    pub symbol_index: RwLock<SymbolIndex>,

    /// Reference graph: maps definition locations to all reference locations.
    pub reference_graph: RwLock<ReferenceGraph>,

    /// Per-expression type map: maps (FileId, ExprId) to HirType.
    pub type_map: RwLock<TypeMap>,

    /// Documentation cache: maps symbol names to doc comments.
    pub doc_cache: RwLock<DocCache>,

    /// Current diagnostics per file.
    pub diagnostics: RwLock<HashMap<FileId, Vec<Diagnostic>>>,

    /// Current HIR per file (for detailed queries).
    pub hirs: RwLock<HashMap<FileId, Hir>>,

    /// CST per file (for formatting and syntax-aware features).
    pub csts: RwLock<HashMap<FileId, glyim_syntax::SyntaxNode>>,

    /// Version counter for change tracking.
    pub version: RwLock<u64>,
}
```

### 3.3 SourceMap

The `SourceMap` is a new module in `glyim-diag` that provides multi-file, line/column-aware span resolution. Currently, `Span` is just `(start, end)` byte offsets with no file association and no line/column mapping. This is the foundational gap that must be closed before any LSP feature can work, because every LSP feature communicates locations as `(Uri, Position)` where `Position` is `(line, column)`.

```rust
// crates/glyim-diag/src/source_map.rs

use crate::Span;
use std::collections::HashMap;
use std::path::PathBuf;

/// Unique identifier for a source file within an analysis session.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, serde::Serialize, serde::Deserialize)]
pub struct FileId(pub u32);

/// Maps file paths to FileIds and provides byte-offset-to-line/column conversion.
pub struct SourceMap {
    /// The file's path on disk.
    pub path: PathBuf,
    /// The file's unique ID.
    pub file_id: FileId,
    /// The source text (owned, for line counting).
    source: String,
    /// Pre-computed line starts: byte offset of the first character of each line.
    line_starts: Vec<usize>,
}

impl SourceMap {
    /// Create a new SourceMap from a file path and source text.
    pub fn new(path: PathBuf, file_id: FileId, source: String) -> Self {
        let line_starts = compute_line_starts(&source);
        Self { path, file_id, source, line_starts }
    }

    /// Convert a byte-offset Span to a (line, column) range.
    /// Line and column are 0-based (LSP uses 0-based positions).
    pub fn span_to_position(&self, span: &Span) -> Option<(LineCol, LineCol)> {
        let start = self.offset_to_line_col(span.start)?;
        let end = self.offset_to_line_col(span.end)?;
        Some((start, end))
    }

    /// Convert a byte offset to (line, column).
    pub fn offset_to_line_col(&self, offset: usize) -> Option<LineCol> {
        let line = self.line_starts.partition_point(|&start| start <= offset) - 1;
        let line_start = *self.line_starts.get(line)?;
        let column = offset - line_start;
        Some(LineCol { line, column })
    }

    /// Convert a (line, column) back to a byte offset.
    pub fn line_col_to_offset(&self, position: LineCol) -> Option<usize> {
        let line_start = *self.line_starts.get(position.line)?;
        Some(line_start + position.column)
    }

    /// Get the source text.
    pub fn source(&self) -> &str {
        &self.source
    }

    /// Update the source text (when the file is edited).
    pub fn update(&mut self, new_source: String) {
        self.source = new_source;
        self.line_starts = compute_line_starts(&self.source);
    }
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, serde::Serialize, serde::Deserialize)]
pub struct LineCol {
    pub line: usize,
    pub column: usize,
}

fn compute_line_starts(source: &str) -> Vec<usize> {
    let mut starts = vec![0];
    for (i, ch) in source.char_indices() {
        if ch == '\n' {
            starts.push(i + 1);
        }
    }
    starts
}
```

The `SourceMap` also introduces `FileId` — a unique integer that replaces the implicit "current file" assumption in the current `Span` type. Every `Span` in the system is now associated with a `FileId` through the `SourceMap`, enabling multi-file analysis. The `Span` struct itself is extended with a `file_id` field:

```rust
// crates/glyim-diag/src/lib.rs (extended)

#[derive(Debug, Clone, PartialEq, Eq, serde::Serialize, serde::Deserialize)]
pub struct Span {
    pub file_id: FileId,            // NEW
    pub start: usize,
    pub end: usize,
    pub expansion_id: Option<u32>,
}
```

This is a breaking change that affects `glyim-hir`, `glyim-parse`, `glyim-typeck`, and `glyim-compiler`. The migration strategy is described in Section 17.

### 3.4 SymbolIndex

The `SymbolIndex` maps symbol names to their definition locations and metadata, enabling goto-definition, completion, and hover features:

```rust
// crates/glyim-lsp/src/symbol_index.rs

use glyim_diag::{FileId, LineCol, Span};
use glyim_hir::types::HirType;
use glyim_interner::Symbol;
use std::collections::HashMap;

/// A resolved symbol with its definition location and metadata.
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct SymbolInfo {
    /// The symbol's name.
    pub name: String,
    /// What kind of symbol this is (function, struct, enum, etc.).
    pub kind: SymbolKind,
    /// Where the symbol is defined.
    pub definition: DefinitionLocation,
    /// The symbol's type signature (for functions: parameter types and return type).
    pub type_signature: Option<TypeSignature>,
    /// Whether the symbol is public (visible to other packages).
    pub is_pub: bool,
    /// Documentation comments attached to the symbol.
    pub documentation: Option<String>,
    /// The package this symbol belongs to (for cross-module navigation).
    pub package: Option<String>,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, serde::Serialize, serde::Deserialize)]
pub enum SymbolKind {
    Function,
    Struct,
    Enum,
    EnumVariant,
    Field,
    TypeParameter,
    Local,
    Module,
    Constant,
    Macro,
}

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct DefinitionLocation {
    pub file_id: FileId,
    pub span: Span,
    pub line_col: Option<(LineCol, LineCol)>,
}

#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct TypeSignature {
    /// For functions: parameter types and return type.
    pub params: Vec<(String, HirType)>,
    pub return_type: Option<HirType>,
}

/// Index for fast symbol lookup.
pub struct SymbolIndex {
    /// Name → list of symbols with that name (multiple overloads/packages).
    by_name: HashMap<String, Vec<SymbolInfo>>,

    /// (FileId, byte_offset) → symbol at that location.
    /// Maps any position in the source to the enclosing symbol.
    by_location: HashMap<(FileId, usize), SymbolInfo>,

    /// All symbols in a file (for document symbol requests).
    by_file: HashMap<FileId, Vec<SymbolInfo>>,
}

impl SymbolIndex {
    pub fn new() -> Self { ... }

    /// Build the index from a HIR and source map.
    pub fn build_from_hir(
        &mut self,
        file_id: FileId,
        hir: &glyim_hir::Hir,
        source_map: &glyim_diag::SourceMap,
        interner: &glyim_interner::Interner,
    ) { ... }

    /// Look up a symbol by name.
    pub fn lookup_by_name(&self, name: &str) -> &[SymbolInfo] { ... }

    /// Look up the symbol at a given source location.
    pub fn lookup_by_location(&self, file_id: FileId, offset: usize) -> Option<&SymbolInfo> { ... }

    /// Get all symbols in a file (for document symbol request).
    pub fn symbols_in_file(&self, file_id: FileId) -> &[SymbolInfo] { ... }

    /// Get all symbols matching a query (for workspace symbol request).
    pub fn query(&self, query: &str, limit: usize) -> Vec<&SymbolInfo> { ... }

    /// Clear the index for a specific file (when the file is re-analyzed).
    pub fn clear_file(&mut self, file_id: FileId) { ... }
}
```

The `SymbolIndex` is built during the analysis phase by walking the HIR. For each `HirItem::Fn`, `HirItem::Struct`, and `HirItem::Enum`, it records the symbol's name, kind, definition location (from the `Span` attached to each HIR node), type signature (from `expr_types` in the type checker), and documentation (from `HirFn::doc`). The `by_location` index is a spatial index that maps byte offsets to the enclosing symbol, enabling goto-definition and hover at cursor position.

### 3.5 ReferenceGraph

The `ReferenceGraph` tracks all references to every symbol, enabling find-all-references and rename:

```rust
// crates/glyim-lsp/src/reference_graph.rs

use glyim_diag::{FileId, Span};
use std::collections::HashMap;

/// A reference to a symbol at a specific source location.
#[derive(Debug, Clone, serde::Serialize, serde::Deserialize)]
pub struct Reference {
    /// The file containing the reference.
    pub file_id: FileId,
    /// The span of the reference in the source.
    pub span: Span,
    /// Whether this is the definition (as opposed to a usage).
    pub is_definition: bool,
    /// The kind of reference (call, type annotation, field access, etc.).
    pub kind: ReferenceKind,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq, serde::Serialize, serde::Deserialize)]
pub enum ReferenceKind {
    Call,
    TypeReference,
    FieldAccess,
    Constructor,
    Pattern,
    Import,
}

/// Maps symbol names to all locations where they are referenced.
pub struct ReferenceGraph {
    /// Symbol name → all references to that symbol.
    references: HashMap<String, Vec<Reference>>,
}

impl ReferenceGraph {
    pub fn new() -> Self { ... }

    /// Build the reference graph from a HIR.
    pub fn build_from_hir(
        &mut self,
        file_id: FileId,
        hir: &glyim_hir::Hir,
        interner: &glyim_interner::Interner,
    ) { ... }

    /// Find all references to a symbol.
    pub fn find_references(&self, symbol_name: &str) -> &[Reference] { ... }

    /// Find all references in a specific file.
    pub fn references_in_file(&self, file_id: FileId) -> Vec<&Reference> { ... }
}
```

The reference graph is built by walking the HIR and recording every `HirExpr::Call(name, ..)`, `HirType::Named(name)`, `HirExpr::FieldAccess(..)`, and `HirExpr::StructConstruction(..)` as a reference. The `dependency_names()` function in `glyim-hir` already computes the set of symbols referenced by each function; the reference graph extends this with precise source locations.

### 3.6 Analysis Driver

The analysis driver is the background thread that runs the incremental pipeline and updates the `AnalysisDatabase`:

```rust
// crates/glyim-lsp/src/driver.rs

use crate::database::AnalysisDatabase;
use glyim_compiler::pipeline::QueryPipeline;
use glyim_diag::{FileId, SourceMap};
use glyim_query::IncrementalState;
use std::collections::HashMap;
use std::path::PathBuf;
use std::sync::Arc;
use std::time::Instant;
use tokio::sync::mpsc;

/// Message from the LSP handler to the analysis driver.
pub enum AnalysisMessage {
    /// A file was opened or changed.
    FileChanged { path: PathBuf, content: String, version: i32 },
    /// A file was closed.
    FileClosed { path: PathBuf },
    /// Request a full re-analysis of all open files.
    FullReanalysis,
    /// Shut down the driver.
    Shutdown,
}

/// The analysis driver that runs in a background thread.
pub struct AnalysisDriver {
    /// The shared analysis database.
    db: Arc<AnalysisDatabase>,
    /// The query-driven incremental pipeline.
    pipeline: QueryPipeline,
    /// File IDs for all open files.
    open_files: HashMap<PathBuf, FileId>,
    /// Source maps for all open files.
    source_maps: HashMap<FileId, SourceMap>,
    /// Receiver for analysis messages.
    rx: mpsc::UnboundedReceiver<AnalysisMessage>,
    /// The next FileId to assign.
    next_file_id: u32,
}

impl AnalysisDriver {
    pub fn new(
        db: Arc<AnalysisDatabase>,
        pipeline: QueryPipeline,
        rx: mpsc::UnboundedReceiver<AnalysisMessage>,
    ) -> Self { ... }

    /// Run the analysis loop.
    pub async fn run(&mut self) {
        while let Some(msg) = self.rx.recv().await {
            match msg {
                AnalysisMessage::FileChanged { path, content, version } => {
                    self.analyze_file(&path, &content);
                }
                AnalysisMessage::FileClosed { path } => {
                    self.close_file(&path);
                }
                AnalysisMessage::FullReanalysis => {
                    self.reanalyze_all();
                }
                AnalysisMessage::Shutdown => {
                    break;
                }
            }
        }
    }

    /// Analyze a single file using the incremental pipeline.
    fn analyze_file(&mut self, path: &PathBuf, content: &str) {
        let start = Instant::now();
        let file_id = self.get_or_create_file_id(path);
        let source_map = SourceMap::new(path.clone(), file_id, content.to_string());

        // Run the incremental pipeline
        let result = self.pipeline.compile_incremental(content, path);

        match result {
            Ok(compiled) => {
                // Update the database
                let mut db = self.db.write_all();

                db.source_maps.write().unwrap().insert(file_id, source_map);
                db.hirs.write().unwrap().insert(file_id, compiled.hir.clone());

                // Rebuild symbol index for this file
                db.symbol_index.write().unwrap().clear_file(file_id);
                db.symbol_index.write().unwrap().build_from_hir(
                    file_id, &compiled.hir, 
                    db.source_maps.read().unwrap().get(&file_id).unwrap(),
                    &compiled.interner,
                );

                // Rebuild reference graph for this file
                db.reference_graph.write().unwrap().build_from_hir(
                    file_id, &compiled.hir, &compiled.interner,
                );

                // Build type map from type checker results
                db.type_map.write().unwrap().build_from_typeck(
                    file_id, &compiled.expr_types, &compiled.interner,
                );

                // Clear diagnostics for this file
                db.diagnostics.write().unwrap().insert(file_id, vec![]);

                tracing::info!(
                    "Analyzed {} in {}ms ({} items)",
                    path.display(),
                    start.elapsed().as_millis(),
                    compiled.hir.items.len(),
                );
            }
            Err(errors) => {
                // Convert errors to LSP diagnostics
                let diagnostics = self.convert_errors(file_id, &errors);
                db.diagnostics.write().unwrap().insert(file_id, diagnostics);

                // Still try to build partial symbol index from any
                // successfully parsed items
                if let Some(partial_hir) = errors.partial_hir() {
                    db.symbol_index.write().unwrap().clear_file(file_id);
                    db.symbol_index.write().unwrap().build_from_hir(
                        file_id, partial_hir,
                        db.source_maps.read().unwrap().get(&file_id).unwrap(),
                        &errors.interner(),
                    );
                }
            }
        }
    }
}
```

The driver uses the `QueryPipeline` from Phase 4 for incremental re-analysis. When a file changes, only the queries whose fingerprints differ are re-executed, giving sub-50ms response times for typical edits (the existing incremental infrastructure caches per-function artifacts and only recompiles changed functions).

## 5. Completion Provider

### 5.1 Completion Sources

The completion provider draws from multiple sources:

1. **Symbol index completion**: All symbols in the current file and imported packages, filtered by the prefix typed so far. This is the primary source for function, struct, and enum names.

2. **Field completion**: When the cursor follows a `.` in a field access expression (`expr.field`), the provider resolves the type of `expr`, looks up the struct definition, and offers all field names as completions.

3. **Enum variant completion**: When the cursor is in a match arm or a pattern position and the matched expression has an enum type, the provider offers all enum variant names.

4. **Keyword completion**: Reserved keywords (`fn`, `struct`, `enum`, `if`, `else`, `match`, `let`, `pub`, `extern`, etc.) are offered when the parser would accept a keyword at the current position.

5. **Snippet completion**: Common patterns like `fn name(params) { ... }`, `if condition { ... }`, `match expr { ... }` are offered as snippets with tab stops.

```rust
// crates/glyim-lsp/src/completion.rs

use crate::database::AnalysisDatabase;
use lsp_types::*;
use glyim_diag::LineCol;

pub fn provide_completions(
    db: &AnalysisDatabase,
    params: &CompletionParams,
) -> Result<CompletionList, ()> {
    let uri = &params.text_document_position.text_document.uri;
    let position = params.text_document_position.position;

    let file_id = db.file_map.read().unwrap().get_by_uri(uri)?;
    let source_map = db.source_maps.read().unwrap().get(&file_id)?;
    let offset = source_map.line_col_to_offset(LineCol {
        line: position.line as usize,
        column: position.character as usize,
    })?;

    let mut items = Vec::new();

    // 1. Symbol completions
    let prefix = extract_prefix(source_map.source(), offset);
    let symbol_index = db.symbol_index.read().unwrap();
    for symbol in symbol_index.query(&prefix, 50) {
        items.push(symbol_to_completion_item(symbol));
    }

    // 2. Field completions (if cursor follows '.')
    if is_field_access(source_map.source(), offset) {
        if let Some(fields) = field_completions(db, file_id, offset) {
            items.extend(fields);
        }
    }

    // 3. Enum variant completions
    if is_match_arm(source_map.source(), offset) {
        if let Some(variants) = enum_variant_completions(db, file_id, offset) {
            items.extend(variants);
        }
    }

    // 4. Keyword completions
    items.extend(keyword_completions());

    // 5. Snippet completions
    items.extend(snippet_completions());

    Ok(CompletionList {
        is_incomplete: items.len() > 50,
        items,
    })
}
```

### 5.2 Completion Item Metadata

Each completion item includes rich metadata for the editor to display:

- **Detail**: Type signature for functions (`fn add(a: Int, b: Int) -> Int`), field type for struct fields, variant signature for enum variants.
- **Documentation**: Doc comments extracted from `DocCache`.
- **Kind**: `CompletionItemKind::Function`, `CompletionItemKind::Struct`, `CompletionItemKind::EnumMember`, etc.
- **Insert text**: The text to insert, possibly with placeholders for function parameters (`add(${1:a}, ${2:b})`).
- **Sort text**: Prioritized by relevance (exact match > prefix match > fuzzy match, local symbols > imported symbols > workspace symbols).

## 6. Hover Provider

### 6.1 Hover Content

The hover provider generates markdown-formatted documentation for the symbol under the cursor:

```rust
// crates/glyim-lsp/src/hover.rs

use crate::database::AnalysisDatabase;
use lsp_types::*;

pub fn provide_hover(
    db: &AnalysisDatabase,
    params: &HoverParams,
) -> Result<Option<Hover>, ()> {
    let uri = &params.text_document_position_params.text_document.uri;
    let position = params.text_document_position_params.position;

    let file_id = db.file_map.read().unwrap().get_by_uri(uri)?;
    let source_map = db.source_maps.read().unwrap().get(&file_id)?;
    let offset = source_map.line_col_to_offset(LineCol {
        line: position.line as usize,
        column: position.character as usize,
    })?;

    // Find the symbol at this location
    let symbol_index = db.symbol_index.read().unwrap();
    let symbol = symbol_index.lookup_by_location(file_id, offset);

    match symbol {
        Some(info) => {
            let mut markdown = String::new();

            // Type signature header
            markdown.push_str(&format!("```glyim\n{}\n```\n", format_signature(info)));

            // Documentation
            if let Some(doc) = &info.documentation {
                markdown.push_str(&format!("\n{}\n", doc));
            }

            // Effect annotation (from Phase 6 EffectAnalyzer)
            if let Some(effects) = get_effects(db, &info.name) {
                if effects.is_pure() {
                    markdown.push_str("\n*This function is pure.*\n");
                } else {
                    let mut effect_list = Vec::new();
                    if effects.writes_mutable { effect_list.push("writes mutable state"); }
                    if effects.performs_io { effect_list.push("performs I/O"); }
                    if effects.may_panic { effect_list.push("may panic"); }
                    markdown.push_str(&format!("\n*Effects: {}*\n", effect_list.join(", ")));
                }
            }

            // Source location
            if let Some(loc) = &info.definition.line_col {
                markdown.push_str(&format!(
                    "\n*Defined at {}:{}:{}*\n",
                    info.definition.file_id.0,
                    loc.0.line + 1,
                    loc.0.column + 1,
                ));
            }

            Ok(Some(Hover {
                contents: HoverContents::Markup(MarkupContent {
                    kind: MarkupKind::Markdown,
                    value: markdown,
                }),
                range: None,
            }))
        }
        None => Ok(None),
    }
}
```

## 7. Navigation Features

### 7.1 Goto Definition

Goto definition resolves the symbol under the cursor to its definition location. For local symbols (functions, structs, enums in the current file), this is a direct lookup in the `SymbolIndex`. For cross-module symbols (functions from dependency packages), the provider uses the `PackageSymbolTable` from Phase 5 to locate the defining package and file.

```rust
// crates/glyim-lsp/src/navigation.rs

pub fn goto_definition(
    db: &AnalysisDatabase,
    params: &GotoDefinitionParams,
) -> Result<Option<GotoDefinitionResponse>, ()> {
    let uri = &params.text_document_position_params.text_document.uri;
    let position = params.text_document_position_params.position;

    let file_id = db.file_map.read().unwrap().get_by_uri(uri)?;
    let source_map = db.source_maps.read().unwrap().get(&file_id)?;
    let offset = source_map.line_col_to_offset(LineCol {
        line: position.line as usize,
        column: position.character as usize,
    })?;

    // Look up the symbol at this location
    let symbol_index = db.symbol_index.read().unwrap();
    let symbol = symbol_index.lookup_by_location(file_id, offset);

    if let Some(info) = symbol {
        let def_source_map = db.source_maps.read().unwrap().get(&info.definition.file_id);
        if let Some(sm) = def_source_map {
            let (start, end) = sm.span_to_position(&info.definition.span).unwrap();
            return Ok(Some(GotoDefinitionResponse::Scalar(Location {
                uri: uri_from_file_id(&info.definition.file_id, &db.file_map),
                range: Range {
                    start: Position { line: start.line as u32, character: start.column as u32 },
                    end: Position { line: end.line as u32, character: end.column as u32 },
                },
            })));
        }
    }

    Ok(None)
}
```

### 7.2 Find All References

Find all references queries the `ReferenceGraph` for all locations where the symbol under the cursor is used:

```rust
pub fn find_references(
    db: &AnalysisDatabase,
    params: &ReferenceParams,
) -> Result<Option<Vec<Location>>, ()> {
    let uri = &params.text_document_position.text_document.uri;
    let position = params.text_document_position.position;

    let file_id = db.file_map.read().unwrap().get_by_uri(uri)?;
    let source_map = db.source_maps.read().unwrap().get(&file_id)?;
    let offset = source_map.line_col_to_offset(LineCol {
        line: position.line as usize,
        column: position.character as usize,
    })?;

    // Find the symbol name at this location
    let symbol_index = db.symbol_index.read().unwrap();
    let symbol = symbol_index.lookup_by_location(file_id, offset);

    if let Some(info) = symbol {
        let ref_graph = db.reference_graph.read().unwrap();
        let references = ref_graph.find_references(&info.name);

        let locations: Vec<Location> = references.iter()
            .filter_map(|r| {
                let sm = db.source_maps.read().unwrap().get(&r.file_id)?;
                let (start, end) = sm.span_to_position(&r.span)?;
                Some(Location {
                    uri: uri_from_file_id(&r.file_id, &db.file_map),
                    range: Range {
                        start: Position { line: start.line as u32, character: start.column as u32 },
                        end: Position { line: end.line as u32, character: end.column as u32 },
                    },
                })
            })
            .collect();

        return Ok(Some(locations));
    }

    Ok(None)
}
```

### 7.3 Rename

Rename computes all references to the symbol under the cursor and produces a `WorkspaceEdit` that renames every occurrence. The rename is text-based (find-and-replace of the symbol name in all reference locations) because the Glyim language does not have a refactoring API. The rename provider validates that the new name is a valid identifier and does not conflict with an existing symbol in the same scope.

```rust
pub fn rename(
    db: &AnalysisDatabase,
    params: &RenameParams,
) -> Result<Option<WorkspaceEdit>, ()> {
    let new_name = &params.new_name;

    // Validate the new name
    if !is_valid_identifier(new_name) {
        return Err(()); // LSP will show an error message
    }

    let uri = &params.text_document_position.text_document.uri;
    let position = params.text_document_position.position;

    // Find all references (same logic as find_references)
    let references = find_references_internal(db, uri, position)?;

    // Group by file
    let mut changes: HashMap<Url, Vec<TextEdit>> = HashMap::new();
    for reference in references {
        let edit = TextEdit {
            range: reference.range,
            new_text: new_name.clone(),
        };
        changes.entry(reference.uri).or_default().push(edit);
    }

    Ok(Some(WorkspaceEdit {
        changes: Some(changes),
        ..Default::default()
    }))
}
```

```rust
/// Unique identifier for a lint rule.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash, Serialize, Deserialize)]
pub struct LintId(pub &'static str);

/// Metadata for a lint rule.
#[derive(Debug, Clone)]
pub struct LintDescriptor {
    /// The lint's unique ID (e.g., "unused_function").
    pub id: LintId,
    /// Human-readable name.
    pub name: &'static str,
    /// Description of what the lint checks.
    pub description: &'static str,
    /// Default severity level.
    pub default_severity: Severity,
    /// Whether the lint can be suppressed with an attribute.
    pub allow_attribute: bool,
    /// The lint group this belongs to (e.g., "style", "correctness", "complexity").
    pub group: LintGroup,
}

#[derive(Debug, Clone, Copy, PartialEq, Eq)]
pub enum LintGroup {
    Correctness,
    Style,
    Complexity,
    Performance,
    Deprecated,
}

/// A single lint diagnostic.
#[derive(Debug, Clone)]
pub struct LintDiagnostic {
    /// The lint that was triggered.
    pub lint_id: LintId,
    /// The severity of this instance.
    pub severity: Severity,
    /// The source location.
    pub span: Span,
    /// Human-readable message.
    pub message: String,
    /// Suggested fix (for code actions).
    pub suggestion: Option<LintSuggestion>,
}

#[derive(Debug, Clone)]
pub struct LintSuggestion {
    /// The replacement text.
    pub replacement: String,
    /// The span to replace.
    pub span: Span,
    /// Description of the suggestion.
    pub label: String,
}

/// The lint registry, containing all registered lints.
pub struct LintRegistry {
    lints: HashMap<LintId, LintDescriptor>,
}

impl LintRegistry {
    pub fn new() -> Self {
        let mut registry = Self { lints: HashMap::new() };

        // Register built-in lints
        registry.register(LintDescriptor {
            id: LintId("unused_function"),
            name: "Unused Function",
            description: "Function is defined but never called",
            default_severity: Severity::Warning,
            allow_attribute: true,
            group: LintGroup::Correctness,
        });
        registry.register(LintDescriptor {
            id: LintId("unused_variable"),
            name: "Unused Variable",
            description: "Variable is bound but never used",
            default_severity: Severity::Warning,
            allow_attribute: true,
            group: LintGroup::Correctness,
        });
        registry.register(LintDescriptor {
            id: LintId("unnecessary_mut"),
            name: "Unnecessary Mutability",
            description: "Variable is declared mutable but never reassigned",
            default_severity: Severity::Warning,
            allow_attribute: true,
            group: LintGroup::Style,
        });
        registry.register(LintDescriptor {
            id: LintId("dead_code"),
            name: "Dead Code",
            description: "Code is unreachable",
            default_severity: Severity::Warning,
            allow_attribute: true,
            group: LintGroup::Correctness,
        });
        registry.register(LintDescriptor {
            id: LintId("complex_function"),
            name: "Complex Function",
            description: "Function has too many branches (cyclomatic complexity > 10)",
            default_severity: Severity::Hint,
            allow_attribute: true,
            group: LintGroup::Complexity,
        });

        registry
    }

    pub fn register(&mut self, descriptor: LintDescriptor) {
        self.lints.insert(descriptor.id, descriptor);
    }

    pub fn get(&self, id: LintId) -> Option<&LintDescriptor> {
        self.lints.get(&id)
    }

    pub fn all_lints(&self) -> impl Iterator<Item = &LintDescriptor> {
        self.lints.values()
    }
}
```

## 10. Partial Diagnostic Collection

### 10.1 Current Error Handling

The current pipeline in `compile_source_to_hir()` bails on the first failing phase:

```rust
if !parse_output.errors.is_empty() {
    return Err(PipelineError::Parse(parse_output.errors));
}
// ...
let type_result = TypeChecker::check(&hir, &interner);
if let Err(type_errors) = type_result {
    return Err(PipelineError::TypeCheck(type_errors));
}
```

This means parse errors prevent type checking, and type errors prevent optimization and code generation. For an IDE, this is unacceptable: the user wants to see all diagnostics at once, not just the first category.

### 10.2 Partial Compilation Pipeline

Phase 7 introduces a `PartialCompilation` mode that collects diagnostics from all phases without bailing:

```rust
// crates/glyim-compiler/src/pipeline.rs (extended)

pub struct PartialCompilation {
    /// Parse errors (may be non-empty even if we proceed).
    pub parse_errors: Vec<glyim_parse::ParseError>,
    /// Type errors (may be non-empty even if we proceed).
    pub type_errors: Vec<glyim_typeck::TypeError>,
    /// Lint diagnostics.
    pub lint_diagnostics: Vec<glyim_lint::LintDiagnostic>,
    /// The HIR, if parsing and lowering succeeded (even with type errors).
    pub hir: Option<Hir>,
    /// The typed HIR, if type checking succeeded (possibly with errors).
    pub typed_hir: Option<CompiledHir>,
}

pub fn compile_partial(
    source: &str,
    path: &Path,
    config: &PipelineConfig,
) -> PartialCompilation {
    let mut result = PartialCompilation::default();

    // Phase 1: Parse (always proceeds, collects errors)
    let parse_output = glyim_parse::parse(&expanded_source);
    result.parse_errors = parse_output.errors.clone();

    // If parse errors are fatal (e.g., unexpected EOF), stop here
    if has_fatal_parse_errors(&parse_output.errors) {
        return result;
    }

    // Phase 2: Lower to HIR (proceeds even with non-fatal parse errors)
    let hir = glyim_hir::lower_with_declarations(&parse_output.ast, ...);
    result.hir = Some(hir.clone());

    // Phase 3: Type check (proceeds, collects type errors)
    let type_result = TypeChecker::check(&hir, &interner);
    match type_result {
        Ok(()) => { /* no type errors */ }
        Err(errors) => {
            result.type_errors = errors.clone();
            // Still produce a partial typed HIR for IDE features
        }
    }

    // Phase 4: Lint (always proceeds if we have a HIR)
    result.lint_diagnostics = glyim_lint::lint(&hir, &interner);

    result
}
```

The `compile_partial()` function is used by the LSP server's analysis driver, while the CLI continues to use `compile_source_to_hir()` (which bails on errors, as appropriate for batch compilation).

```rust
pub fn convert_all(
    file_id: FileId,
    source_map: &SourceMap,
    partial: &PartialCompilation,
) -> Vec<Diagnostic> {
    let mut diagnostics = Vec::new();

    // Parse errors
    for error in &partial.parse_errors {
        diagnostics.push(convert_parse_error(file_id, source_map, error));
    }

    // Type errors
    for error in &partial.type_errors {
        diagnostics.push(convert_type_error(file_id, source_map, error));
    }

    // Lint diagnostics
    for lint in &partial.lint_diagnostics {
        diagnostics.push(convert_lint(file_id, source_map, lint));
    }

    diagnostics
}

fn convert_parse_error(
    file_id: FileId,
    source_map: &SourceMap,
    error: &ParseError,
) -> Diagnostic {
    let span = error.span(); // Returns (usize, usize) — to be migrated to Span
    let lsp_span = Span {
        file_id,
        start: span.0,
        end: span.1,
        expansion_id: None,
    };
    let (start, end) = source_map.span_to_position(&lsp_span).unwrap();

    Diagnostic {
        range: Range {
            start: Position { line: start.line as u32, character: start.column as u32 },
            end: Position { line: end.line as u32, character: end.column as u32 },
        },
        severity: Some(DiagnosticSeverity::ERROR),
        source: Some("glyim".to_string()),
        message: format_parse_error(error),
        ..Default::default()
    }
```

## 14. Error Handling & Recovery

### 14.1 Analysis Failures

If the analysis driver fails to compile a file (parse error, type error, etc.), it does not crash or lose the previous analysis state. Instead, it:

1. Reports the error diagnostics to the editor.
2. Retains the previous `SymbolIndex`, `ReferenceGraph`, and `TypeMap` entries for the file (stale but better than empty).
3. Marks the file's database entries as "stale" so that LSP responses include a note that the analysis is out of date.
4. Reattempts the analysis on the next file change.

### 14.2 Server Crashes

The language server is designed to be resilient to internal panics. The analysis driver runs in a separate task that is supervised by the main server task. If the analysis driver panics, the server:

1. Logs the panic with a full backtrace.
2. Restarts the analysis driver.
3. Sends a `window/showMessage` notification to the editor indicating that analysis was restarted.

### 14.3 Large Workspaces

For large workspaces with many files, the analysis driver uses a priority queue:

1. **High priority:** Files that are currently open in the editor (analyzed immediately on change).
2. **Medium priority:** Files that were recently open (analyzed when idle).
3. **Low priority:** Files in the workspace that have never been opened (analyzed in the background, on demand).

This ensures that the user's active editing experience is always responsive, even in large workspaces.

## 20. Migration Strategy

### 20.1 Span Migration (Feature-Flagged)

The `Span` struct extension is migrated in two steps:

1. **Add `file_id: Option<FileId>` (non-breaking):** All existing code creates `Span` with `file_id: None`. The `SourceMap` can handle both old and new spans. This is merged first, and all crates compile.

2. **Make `file_id: FileId` (breaking):** After all call sites are updated to pass `FileId`, the `Option` wrapper is removed. This is a workspace-wide refactor in a single PR.

### 20.2 LSP Server Rollout

1. **Phase 7C:** The LSP server is functional but only supports text synchronization and diagnostics. The `glyim lsp` command starts the server, and editors can connect, but navigation features are not yet available.

2. **Phase 7D:** Completion, hover, and navigation features are added incrementally. Each feature is tested independently with the VS Code extension.

3. **Phase 7E:** The formatter and lints are integrated. These are independent of the navigation features and can be rolled out in parallel.

4. **Phase 7F:** Editor configurations and documentation are finalized. The VS Code extension is packaged and published to the Open VSX Registry.

### 20.3 Formatter Migration

The formatter replaces the `glyim-fmt` placeholder, but the `--check` mode behavior changes: previously, `--check` was a trivial identity comparison; now it checks whether the formatted output differs from the input. CI configurations that use `glyim fmt --check` will start catching formatting issues for the first time. This is a deliberate improvement, but it may require updating CI scripts.

## 21. Success Criteria

1. **Diagnostics in < 50ms:** After editing a single function in a 1000-line file, the editor shows updated diagnostics within 50ms of the last keystroke.

2. **Goto definition works across files:** Clicking a call to a function defined in a dependency package navigates to the correct file and line.

3. **Completion provides relevant suggestions:** Typing a function name prefix in a 50-function file returns the correct completion within 20ms.

4. **Hover shows type and documentation:** Hovering over any function, variable, or type shows its type signature, documentation, and effect annotation.

5. **Find all references is accurate:** Finding references to a function returns all call sites across all open files, with no false positives.

6. **Formatter is idempotent:** Running the formatter on its own output produces the same result.

7. **Lint diagnostics appear in the editor:** Configurable lint warnings are shown as inline diagnostics with quick-fix suggestions.

8. **VS Code extension is functional:** The extension starts the language server, provides syntax highlighting, and all LSP features work out of the box.

9. **Server survives errors:** A malformed source file does not crash the language server; it reports diagnostics and continues serving other files.

10. **Memory usage is bounded:** The language server uses less than 500MB for a 100-file workspace with lazy analysis.
