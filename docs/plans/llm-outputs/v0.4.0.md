# v0.4.0 Phase 1: Foundation Types — Corrected Implementation Plan

> **For agentic workers:** REQUIRED: Use superpowers:subagent-driven-development (if subagents available) or superpowers:executing-plans to implement this plan. Steps use checkbox (`- [ ]`) syntax for tracking.

**Goal:** Add `HirType::Generic`, `HirType::Tuple`, `HirPattern::Tuple`, promote `ExprId` to a proper newtype, and add `type_params` to `HirFn`. Optionally restructure `node.rs` → `node/` module. Zero behavior change — all existing tests pass unchanged.

---

## What the original plan got wrong

The original plan was written against a speculative v0.3.0 codebase that didn't match reality. Here's every error, so you can see what changed:

| Original plan claim | Actual codebase state | Impact |
|---|---|---|
| `node.rs` is ~55 LOC with 9 HirExpr variants | `node.rs` is ~120 LOC with **17** HirExpr variants (IntLit, FloatLit, BoolLit, StrLit, Ident, UnitLit, Binary, Unary, Block, If, Println, Assert, As, Match, FieldAccess, StructLit, EnumVariant) | Plan's `expr.rs` dropped 8 variants — would break compilation |
| `HirType`, `HirPattern`, `ExprId` don't exist yet | All three already exist in `types.rs` | Plan created duplicate definitions that would conflict |
| `HirItem`, `Hir`, struct/enum defs don't exist yet | All exist in `item.rs` and `node.rs` | Plan created duplicate types with different names (`HirStructDef` vs `StructDef`) |
| `lower()` takes `(&Ast, &Interner)` | Takes `(&Ast, &mut Interner)` — interner is mutated during lowering (TryExpr, SomeExpr, etc. intern new symbols) | Plan's replacement lower.rs would fail to compile |
| `hir.fns` field exists in `Hir` | Already `hir.items: Vec<HirItem>` | Plan's codegen change was already done |
| codegen iterates `hir.fns` | Already iterates `hir.items` with match on `HirItem::Fn/Struct/Enum/Extern` | Plan's codegen change was already done |
| `item.rs` doesn't exist | Already exists with `HirItem`, `StructDef`, `EnumDef`, `ExternBlock`, `ExternFn`, `FnSig`, `StructField`, `HirVariant` | Plan created a competing `item.rs` with different type names |
| `types.rs` doesn't exist | Already exists with `HirType` (11 variants), `HirPattern` (13 variants), `ExprId` (type alias) | Plan created duplicate types |
| Add `glyim-diag` dependency for `Span` in item types | `item.rs` has no `Span` fields; adding them is unnecessary for Phase 1 | Would add a dependency for no reason |
| Plan's `lower.rs` replacement handles 8 expression kinds | Actual `lower.rs` handles **22** expression kinds (all the v0.3.0 additions) | Plan's replacement silently dropped FloatLit, BoolLit, UnitLit, StructLit, Match, EnumVariant, FieldAccess, As, MacroCall, TryExpr, Some/None/Ok/Err, Pointer |
| `HirImplDef` needed | No parser support for impl blocks exists; adding the type now creates dead code | Defer to Phase 8 |
| Split into `node/` directory with 7 files | Files are all under 150 LOC; splitting adds navigation cost with no clarity gain | Optional — include as separate task |

**1 optional refactor:**
5. Split `node.rs` → `node/` module (purely structural, no behavior change)

## Chunk 2: ExprId Newtype Promotion

### Task 3: Convert ExprId from type alias to newtype

**Files:** `crates/glyim-hir/src/types.rs`, `crates/glyim-typeck/src/lib.rs`

- [ ] **Step 1: Replace type alias with newtype in types.rs**

Change:
```rust
pub type ExprId = u32;
```
To:
```rust
#[derive(Debug, Clone, Copy, PartialEq, Eq, Hash)]
pub struct ExprId(u32);

impl ExprId {
    pub fn new(id: u32) -> Self {
        Self(id)
    }
    pub fn as_usize(self) -> usize {
        self.0 as usize
    }
}
```

- [ ] **Step 2: Update typeck/lib.rs — field type and fresh_id()**

In `TypeChecker`, change:
```rust
    next_expr_id: ExprId,
```
This stays the same type-wise (ExprId is still Copy).

Change `fresh_id()` from:
```rust
    fn fresh_id(&mut self) -> ExprId {
        let id = self.next_expr_id;
        self.next_expr_id += 1;
        self.expr_types.push(HirType::Never);
        id
    }
```
To:
```rust
    fn fresh_id(&mut self) -> ExprId {
        let id = self.next_expr_id;
        self.next_expr_id = ExprId::new(id.as_usize() as u32 + 1);
        self.expr_types.push(HirType::Never);
        id
    }
```

Change `set_type()` from:
```rust
    fn set_type(&mut self, id: ExprId, ty: HirType) {
        self.expr_types[id as usize] = ty;
    }
```
To:
```rust
    fn set_type(&mut self, id: ExprId, ty: HirType) {
        self.expr_types[id.as_usize()] = ty;
    }
```

Change `new()` from:
```rust
            next_expr_id: 0,
```
To:
```rust
            next_expr_id: ExprId::new(0),
```

- [ ] **Step 3: Add tests for ExprId newtype**

In `types.rs` test module:

## What this unblocks for later phases

- **Phase 2 (Parsing):** Can parse `struct Node<T> { value: T }` and produce `HirStructDef` with `type_params: [T]` — but `StructDef` in `item.rs` doesn't have a `type_params` field yet. That gets added when Phase 2 parsing is implemented.
- **Phase 3 (Type Checker):** Can instantiate `HirType::Generic(sym, vec![HirType::Int])` during monomorphization.
- **Phase 6 (Tuples):** `HirType::Tuple` and `HirPattern::Tuple` are ready for tuple syntax and codegen.
- **Phase 1 continued:** The type map (`expr_types: Vec<HirType>` indexed by `ExprId`) now uses a proper newtype, making it impossible to accidentally mix up expression IDs with other `u32` values.

## What this does NOT do (intentionally)

- Does NOT add `type_params` to `StructDef` or `EnumDef` — those are in `item.rs` and will get the field when Phase 2 adds generic syntax parsing
- Does NOT add `HirImplDef` — no parser support exists, defer to Phase 8
- Does NOT add `ExprId` field to `HirExpr` — defer to when the type map needs to be persisted (Phase 3 two-pass restructure)
- Does NOT add `glyim-diag` dependency to `glyim-hir` — no `Span` fields needed yet
- Does NOT restructure `item.rs` or `types.rs` — both are under 150 LOC, well-organized, and don't need splitting
