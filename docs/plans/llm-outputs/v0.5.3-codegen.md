Here are the exact required changes in `glyim-codegen-llvm` after implementing the Monomorphizer v2. 

The overarching theme is **removing defensive hacks**. Because the new monomorphizer guarantees fully concrete types and correctly mangled names, the codegen no longer needs `type_overrides` patching, `Generic` type resolution, string-based function lookup fallbacks, or on-the-fly monomorphization caches.

Here is the breakdown of required changes by file/component:

---

### 1. Core API & Struct Changes (`codegen/mod.rs` & `lib.rs`)

The codegen struct and entry points must be updated to accept the new `MonoResult` format.

**Remove `type_overrides` and `mono_cache`:**
Delete `type_overrides: HashMap<ExprId, HirType>` and `mono_cache: RefCell<HashMap<...>>>` from the `Codegen` struct. 

**Add `expr_types: Vec<HirType>`:**
Replace them with a single `expr_types: Vec<HirType>` field.

```rust
// BEFORE (Old Codegen struct)
pub struct Codegen<'ctx> {
    // ...
    pub merged_types: Vec<HirType>,
    pub type_overrides: HashMap<ExprId, HirType>,
    pub mono_cache: RefCell<HashMap<(Symbol, Vec<HirType>), FunctionValue<'ctx>>>,
}

// AFTER (New Codegen struct)
pub struct Codegen<'ctx> {
    // ...
    pub expr_types: Vec<HirType>,
}
```

**Update `compile_items_to_objects` in `lib.rs`:**
Change the signature to accept `MonoResult` directly instead of separate HIR and type parameters.

```rust
// BEFORE
pub fn compile_items_to_objects(
    hir: &glyim_hir::Hir,
    merged_types: &[HirType],
    type_overrides: &HashMap<ExprId, HirType>,
    // ...
) -> Result<...>

// AFTER
pub fn compile_items_to_objects(
    mono_result: &glyim_hir::monomorphize::MonoResult,
    // ...
) -> Result<...> {
    // Use mono_result.hir and mono_result.expr_types directly
}
```

---

### 2. Type Resolution Simplification (`codegen/types.rs`)

This is the biggest cleanup. The codegen currently has complex fallback logic for `Generic` types. Since Mono v2 guarantees **Zero `Generic` types**, these arms should be replaced with panics/warnings to catch monomorphizer bugs, removing the dead/hacky logic.

**Simplify `resolve_struct_type`:**
```rust
// BEFORE
pub(crate) fn resolve_struct_type(&self, ty: &HirType) -> Option<StructType<'ctx>> {
    let struct_sym = match ty {
        HirType::Named(sym) => *sym,
        HirType::Generic(sym, args) => {
            // 30+ lines of string manipulation, interner lookups, 
            // fallback guessing the mangled name...
        }
        HirType::RawPtr(inner) => return self.resolve_struct_type(inner),
        _ => return None,
    };
    self.struct_types.borrow().get(&struct_sym).copied()
}

// AFTER
pub(crate) fn resolve_struct_type(&self, ty: &HirType) -> Option<StructType<'ctx>> {
    match ty {
        HirType::Named(sym) => self.struct_types.borrow().get(sym).copied(),
        HirType::RawPtr(inner) => self.resolve_struct_type(inner),
        
        // INVARIANT: Mono v2 converts all Generic to Named(mangled). 
        // If we hit this, the monomorphizer has a bug.
        HirType::Generic(sym, _) => {
            tracing::error!("BUG: Unresolved Generic '{}' reached codegen resolve_struct_type", 
                            self.interner.resolve(*sym));
            None
        }
        _ => None,
    }
}
```

**Simplify `hir_type_to_llvm`:**
Apply the exact same philosophy to the `Generic` arm here. Remove the recursive `resolve_struct_type` guessing.

```rust
// BEFORE
HirType::Generic(sym, args) => {
    if let Some(st) = self.resolve_struct_type(&HirType::Generic(*sym, args.clone())) {
        Some(st.into())
    } else {
        self.struct_types.borrow().get(sym).map(|st| (*st).into())
    }
}

// AFTER
HirType::Generic(sym, _) => {
    tracing::error!("BUG: Unresolved Generic '{}' reached codegen hir_type_to_llvm", 
                    self.interner.resolve(*sym));
    Some(self.i64_type.into()) // Fallback to prevent codegen crash, but error is logged
}
```

### 3. Expression Type Lookups (Throughout Codegen)

Everywhere the codegen looks up an expression's type, it currently checks `type_overrides` first, then falls back to `merged_types`. This two-step dance is gone. Now, it's a simple array lookup by `ExprId`.

```rust
// BEFORE
let ty = self.type_overrides.get(&id)
    .cloned()
    .unwrap_or_else(|| self.merged_types.get(id.as_usize()).cloned().unwrap_or(HirType::Int));

// AFTER
let ty = self.expr_types.get(id.as_usize())
    .cloned()
    .unwrap_or(HirType::Int);
```

### 4. Call Codegen Simplification (`codegen/string.rs` or `expr/mod.rs`)

The old codegen had a **4-level fallback chain** to find functions in the LLVM module because the old monomorphizer didn't consistently mangle callee names. Mono v2 mangles all generic call targets correctly, so the fallback chain can be destroyed.

**Remove the prefix/suffix scanning:**
```rust
// BEFORE (Massive fallback chain)
fn codegen_call(...) {
    let mut fn_val = cg.module.get_function(fn_name);
    if fn_val.is_none() {
        // Fallback 1: prefix scan
        // Fallback 2: fn_name__i64
        // Fallback 3: fn_name__i64_i64
        // Fallback 4: search any function starting with fn_name__
    }
    // ...
}

// AFTER (Direct lookup)
fn codegen_call(...) {
    let fn_val = match cg.module.get_function(fn_name) {
        Some(f) => f,
        None => {
            cg.report_error(format!("function '{}' not found (expected after monomorphization)", fn_name));
            return Some(cg.i64_type.const_int(0, false));
        }
    };
    // ... build call args ...
}
```

### 5. Method Call Simplification (`codegen/expr/mod.rs`)

The new monomorphizer desugars resolved `MethodCall` nodes into regular `Call` nodes (where `receiver` is the first argument). The codegen's `MethodCall` handler only needs to serve as a fallback for unresolvable methods (which shouldn't exist in standard mono'd code, but might exist in a fast-path compilation without mono).

**Remove dynamic string scanning:**
```rust
// BEFORE
HirExpr::MethodCall { receiver, method_name, args, .. } => {
    // ... complex logic trying to build mangled name ...
    // ... scanning module for prefix matches ...
    // ... falling back to raw method name ...
}

// AFTER
HirExpr::MethodCall { receiver, method_name, args, .. } => {
    // Mono v2 should have converted this to a Call.
    // This handler is a strict fallback.
    let receiver_val = codegen_expr(cg, receiver, fctx)?;
    
    // Try direct type-based mangling (just in case)
    let receiver_ty = cg.expr_types.get(receiver.get_id().as_usize()).cloned();
    let inner_ty = match &receiver_ty {
        Some(HirType::RawPtr(inner)) => inner.as_ref().clone(),
        Some(other) => other.clone(),
        None => HirType::Int,
    };

    let mangled_name = match &inner_ty {
        HirType::Named(type_name) => format!("{}_{}", cg.interner.resolve(*type_name), cg.interner.resolve(*method_name)),
        _ => return Some(cg.i64_type.const_int(0, false)), // Fail gracefully
    };

    if let Some(fn_val) = cg.module.get_function(&mangled_name) {
        // Build call with receiver as first arg...
    } else {
        cg.report_error(format!("Method '{}' not found", mangled_name));
        Some(cg.i64_type.const_int(0, false))
    }
}
```

### 6. Struct/Enum Registration (`codegen/mod.rs`)

Because `MonoResult.hir` now contains specialized structs/enums with mangled names (e.g., `Vec__i64`) as `HirItem::Struct`, the codegen's struct registration loop doesn't need `Generic` resolution anymore. It just iterates over the items and registers them by their `name` symbol directly.

```rust
// BEFORE
for item in &hir.items {
    if let HirItem::Struct(s) = item {
        // If s has type_params, maybe skip or try to register generically...
    }
}

// AFTER
for item in &mono_result.hir.items {
    if let HirItem::Struct(s) = item {
        // s.name is already mangled (e.g. "Vec__i64") and fields are concrete
        self.register_struct(s); 
    }
}
```

### Summary of Impact

By guaranteeing that Monomorphizer v2 produces a completely concrete HIR and `expr_types` vector, you are effectively deleting **hundreds of lines** of defensive, string-manipulation, and guessing code from the codegen. The codegen transforms from a "type-resolver" back into a pure "IR-translator".
