Looking across these devlogs, the same classes of problems recur at every version bump. Most aren’t deep compiler-theory issues—they’re mechanical, structural, and process ones that can be systematically avoided. Here are patterns that would prevent them.

---

### 1. **Make illegal states unrepresentable (or at least detect them immediately)**

**Problem pattern:** Missing match arms, missing imports, dead code, incomplete snapshots, `HirExpr::Return` added in codegen but not in HIR definition.

**Fix:**  
- **Use exhaustive matches everywhere** — never use wildcard `_ => {}` in a match on an enum that may grow. Deny `non_exhaustive_omitted_patterns` at the crate level.
- **Generate boilerplate.** For every new AST/HIR variant, a code-generation step or a macro can ensure that a new variant is automatically added to all relevant match sites. Even a simple `check_all_variants!` macro that forces you to list every case, updated via a script, would catch the “added in codegen but not in lowering” errors.
- **Deny `dead_code` and `unused_imports` at CI level.** The logs show that `float_ops` was dead code, and multiple tests were duplicated or unused. Treat warnings as errors in CI; they would have flagged the dead code immediately.

---

### 2. **Automate architecture invariants**

**Problem pattern:** Tier violations (codegen depending on typeck), cyclic crate dependencies, parser module visibility tangles.

**Fix:**  
- **Write a crate-dependency checker** (like `cargo-deny` but for your own DAG). The `glyim-*` crates already have a defined tier DAG. A CI step that parses `Cargo.toml` and asserts that `glyim-codegen` never depends on `glyim-typeck` would catch the near-mistake that forced JIT orchestration into the CLI crate.
- **Module visibility linting.** The parser modularisation required many `pub(crate)` exports; a script that checks that no internal modules expose more than intended (e.g., `pub` items that should be `pub(crate)`) would avoid accidental API leaks.

---

### 3. **Incremental compilation of the compiler itself is a risk – simulate it**

**Problem pattern:** Adding a new enum variant (e.g., `HirType::Generic`, `HirExpr::Match`) causes 50+ non-exhaustive match errors across 10 crates.

**Fix:**  
- **Add a “variant impact” test early.** After defining a new variant, run `cargo check` immediately within the crate where it’s defined, and fix all matches there, *then* propagate outward. The logs show cases where the variant was added in one place and other crates were fixed later, leading to cascading breakage.
- **Use a helper crate with a compile-time assertion.** For each `Hir*` enum, write a test that ensures every variant is handled in the key functions (e.g., `lower_expr`, `check_expr`, `codegen_expr`). That test fails at compile time, not at the end of a long editing session.

---

### 4. **Testing infrastructure must test itself**

**Problem pattern:** The UI test harness was broken for a whole version cycle. Snapshots silently emptied. `insta` updates swept real regressions under the rug.

**Fix:**  
- **Meta-tests for the test harness.** Add a test that *deliberately* produces a bad diagnostic and asserts that the UI test harness fails. If the harness is broken, this meta-test fails.
- **Lock snapshot review.** Never run `INSTA_UPDATE=always` blindly. Instead, run it and then commit only the snapshots that changed in expected ways. A CI step can check that no snapshot file is empty after `cargo test`.
- **Color-stripping detection.** The harness switch from ANSI to plain text broke 6 snapshots. An integration test that asserts the output of `glyim build` contains no ANSI escapes would have caught the mismatch immediately.

---

### 5. **Treat the file-size limit as a release gate, not a suggestion**

**Problem pattern:** Parser grew from 400 to 1,000+ lines, limit raised to 1,200 with a promise to refactor later; refactor never happened.

**Fix:**  
- **Enforce the limit at the granularity you want.** Instead of a single `MAX_LINES = 500` on `parser.rs`, enforce per-module limits. Split first, then add features: before adding `parse_match_expr`, move existing parsing into sub-modules so you never exceed the limit.
- **CI gate with auto-failure.** If a file exceeds the limit, the build fails *and prints the refactoring instructions*. The logs show the limit was raised manually and never restored; make it painful to raise without conscious effort (e.g., require a specific comment in the source explaining why the exception is granted).

---

### 6. **API boundaries must be explicit and versioned**

**Problem pattern:** Inkwell API changes between LLVM versions (typod `build_bit_cast`, `CallSiteValue` return type). The interner wasn’t accessible in `lower()`, requiring a signature change that rippled through 15 call sites.

**Fix:**  
- **Wrap unstable external APIs.** For LLVM/Inkwell, create a thin wrapper crate that exposes exactly the operations you use. When Inkwell changes, you fix it in one place.
- **Use newtypes for all semantic identifiers.** The logs show `ExprId` was promoted from `type ExprId = u32` to a newtype. Do this from the start for all IDs (Symbol, Span, etc.). It prevents accidental interchange and makes future refactors trivial.
- **Add a “required context” marker to functions.** For `lower()`, the lack of interner wasn’t caught until codegen. A comment like `// REQUIRES: &Interner` or a simple compile-time attribute (a trait that must be in scope) would surface missing arguments earlier.

---

### 7. **Platform-specific scripting is a bug source**

**Problem pattern:** `sed` incompatibility on macOS caused repeated iterations; Python eventually replaced it.

**Fix:**  
- **Use a cross-platform task runner.** The build script uses `sed` for file edits; replacing all such edits with a small Rust binary or a Python script (as was done eventually) avoids OS-dependent behavior. Add a CI runner on macOS to catch these early.
- **No inline `sed` in justfile.** All text transformations go through a checked-in script that is tested on both Linux and macOS.

---

### 8. **Late-discovered bugs are a symptom of missing integration tests for the pipeline contract**

**Problem pattern:** `parse_extern_type` returned the wrong type (unit) for a whole version; tuple field access GEP index was hardcoded to 0; struct field access picked an arbitrary struct type from the map. These were found only when adding new features.

**Fix:**  
- **Pipeline contract tests.** For each phase boundary (parse→lower, lower→typeck, typeck→codegen), write a test that feeds a deliberately complex source and asserts every HIR node and type annotation is correct. For example, `extern fn foo(x: *mut u8) -> i64` should produce an `ExternFn` with a `RawPtr` parameter and `Int` return type – a single test that would have caught the `parse_extern_type` returning `unit`.
- **Assert on shape, not snapshots.** Snapshots are great for regressions but poor at catching missing fields. Use explicit `assert_eq!` on key AST fields for critical constructs.

---

### 9. **Process: “Problems Hit” should become regression tests immediately**

The devlogs themselves are a goldmine, but the problems are fixed and forgotten.  
**Fix:**  
- **For every entry in “Problems Hit”, add a regression test *before* marking it resolved.** E.g., the `None` literal token check fix – a test that `None` is parsed correctly even when the next token is consumed. That test would have failed before the fix and now ensures it never regresses.
- **Tag these tests with `#[regression]`** so they can be easily identified and never removed.

---

### Summary of concrete actions

| Issue class | Concrete pattern |
|------------|------------------|
| Missing variant handling | Exhaustive match lint, auto-generation of boilerplate |
| Architectural violations | Crate-level dependency lint in CI, enforced tier DAG |
| Cascading breakage | Incremental propagation with immediate `cargo check` per crate |
| Broken test infrastructure | Meta-tests for harness, CI checks for non-empty snapshots |
| File size creep | Enforce per-module limits, fail CI with refactor instructions |
| External API instability | Thin wrapper crate, minimal API surface |
| Platform scripting bugs | Cross-platform task runner, macOS CI runner |
| Late-discovered pipeline bugs | Pipeline contract tests with explicit structure assertions |
| Recurring problems | Mandatory regression test for every resolved problem |
