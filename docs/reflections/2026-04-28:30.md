## 1. Make the compiler codebase refactoring‑resistant by *eliminating* manual variant propagation

**Recurring issue**  
Adding `Deref`, `MethodCall`, `AssignDeref`, `LetPat`, etc. instantly created non‑exhaustive match errors in every crate. This is the same pain as every previous version bump, just with new variant names.

**Stronger fix**  
*Generate* the match arms.  
Don’t rely on developers to remember to update 10 match sites. Write a proc‑macro or a `build.rs` that reads the `HirExpr`/`HirStmt`/`HirType` definitions and produces a macro like `for_all_variants!(handle_expr, |variant| { ... })`. The macro guarantees that forgetting a new variant is a *compile error* in a single central location, not in 15 separate files that silently compile with wildcards.

**New permanent guard**  
*Deny wildcard arms on key enums.* In the few true “catch‑all” spots, use `#[allow(non_exhaustive_omitted_patterns)]` with a comment. Everywhere else, let the compiler force you to visit every variant.

## 2. Hard‑wire the architectural DAG into CI, not just a README diagram

**Problems that re‑surface**  
The original answer flagged tier violations; the new log shows that JIT orchestration still had to be kept in the CLI crate to avoid a codegen‑on‑typeck dependency. The temptation to break the DAG is constant.

**Automation**  
Add a CI step (`cargo-deny` with custom rules, or a small script) that parses `Cargo.toml` and asserts that `glyim-codegen` does not depend on `glyim-typeck` or `glyim-typeck` on `glyim-parse`’s public items. Fail the build if the DAG is violated, even in a `[dev-dependencies]` or `#[cfg(test)]` block.

## 3. Pipeline contracts must be tested at every stage boundary—**not** just in integration

**New example**  
- `parse_extern_type` returned unit for a whole version cycle.
- Tuple field GEP was hardcoded to `0`.
- Struct field access picked an arbitrary type from the map.

These are exactly the kind of mistakes that a single “contract test” would have caught. We added some later, but the pattern was reactive, not proactive.

**Enforce it**  
For every `parse` → `lower` → `typeck` → `codegen` boundary, write a *canonical shape test*:
```rust
// Feed a deliberately complex input through parse and lower,
// then assert_eq! on every HIR field, not just snapshots.
assert_eq!(func.return_type, HirType::Int);  // not unit
assert_eq!(field_access.index, 1);           // not 0
```
These tests live next to the definitions and run on every `cargo test`. When a new variant is added, the test *must* be updated—which forces you to consider every field.

**Recurrence**  
The `sed` portability battle returned, requiring multiple iterations. The eventual fix (full file overwrites, heredocs, Python) worked, but the pain was avoidable.

## 5. Test infrastructure must be self‑testing and locked down

**Problems seen again**  
UI snapshots became empty, harness ran only parsing instead of the full pipeline, `INSTA_UPDATE=always` swept real regressions under the rug.

**Concrete guards**  
- **Meta‑test for the harness**: A test that deliberately produces a type error and verifies the UI harness captures it. If the harness is broken, CI fails.
- **Empty‑snapshot check**: A CI step that fails if any `.snap` file is empty (using `! -s` or a Rust script).
- **`INSTA_UPDATE=always` is never a CI command.** Instead, use `INSTA_UPDATE=new` and review every change. Add a `cargo test --review-snapshots` alias that diffs snapshots and requires manual acknowledgement.

## 6. Internal IR changes must trigger a full consumer audit

**New instance**  
Switching from `Let` to `LetPat` broke immutability enforcement, let‑mut tests, lowering tests, and an accidentally unmarked test. The change was made in lowering, but the consequences rippled unchecked.

**Pattern**  
1. **Co‑location of tests**: The unit tests for `lower_stmt` are updated *in the same commit* that changes the lowering. No separate “test update” phase.
2. **Behaviour‑oriented tests**: Instead of testing for specific enum variants (`assert!(matches!(stmt, HirStmt::LetPat))`), test for semantic properties (“after `let x = 10`, `x` is bound and immutable”). When the IR changes, the test still passes if the behaviour is preserved.
3. **Consumer‑check script**: When `HirStmt` is modified, a CI script detects it and runs the full test suite of every crate that depends on `glyim-hir`, *plus* a special test that verifies all downstream crates still compile against the changed types.

## 7. Explicit contracts for data structures that cross module boundaries

**New confusion**  
The `MethodCall` codegen mistakenly added the receiver as an extra argument because the parser already placed it in `args[0]`. The contract “receiver is the first element of `args`” was implicit.

**Fix**  
- **Document the shape inline**: On the `MethodCall` HIR variant, write a doc comment: `// args[0] is the receiver; do NOT add it again in codegen.`
- **Use a newtype to make the contract un‑missable**:
  ```rust
  struct MethodCallExpr {
      receiver: ExprId,
      method_name: Symbol,
      args: Vec<ExprId>,   // args does NOT include receiver
  }
  ```
  This would have prevented the codegen error entirely.

## 8. Leftover debug artifacts are a permanent risk—automate their removal

**Recurring**  
`eprintln!("----- LLVM IR -----")` left in `pipeline.rs` after debugging.

**Prevention**  
- **Deny `print`‑family macros in library crates.** Only the CLI binary may print. Use `#![deny(clippy::print_stdout)]` in all `glyim-*` libs.
- **Pre‑commit hook** that greps for `eprintln!`, `println!`, `dbg!` and refuses to commit unless an override comment is present.

## 9. Refactoring without unit‑test safety net causes regressions

**Example**  
Inlining `lower_assert_call` lost the message argument. This was spotted late because the assert‑lowering path had no dedicated unit test—only integration tests that passed by accident.

**Rule**  
*Every function extraction or inlining is immediately accompanied by a unit test that exercises the exact boundary.*  
If the assert lowering had its own test, the regression would have appeared at the moment of the refactor, not at the next full‑pipeline run.

## 10. Async runtime foot‑guns need linting

**New class of error**  
Mixing blocking and async runtimes caused “Cannot drop a runtime” panics. The fix was specific (wiremock + separate reqwest clients), but the root cause is a Rust‑ecosystem trap.

**Automate**  
- **Lint against mixing `reqwest::blocking` and `tokio::test` in the same crate.** A custom Clippy lint or a simple `grep` in CI would prevent the combination.
- **Use a dedicated `test-utils` crate** that provides the correct async test setup, hiding the complexity behind a single function.

## 1. Missing Variant Handling → Generate All Match Arms

**Symptom:** You add `HirExpr::Return`, `HirExpr::MethodCall`, or `HirType::Float` and get 15 “non-exhaustive match” errors across 10 crates.

**Solution:** Write a proc-macro that, given a `HirExpr` definition, auto-generates a "visit every variant" construct. `synstructure` ([docs](https://docs.rs/synstructure/)) is purpose-built for this: it lets you match on every enum variant and extract bindings to each field generically.

```rust
fn generate_lower_match(s: synstructure::Structure) -> TokenStream {
    s.each(|binding| {
        quote! { /* handle #binding */ }
    })
}
```

**Implementation steps:**
1. Add `synstructure` to a new `glyim-derive` crate.
2. Define one derive macro per key IR enum (e.g., `#[derive(GenerateLower)]` on `HirExpr`).
3. CI gate: add a test that asserts `matches!` on all variants of each enum if you still use manual `match` arms.
4. **Deny wildcard matches**: add `#![deny(clippy::wildcard_enum_match_arm)]` (nightly) or a manual CI grep for `_ => {}` in `lower`, `codegen`, `typeck`.

| Variant Added | Old Way (manual) | New Way (generated) |
| :--- | :--- | :--- |
| `HirExpr::MethodCall` | 15 compile errors in 10 crates | 1 source-of-truth trait updated, all arms generated |
| `HirType::Float` | Eq derive breaks, 8 files touched | Trait macro auto-handles `f64` special case |

## 2. Architectural Tier Violations → CI-Enforced Crate Dependency DAG

**Symptom:** `glyim-codegen` depends on `glyim-typeck`, breaking your 5-tier DAG. Previously fixed by moving JIT orchestration into `glyim-cli`.

**Solution:** `cargo-deny` + a custom CI script.
* `cargo-deny` ([Embark Studios](https://embarkstudios.github.io/cargo-deny/)) enforces allowed/disallowed crate dependencies. Its `[bans]` section can **deny specific crates from depending on each other**.
* **Custom script**: parse `Cargo.toml` files and check that `glyim-codegen` never lists `glyim-typeck` in `[dependencies]`.

**`deny.toml` snippet:**
```toml
[bans]
# Prevent codegen from depending on typeck
deny = [
    { crate = "glyim-typeck", reason = "Tier violation: codegen sits below typeck in the DAG" },
]
# Applies only to glyim-codegen via `exclude` in graph config
```

**Implementation steps:**
1. Add `deny.toml` at workspace root with tier rules.
2. Add `cargo deny check bans` to CI (`just check-arch` or GitHub Actions step).
3. Write a 20-line Rust/Python script that parses `Cargo.toml` dependencies and asserts the DAG—run as a CI gate.

## 3. Pipeline Contract Bugs → Shape-Assertion Tests at Every Boundary

**Symptom:** `parse_extern_type` returned `()` (unit) instead of `Option<TypeExpr>` for a full version cycle. Tuple field GEP was hardcoded to `0`. Struct field access picked an arbitrary type from the map.

**Solution:** **Contract tests**: feed a fixed input through one boundary (parse→lower, lower→typeck) and assert on the **exact shape** of every output field.

**Example contract test for parse→lower:**
```rust
#[test]
fn contract_extern_fn_type_must_not_be_unit() {
    let hir = lower("extern { fn foo(x: *mut u8) -> i64; }");
    let extern_fn = hir.items[0].as_extern_fn().unwrap();
    assert_eq!(extern_fn.params[0].ty, HirType::RawPtr(Box::new(HirType::Int)));
    assert_eq!(extern_fn.return_type, HirType::Int); // NOT unit!
    assert_eq!(extern_fn.name, "foo");
}
```

**Implementation steps:**
1. For each major feature (extern, struct lit, match, impl block), write one contract test that asserts every HIR field.
2. Run these tests on every `cargo test`—they're fast (no codegen needed).
3. CI gate: if a `HirExpr` variant is added but no contract test covers it, fail.

## 4. Cross-Platform Scripting Breakage → Ban `sed`/`awk`, Enforce macOS CI

**Symptom:** `sed -i ''` works on macOS, fails on Linux; `sed -i` vice versa. Repeatedly fixed by "several iterations" of find-and-replace.

**Solution:** 
* **Rule**: No `sed`, `awk`, or `grep -P` in any `justfile` recipe or build script.
* **Replacement**: Python 3 (pre-installed on both platforms) for all text transforms.
* **CI enforcement**: add a macOS CI runner that runs the full test suite and greps for banned commands in scripts.

**Implementation steps:**
1. Extract all text transformations from `justfile` recipes into `scripts/*.py`.
2. Add a `just lint-scripts` target that runs `grep -En "sed|awk|grep\s+-P" scripts/` and returns exit 1 if any found.
3. Add macOS CI runner (GitHub Actions `macos-latest`).

## 5. Snapshot Rot & Regressions → CI Guardrails on `insta`

**Symptom:** UI harness silently broke, snapshots were empty for a whole version. `INSTA_UPDATE=always` swept real regressions under the rug.

**Solution 1:** Use `INSTA_UPDATE=new` (never `always`) — this forces explicit review of new snapshots.
**Solution 2:** CI check: fail if any `.snap` file is empty.
**Solution 3:** Meta-tests for the harness itself.

**`justfile` target:**
```makefile
test-ui:
    INSTA_UPDATE=new cargo test --test ui
    @echo "Checking for empty snapshots..."
    @! find crates -name "*.snap" -empty | grep .
```

**Harness meta-test:**
```rust
#[test]
fn harness_captures_type_errors() {
    let output = run_ui_test("tests/ui/type_mismatch.g");
    assert!(output.contains("type mismatch"), "UI harness must capture type errors");
    assert!(!output.is_empty(), "UI harness must not produce empty output");
}
```

**Implementation steps:**
1. Update CI to use `INSTA_UPDATE=new`.
2. Add the empty-snapshot check to `just test` or CI.
3. Add the harness meta-test as a regular `#[test]`.
4. Use `cargo insta review` locally for review flow—never blind-accept.

## 6. Async Runtime Foot-Guns → Custom Clippy Lint or CI Grep

**Symptom:** `#[tokio::test]` mixed with `reqwest::blocking` caused "Cannot drop a runtime" panic.

**Solution:** A Clippy lint to detect blocking calls inside async contexts exists as a [proposal (#4427)](https://github.com/rust-lang/rust-clippy/issues/4427) and a working fork by `michaelsproul`. You can also use a lightweight CI grep:

```bash
# CI step: detect reqwest::blocking imports in test files
grep -r "reqwest::blocking" crates/*/tests/ && echo "ERROR: blocking reqwest in tests" && exit 1
```

**Implementation steps:**
1. Add the grep check to CI as a fast lint step.
2. For async tests, standardize on `wiremock` + `reqwest` (async) via a shared `test-utils` crate.
3. Once the Clippy lint stabilizes, enable it project-wide.

## 7. File Size Creep → Hard CI Gate with `loq` or Custom Script

**Symptom:** Parser grows from 400 to 1,000 lines, limit raised to 1,200 with a "refactor later" promise that never happens.

**Solution:** `loq` ([crates.io](https://crates.io/crates/loq_core)) is a file-size enforcement tool written in Rust that runs in CI. It can enforce per-module limits and even ratchet down over time.

**Alternative: Custom script (your existing `check_file_sizes.py` tightened):**
```python
MAX_LINES = {
    "parser.rs": 500,
    "parser/items.rs": 200,
    "parser/exprs.rs": 200,
}
# ... check and exit 1 if violated
```

**Implementation steps:**
1. Ship `loq` or a custom script in CI.
2. Make `just check-files` fail CI if any file exceeds its limit.
3. Never raise a limit without an `ADR` explaining why, and the ADR must include a refactoring plan with a deadline.

## 8. Copy-Paste Regressions → Dedicated Regression Tests

**Symptom:** Assert message regression, duplicated test functions, `#[test]` left off.

**Solution:** When you fix a "Problem Hit," **immediately** write a regression test tagged `#[regression]` before marking the issue resolved. This test must fail before the fix and pass after.

```rust
#[test]
#[regression] // tag for grouping
fn assert_keeps_message_after_inlining() {
    let output = compile("assert(true, \"msg\");");
    assert!(output.contains("msg"));
}
```

**Implementation steps:**
1. Add a convention: every "Problems Hit" entry gets a `#[regression]` test in the same PR as the fix.
2. Run `cargo test -- regression` as a dedicated CI step.
