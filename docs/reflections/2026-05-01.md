## 🎯 Recommendations

### 1. Replace Ad‑Hoc Debug Logging with Structured Tracing

**Problem:** Inserting `eprintln!` across multiple files led to unmatched braces, orphaned arguments, and file corruption (the “debug‑damage spiral”). Recovering required `git checkout` and re‑application of fixes.

**Recommendation:**  
- Integrate a structured logging crate like `tracing` (already included, judging by the mention of duplicate `tracing` initialization cleanup). Use `trace!`/`debug!` **everywhere** that you previously used `eprintln!`, guarded by `RUST_LOG` environment variables.
- Never inject temporary debug logging into more than one file at a time without committing or stashing first. If you need to trace across multiple crates, write a minimal single‑file reproducer (e.g., `cargo run --bin glyim -- ir <file>`) and rely on unit tests instead of manual `println!` debugging.
- Enable `tracing` spans early in the pipeline so that even unexpected panics produce a trace of the compiler phase in which they occurred.

### 2. Automate Test File Generation and Snapshot Management

**Problem:** Adding test parameters (`target: Option<&str>`) required a Python script to insert `, None` at dozens of call sites, and manual additions frequently caused unmatched braces and duplicate function definitions.

**Recommendation:**
- Use a snapshot‑testing library like `insta` (excellent for Rust) for all UI test and IR snapshot comparisons. `insta` automates updating `.snap` files and eliminates the need for manual snapshot files.
- For integration tests that call pipeline functions with many arguments, consider creating a parameter‑object builder (e.g., `Config::new().target("...").jit_mode()`) so that adding new fields does not require mass‑updating call sites.
- Split the monolithic integration test file into multiple modules by feature area (e.g., `tests/vec.rs`, `tests/strings.rs`, `tests/cross.rs`) using the standard `tests/` directory layout; this avoids single‑file corruption and reduces merge conflicts.

### 3. Improve Borrow‑Checker Ergonomics with Upfront Data Collection

**Problem:** The monomorphizer repeatedly faced immutable/mutable borrow conflicts when iterating over maps while needing to intern symbols.

**Recommendation:**
- Adopt a “collect then act” pattern: in functions that need to both iterate over `fn_specs` and call `self.interner.intern()`, collect the data needed for the iteration into a `Vec` (or other owned container) first. This can be done by cloning keys or values as you already started doing, but do it systematically at the top of the function.
- Consider refactoring `Interner` to allow infallible cloning/intern of symbols without a mutable reference, e.g., by pre‑interning all known symbols during an earlier phase.
- Use Rust’s `entry` API more aggressively to minimise double-lookups that force temporary borrows.

### 4. Automate Syntax‑Kind Definitions and AST/HIR Boilerplate

**Problem:** Adding `While` or `Amp` required manually updating `SyntaxKind::COUNT` and corresponding test assertions, leading to out‑of‑sync failures.

**Recommendation:**
- Use a macro or code generator for AST and HIR node definitions. For example, the `Rowan` or `ungrammar` approach can auto‑generate `SyntaxKind` enums with correct `COUNT` values. Alternatively, define `SyntaxKind` using a `macro_rules!` list that counts variants automatically (e.g., `seq!` from `seq-macro`).
- Add compile‑time assertions where `match` exhaustiveness is critical, so that missing arms are caught by the compiler, not by test failures.
- Dedicate a CI check that runs after adding any new language feature to verify that all enumerated counts match.

### 5. Prefer Subprocess‑Based Test Isolation Over `setjmp`/`longjmp`

**Problem:** Tests that call `abort()` crash the test runner, forcing the use of fragile, platform‑dependent `setjmp`/`longjmp` hacks.

**Recommendation:**
- Run tests expecting a fatal exit (like assertion failures or stack overflows) in separate processes using `std::process::Command` or a helper like `assert_cmd`. This is the standard approach for testing “should panic” scenarios in compilers and systems languages (e.g., Rust’s own `compiletest`).
- The JIT‑based tests can be kept fast by spawning a child that runs only the one test case, but in‑process `setjmp` is not worth the maintenance burden and undefined behaviour risks.
- Remove the custom abort handler injection from the JIT pipeline once subprocess execution is adopted; this simplifies the `Codegen`/pipeline API.

### 6. Stabilise the Method‑Call Monomorphizer with a Dedicated Integration Test Suite

**Problem:** The `MethodCall`‑to‑`Call` rewrite is fragile, and the “chained rebinding” pattern (`let v = v.push(...)`) is still not working. This is the largest remaining language gap.

**Recommendation:**
- Create a separate `tests/monomorphize/` directory with small Glyim programs that specifically test:
  - Simple struct methods
  - Generic methods with multiple concrete instantiations
  - Chained calls (`v.push(1).push(2)`)
  - Methods with type annotations on the binding
- Drive the monomorphizer rewrite pass to pass every test, one at a time. This makes the problem tractable and prevents regressions when other features are added.
- Consider temporarily switching to a **lower‑IR monomorphization** approach where generics are erased early and methods are resolved to concrete functions before HIR lowering, if the current AST‑level rewrite continues to be error‑prone. The existing `VecI64` wrapper already proves this can work.

### 7. Enhance Build System & CI for Cross‑Compilation and Performance

**Problem:** Cross‑compilation was added but there are only 4 tests; it’s easy to break. ThinLTO was added but no benchmarks exist.

**Recommendation:**
- Set up a CI matrix that builds for `aarch64-unknown-linux-gnu` (or a local `qemu`-based smoke test) on every commit.
- Add a benchmark suite (using `criterion`) that measures compiled code performance for numeric loops and collection operations, comparing builds with and without ThinLTO/PGO flags. This catches regressions from linker changes like the `RelocMode::Default` fix.

### 8. Formalize ADRs and Project Documentation

**Problem:** ADR numbers are assigned chronologically but reside only inside devlog entries. Their context may be lost later.

**Recommendation:**
- Maintain a `docs/adr/` directory in the repository with one file per decision, including date, status, and context. Link to the ADR from the devlog instead of inlining the full text.
- Keep a living document of “remaining gaps” so that half‑implemented features like native generic `Vec<T>` (raw‑pointer based) or `vec.g` codegen hang are not forgotten.
