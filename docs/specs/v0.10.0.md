# Glyim v0.10.0 Specification Suite: The Production Release

This document suite constitutes the fully fledged specification for Glyim v0.10.0. It applies evidence-based requirements engineering, architecture documentation, and behavioral specification practices, structured across five specification levels to ensure strategic alignment, stakeholder satisfaction, technical feasibility, and verifiable delivery.

---

## Document 1: Vision & Strategic Alignment (Level 0)

**Document ID:** GLYIM-V100-VISION-001  
**Version:** 1.0  
**Status:** Approved  
**Author:** Glyim Core Team  

### 1.1 Vision & Elevator Pitch
**Vision:** To make Glyim the safest, most observable, and most integrable language for enterprise production environments.  
**Elevator Pitch (Moore Template):** For systems engineers and SREs dissatisfied with the friction of integrating Rust with legacy C++ or the lack of native supply-chain security in Go, Glyim is a production-grade systems language. Unlike its competitors, Glyim pairs macro-driven, memory-safe C/C++ FFI with built-in OpenTelemetry tracing, zero-overhead async profiling, and native SLSA-compliant build provenance—eliminating the gap between writing a prototype and shipping to a secured Kubernetes cluster.

### 1.2 Problem & Business Context
Glyim v0.9.0 successfully eliminated the learning curve. However, empirical evidence from enterprise adoption patterns (e.g., early Rust in fintech, Go in cloud-native) shows that documentation alone does not drive enterprise procurement. The current state prevents production deployment due to three hard blockers: (1) calling existing C/C++ libraries requires brittle, unsafe manual bindings; (2) observing async workflows in production requires third-party, high-overhead instrumentation; and (3) compliance teams block deployment because the language toolchain lacks cryptographic supply-chain provenance and SBOM generation. Without v0.10.0, Glyim remains an island.

### 1.3 Target Users (Target Group)
1. **The Systems Integrator:** Backend engineer who must wrap OpenSSL, libcurl, or internal C++ payment processing libraries. Requires safety, not raw pointers.
2. **The Site Reliability Engineer (SRE):** On-call engineer debugging latency spikes in a distributed Glyim microservice. Requires async-aware flamegraphs and OTel span context.
3. **The Security/Compliance Engineer:** Professional responsible for ensuring software meets US EO 14028 (SBOMs) and internal SLSA Level 3 requirements.

### 1.4 Desired Outcomes & Success Metrics (OKRs)
* **Objective 1: Eliminate FFI friction.**
  * **KR 1:** A user can safely wrap a complex C++ class (e.g., a custom database client) using `glyim-ffi` in under 50 lines of Glyim code, with zero `unsafe` blocks.
  * **KR 2:** `glyim-ffi` macro expansion time adds <10% to clean build times.
* **Objective 2: Make async Glyim inherently observable.**
  * **KR 1:** The `glyim::tracing` library adds <2% runtime overhead when tracing is disabled, and <5% when enabled.
  * **KR 2:** `glyim profile` generates an SVG flamegraph that accurately maps OS threads back to logical async tasks >99% of the time.
* **Objective 3: Achieve default security compliance.**
  * **KR 1:** `glyim build` outputs a valid SPDX 2.3 JSON SBOM natively, without third-party CLI tools.
  * **KR 2:** 100% of official Glyim toolchain binaries published to the registry include SLSA v1.0 provenance signatures.

### 1.5 Goals and Non-Goals (Scope / Anti-scope)
**Goals (P0/P1):**
* Implement `glyim-ffi` proc-macro for C/C++ header parsing and safe wrapper generation.
* Ship first-party `glyim::tracing` library with automatic async span context.
* Ship `glyim profile` CLI tool with async-aware flamegraph generation.
* Implement native SBOM generation (SPDX) in the build pipeline.
* Implement SLSA provenance generation and signing.

**Non-Goals (Anti-scope):**
* **No Objective-C/Swift Interop:** Focus strictly on C/C++ ABI, which covers 99% of low-level systems integration.
* **No Distributed Tracing Backend:** We provide the SDK (spans/metrics export); we are not building a Jaeger/Tempo replacement.
* **No Automatic CVE Patching:** `glyim audit` will report vulnerabilities; automatic PR generation is deferred.

## Document 2: Business & Stakeholder Requirements (Level 1)

**Document ID:** GLYIM-V100-BRS-001  

### 2.1 Stakeholders and User Classes
| User Class | Characteristics | Primary Needs | Disfavored Behaviors |
| :--- | :--- | :--- | :--- |
| **Integrator** | Expert in C/C++ memory models; skeptical of GC/FFI overhead. | Seamless ABI translation; mapping C++ exceptions to Glyim `Result`; no manual `malloc`/`free`. | Writing raw `extern "C"` wrappers and manually calculating struct padding. |
| **SRE** | Expert in eBPF, `perf`, OpenTelemetry; intolerant of "black box" runtimes. | Correlating a specific `.await` point to a CPU cycle; structured JSON logging tied to trace IDs. | Profilers that only show OS thread names for highly concurrent async runtimes. |
| **Compliance** | Driven by regulatory frameworks (NIST, EO 14028); risks career on supply chain attacks. | Machine-readable inventory of dependencies; cryptographic proof of build origin. | Manual Excel spreadsheets tracking library versions. |

### 2.2 Stakeholder Needs & Value Proposition
* **SN-001 (Integrator):** "I need to link against a proprietary C++ shared library without leaking memory if a C++ exception is thrown." -> *Value Prop: `glyim-ffi` automatic C++ exception catching and translation to `Err(GlyimError)`.*
* **SN-002 (SRE):** "I need to know which logical async task is burning CPU, not just that 'Thread 4' is busy." -> *Value Prop: `glyim profile` runtime injection that maps async state-machine pointers to task names.*
* **SN-003 (Compliance):** "I need to generate an SBOM for a containerized Glyim app before it hits the artifact registry." -> *Value Prop: `glyim build --sbom` outputs SPDX natively as part of the CI `docker build` step.*

### 2.3 Ubiquitous Language (Glossary Seed)
* **FFI (Foreign Function Interface):** The mechanism for calling C/C++ code from Glyim.
* **SBOM (Software Bill of Materials):** A nested inventory of software components (SPDX format).
* **SLSA (Supply-chain Levels for Software Artifacts):** A security framework ensuring artifact integrity and provenance.
* **Span:** A single operation within a trace (OpenTelemetry concept), automatically created at `.await` points.
* **Flamegraph:** A visualization of profiled software, showing which code paths are on-CPU.

## Document 3: Software Requirements Specification (Level 2)

**Document ID:** GLYIM-V100-SRS-001  
**Traceability:** Maps to Level 1 SN-001 through SN-003 and Level 0 OKRs.

### 3.1 Functional Requirements: Interop (FFI)

**FR-FFI-001: Header Parsing Macro**
* **Description:** The `glyim-ffi` procedural macro shall parse C/C++ header files specified as a string literal or file path.
* **Fit Criterion:** The macro must accurately translate C primitive types, structs, unions, and function pointers into Glyim equivalents without requiring an external `libclang` dependency at runtime (leveraging the existing Rowan parser infrastructure).

**FR-FFI-002: Safe Wrapper Generation**
* **Description:** The macro shall generate idiomatic Glyim wrapper functions that handle memory pinning, lifetime management, and null-pointer checks.
* **Fit Criterion:** A generated wrapper calling a C function that returns `NULL` must automatically return `Err(Error::NullPointer)` in Glyim. The user must not be required to write an `unsafe` block to call a wrapped C function.

**FR-FFI-003: C++ Exception Interception**
* **Description:** For C++ interop, the generated bindings shall intercept C++ exceptions at the ABI boundary.
* **Fit Criterion:** An unhandled C++ exception thrown in a wrapped C++ function must be caught, translated to a string error message, and returned as `Err(Error::CppException(msg))` in Glyim, preventing C++ stack unwinding from crossing into Glyim frames.

### 3.2 Functional Requirements: Observability

**FR-OBS-001: `glyim::tracing` Crate**
* **Description:** A first-party crate providing structured diagnostic events, span creation, and OpenTelemetry export capabilities (OTLP/HTTP).
* **Fit Criterion:** The crate must expose a `[instrument]` attribute macro to automatically create spans for function entry/exit.

**FR-OBS-002: Implicit Async Spans**
* **Description:** The Glyim async runtime shall natively integrate with `glyim::tracing`.
* **Fit Criterion:** Every `.await` point in an async function must implicitly attach the current execution context to the active OpenTelemetry span, without requiring the user to manually pass context objects.

**FR-OBS-003: `glyim profile` CLI Tool**
* **Description:** A CLI subcommand that executes a target binary, samples its execution, and outputs an SVG flamegraph.
* **Fit Criterion:** The tool must wrap Linux `perf` or eBPF under the hood. It must accept an `--async-aware` flag.

**FR-OBS-004: Async-to-Thread Mapping**
* **Description:** The profiler must resolve logical async tasks to OS threads.
* **Fit Criterion:** When viewing the `--async-aware` flamegraph, the user must see frames labeled with the async task name (e.g., `async fn process_request`) rather than the runtime's internal polling function (e.g., `epoll_wait` or `reactor::poll`).

### 3.3 Functional Requirements: Supply Chain Security

**FR-SEC-001: Native SBOM Generation**
* **Description:** The `glyim build` pipeline shall calculate the dependency graph and output an SBOM.
* **Fit Criterion:** Running `glyim build --sbom=spdx.json` must generate a file conforming to the SPDX 2.3 specification, including all transitive dependencies from `glyim.lock`, their versions, and cryptographic hashes (SHA256) of the source archives.

**FR-SEC-002: Dependency Auditing (`glyim audit`)**
* **Description:** A CLI command to check the current lockfile against a known vulnerability database.
* **Fit Criterion:** `glyim audit` must query an upstream API (hosted by the Glyim project), check all direct and transitive dependencies, and exit with code 1 if a CVE with severity >= "High" is found, printing the affected crate and CVE ID.

**FR-SEC-003: SLSA Provenance**
* **Description:** The build system shall generate SLSA provenance.
* **Fit Criterion:** `glyim build --provenance` must output a `intoto.jsonl` file containing the builder ID, build timestamps, source commit hash, and dependency hashes, formatted according to the SLSA v1.0 provenance specification.

### 3.4 Non-Functional Requirements (NFRs)

**NFR-PERF-001 (Tracing Overhead):**
* When `glyim::tracing` is configured to export nothing (disabled), the execution time of a compute-bound async application shall not increase by more than **2%** compared to a build without the tracing crate.

**NFR-PERF-002 (Profile Overhead):**
* Binaries executed under `glyim profile` shall exhibit a CPU overhead of **<10%** compared to native execution, measured using a standard microbenchmark suite.

**NFR-SEC-001 (Cryptographic Integrity):**
* SLSA provenance outputs must be signed using Sigstore (Fulcio certificate authority and Rekor transparency log) natively, without requiring the user to manage GPG keys.

## Document 4: Architecture & Design Specification (Level 3)

**Document ID:** GLYIM-V100-ARCH-001  

### 4.1 Context & System Boundary (C4 Level 1)
v0.10.0 extends the compiler and runtime with three new internal components and one CLI tool:
1. **FFI Macro Engine:** Compile-time component parsing C/C++ ASTs.
2. **Tracing Runtime:** Low-level hooks in the async scheduler.
3. **Profile Mapper:** eBPF/`perf` data processor.
4. **Security Pipeline:** Build-graph to SPDX/SLSA serializers.

### 4.2 Architecture Decision Records (ADRs)

**ADR-012: Rowan-based C/C++ Parsing over `libclang`**
* **Context:** We need to parse C/C++ headers inside the `glyim-ffi` proc-macro. Rust's `bindgen` uses `libclang`, which requires a heavy C++ toolchain dependency, causing massive CI bloat and cross-compilation pain for users.
* **Decision:** Implement a restricted, pragmatically scoped C/C++ parser using Glyim's existing Rowan infrastructure. We only parse what is necessary for FFI (structs, enums, function signatures, basic macros). We explicitly *do not* parse C++ template instantiations or complex C++ inheritance hierarchies initially.
* **Consequences:**
  * *(+)* Zero external C++ dependency for the macro.
  * *(+)* Deep integration with Glyim's error reporting (spans mapped directly to the C header).
  * *(-)* Will fail on heavily templated C++ (e.g., STL internals). Users will fall back to a manual `unsafe` block for <5% of edge cases.

**ADR-013: Thread-Local Async Context for Tracing**
* **Context:** OpenTelemetry requires a "Current Context" (a map of key-value span data) that propagates through call stacks. In async code, this context must survive across `.await` suspension points, which move the future between threads.
* **Decision:** Store the OTel Context as a field inside the Glyim Async Task state machine struct. When a task is polled, the runtime extracts the context from the task and stores it in a native OS Thread-Local variable. When the task yields, it is packed back into the task struct.
* **Consequences:**
  * *(+)* Zero-cost abstraction when tracing is disabled (the field is a zero-sized type).
  * *(+)* Perfectly accurate span correlation even if a task hops between 10 different OS threads.
  * *(-)* Adds slight memory overhead to every async task struct (one pointer-sized field).

**ADR-014: eBPF Trampoline Injection for Async Profiling**
* **Context:** Standard Linux `perf` profiles kernel stack traces. It does not understand Glyim's async state machine. To map a CPU tick on Thread 4 back to "Task A", we need a mapping table.
* **Decision:** The compiler will inject a lightweight trampoline function at the entry of every `poll()` function of an async state machine. This trampoline writes the logical Task ID and function name to a shared memory map (eBPF `BPF_MAP_TYPE_PERCPU_ARRAY`). The `glyim profile` CLI attaches an eBPF program that reads this map on `perf` events and synthesizes fake stack frames.
* **Consequences:**
  * *(+)* Accurate async profiling without modifying the Linux kernel.
  * *(+)* Overhead is negligible (one memory write per async poll).
  * *(-)* Linux-only initially (macOS `dtrace` support deferred to v0.11.0).

## Document 5: Behavioral Specification & Test Verification (Level 4/5)

**Document ID:** GLYIM-V100-BEHAV-001  
**Verification Method:** Automated Test (Unit for macros, Integration for eBPF/Security, Schema validation for SPDX/SLSA).

### 5.1 Specification by Example (SbE): Safe FFI Wrapping

**Feature:** `glyim-ffi` Safe Wrapper Generation

```gherkin
Scenario: Successfully wrap a C struct and function returning a pointer
  Given a C header file "calc.h" containing:
    """
    typedef struct { int x; int y; } Point;
    Point* create_point(int x, int y);
    void free_point(Point* p);
    """
  And the user writes a Glyim macro: `ffi_bridge!("calc.h");`
  When the user calls `calc::create_point(10, 20)`
  Then the macro-generated wrapper shall return a safe Glyim wrapper type `calc::Point`
  And the wrapper type shall implement the `Drop` trait
  And when the wrapper goes out of scope, `free_point` is automatically invoked

Scenario: Handle a C function returning NULL safely
  Given a C function `Point* create_point(int x, int y)` that returns NULL when allocation fails
  When the user calls `calc::create_point(-1, -1)` 
  Then the wrapper shall return `Err(glyim_ffi::Error::NullPointer)`
  And the program shall not segfault
```

### 5.2 Specification by Example: Async Profiling

**Feature:** `glyim profile` Async Awareness

```gherkin
Scenario: Generate an async-aware flamegraph for a multi-task server
  Given a running Glyim async HTTP server handling 100 concurrent requests
  And the server contains an async function `async fn handle_request()`
  When the developer runs "glyim profile --async-aware -o flame.svg -- target/release/server"
  Then the generated SVG file is valid XML
  And the SVG contains a frame labeled "async fn handle_request"
  And the SVG does NOT contain frames labeled "runtime::poll" or "sys::epoll_wait" taking up >5% of the total width
```

### 5.3 Specification by Example: Supply Chain Security

**Feature:** SLSA Provenance and SBOM Generation

```gherkin
Scenario: Generate a valid SPDX SBOM
  Given a Glyim project with a `glyim.lock` file containing 3 dependencies
  When the developer runs "glyim build --sbom=sbom.spdx.json"
  Then the exit code is 0
  And the file "sbom.spdx.json" is created
  And the JSON validates strictly against the SPDX 2.3 JSON Schema
  And the JSON contains exactly 4 packages (the app + 3 dependencies)
  And each package object contains a "sha256" algorithm and a corresponding digest

Scenario: Block CI on High Severity CVE
  Given a `glyim.lock` file containing a dependency with a known CVE severity "Critical"
  When the CI pipeline runs "glyim audit --fail-on=critical"
  Then the CLI exits with code 1
  And stderr contains the string "CRITICAL"
  And stderr contains the CVE ID of the vulnerable dependency
```

### 5.5 Test Strategy & Edge Cases
* **FFI Edge Cases (Miri):** All `glyim-ffi` generated wrappers must pass testing under Rust's MIRI interpreter (adapted for Glyim) to prove absence of undefined behavior (UB) regarding pointer provenance, alignment, and aliasing.
* **Tracing Edge Cases:** Test that span context propagates correctly through deeply nested `.await` calls (>100 depth) and across `tokio::spawn`/`select!` equivalents in Glyim.
* **SBOM Edge Cases:** Verify that local path dependencies (e.g., `path = "../local_lib"`) are correctly flagged in the SBOM as "Package-Source: Local" and do not attempt to calculate a remote SHA256.
