# Glyim v0.11.0 Specification Suite: The IDE Intelligence Release

This document suite constitutes the fully fledged specification for Glyim v0.11.0. It applies evidence-based requirements engineering, architecture documentation, and behavioral specification practices, structured across five specification levels to ensure strategic alignment, stakeholder satisfaction, technical feasibility, and verifiable delivery.

---

## Document 1: Vision & Strategic Alignment (Level 0)

**Document ID:** GLYIM-V110-VISION-001  
**Version:** 1.0  
**Status:** Approved  
**Author:** Glyim Core Team  

### 1.1 Vision & Elevator Pitch
**Vision:** To provide the fastest, most semantically rich editing experience of any systems language, where the IDE understands code as deeply as the compiler does.  
**Elevator Pitch (Moore Template):** For power users dissatisfied with laggy IDEs and broken macro navigation in complex languages like Rust, Glyim provides a zero-latency, deeply semantic editing experience. Unlike rust-analyzer, which suffers from macro span degradation and CPU-heavy push diagnostics, Glyim pairs a strict LSP 3.17 Pull model with MIR-based macro span mapping and lightweight SSA analysis—eliminating UI freezes and making macro-heavy, highly-generic code instantly comprehensible.

### 1.2 Problem & Business Context
The v0.7.0 LSP skeleton proved the concept, but empirical evidence from Rust developer surveys consistently shows that IDE performance degrades as codebase complexity (macros, generics, async) increases. The current v0.7.0 architecture uses a "Push" diagnostic model, spamming the client with type-checking errors on every keystroke, and its macro span resolution breaks on nested procedural macros, sending users to compiler internals instead of their source code. To attract power users building complex systems, the IDE must not just be "feature-complete"; it must be "instantaneous and infallible."

### 1.3 Target Users (Target Group)
1. **The Framework Author:** Writes deep procedural macros (e.g., `derive` macros generating routing code). Requires reliable "Go to Definition" through multiple layers of macro expansion.
2. **The Productive Polyglot:** Uses Glyim alongside Rust/Go. Expects VS Code intellisense to feel as snappy as Go's `gopls` while offering the type richness of Rust.
3. **The Code Reviewer:** Reads complex pull requests. Relies on semantic highlighting (e.g., mutable vs. immutable, dead code fading) to catch logic errors at a glance.

### 1.4 Desired Outcomes & Success Metrics (OKRs)
* **Objective 1: Achieve zero-perceived input latency.**
  * **KR 1:** 99th percentile (p99) time from keystroke to cursor advance/caret update is <50ms on a 100k LOC workspace.
  * **KR 2:** Opening a 5,000-line file never causes a UI freeze (>100ms unresponsiveness) in VS Code.
* **Objective 2: Perfect macro intelligence.**
  * **KR 1:** "Go to Definition" inside code generated by 3+ levels of nested macros resolves to the correct original source location in >99% of cases.
* **Objective 3: Maximize developer velocity via assists.**
  * **KR 1:** 80% of active daily users trigger at least one LSP Code Action (Assist) per session.

### 1.5 Goals and Non-Goals (Scope / Anti-scope)
**Goals (P0/P1):**
* Implement LSP 3.17 Diagnostic Pull Model.
* Implement Inlay Hints (parameters, return types, chained types).
* Implement Type Hierarchy (Trait supertypes/subtypes).
* Rewrite Macro Span Mapping using MIR-based hygiene chains.
* Implement core Assists (fill match arms, add `#[derive]`, desugar macro).
* Implement Semantic Syntax Highlighting (mutability, dead code).

**Non-Goals (Anti-scope):**
* **No Async DAP View:** Debugging suspended async tasks is deferred to v0.12.0.
* **No WASM-based LSP:** Running the LSP natively in the browser via WASM is a P2 research project.
* **No GUI Refactoring Dialogs:** All assists are single-action quick-fixes; multi-step interactive wizards are out of scope.

## Document 2: Business & Stakeholder Requirements (Level 1)

**Document ID:** GLYIM-V110-BRS-001  

### 2.1 Stakeholders and User Classes
| User Class | Characteristics | Primary Needs | Disfavored Behaviors |
| :--- | :--- | :--- | :--- |
| **Framework Author** | Writes proc-macros using v0.6.0 `quote!`/`splice!`. | Clicking "Peek Definition" on generated code shows the *original* macro invocation, not raw AST dumps. | Being sent to `<::glyim_macros::quote::internal::GeneratedAst>`. |
| **Speed Demanding Dev** | Large monitors, fast typist, low patience for spinners. | Completions appear before the next keystroke finishes; background analysis never steals CPU from the foreground. | IDE blocking the main thread while type-checking an unrelated file in another tab. |
| **Visual Learner (Reviewer)** | Relies on color and shape to parse code structure quickly. | `mut` variables are visually distinct; unused variables are visually suppressed. | Standard textmate syntax highlighting where `let x = 5;` and `let mut x = 5;` look identical except for the keyword. |

### 2.2 Stakeholder Needs & Value Proposition
* **SN-001 (Framework Author):** "I need to trust that refactoring my macro input won't break IDE navigation." -> *Value Prop: MIR-based span hygiene chain.*
* **SN-002 (Speed Demanding Dev):** "I need my laptop fan to stay off while I'm just typing comments." -> *Value Prop: Pull diagnostics + Salsa cancellation.*
* **SN-003 (Visual Learner):** "I need to see dead code without running the compiler." -> *Value Prop: Lightweight LSP-side SSA reachability analysis.*

### 2.3 Ubiquitous Language (Glossary Seed)
* **Pull Diagnostics:** Client-requested error reporting (LSP 3.17), replacing server-pushed errors.
* **Inlay Hints:** Virtual text rendered inline by the editor (e.g., inferred types) that is not part of the source code.
* **Assist:** A localized code transformation triggered by the user (e.g., `Alt+Enter`), also known as a Code Action.
* **Span Hygiene Chain:** A linked list of source locations tracking an identifier through macro expansion, AST lowering, and MIR generation.
* **Semantic Tokens:** LSP protocol for providing syntax highlighting based on compiler semantic analysis, rather than regex/textmate.

## Document 3: Software Requirements Specification (Level 2)

**Document ID:** GLYIM-V110-SRS-001  
**Traceability:** Maps to Level 1 SN-001 through SN-003 and Level 0 OKRs.

### 3.1 Functional Requirements: Performance & Feedback

**FR-LSP-001: Diagnostic Pull Model**
* **Description:** The LSP server shall implement the `textDocument/diagnostic` request from LSP 3.17.
* **Fit Criterion:** The server must not send `textDocument/publishDiagnostics` for a file unless explicitly requested by the client for the active visible document. Background files must be analyzed for indexing but must not emit diagnostic updates.

**FR-LSP-002: Strict Request Cancellation**
* **Description:** The LSP server must honor `$/cancelRequest` notifications instantly.
* **Fit Criterion:** If a completion request is in progress and the user types another character, the LSP must abort the completion computation within 1ms and return an LSP error code `-32800` (ContentModified/Cancelled).

### 3.2 Functional Requirements: Semantic Intelligence

**FR-LSP-003: Inlay Hints**
* **Description:** The LSP shall provide `textDocument/inlayHint` responses.
* **Fit Criterion:** Hints must be generated for: (1) Function call parameters (showing argument names), (2) Inferred variable type bindings (`let x = 5;` shows `: i32`), (3) Implicit return type hints at function closures. Hints must be suppressed if they overlap with existing source text.

**FR-LSP-004: Type Hierarchy**
* **Description:** The LSP shall implement `typeHierarchy/supertypes` and `typeHierarchy/subtypes`.
* **Fit Criterion:** Requesting supertypes on a struct must return all implemented traits. Requesting subtypes on a trait must return all structs implementing that trait across the workspace, resolving through generic bounds (e.g., `impl<T> MyTrait for T where T: Debug`).

**FR-LSP-005: Semantic Syntax Highlighting**
* **Description:** The LSP shall return `textDocument/semanticTokens/full`.
* **Fit Criterion:** The server must classify identifiers with semantic modifiers: `mutable` (for bindings assigned to with `mut`), `declaration` (first occurrence), `static` (for globals/consts), and `deadCode` (for locally declared but unused variables based on reachability analysis).

### 3.3 Functional Requirements: Macro Intelligence

**FR-LSP-006: MIR-Based Macro Span Mapping**
* **Description:** The LSP must accurately map identifiers in generated macro code back to the original source text, regardless of macro nesting depth.
* **Fit Criterion:** Given a procedural macro `foo!` that expands to `bar!(x)`, which expands to `let y = x;`, hovering over `y` must show the hover documentation from the original definition of `x`, and "Go to Definition" must move the cursor to `x` in the user's source file.

**FR-LSP-007: Macro Desugar Assist**
* **Description:** The LSP shall provide an Assist to view macro expansions inline.
* **Fit Criterion:** Triggering "Desugar Macro" on `my_macro!(args)` must replace the text in the editor view with the expanded AST (pretty-printed), appending a comment `// expanded from my_macro!`. This is a visual-only transformation that does not modify the underlying file on disk unless saved.

### 3.4 Functional Requirements: Assists (Code Actions)

**FR-LSP-008: Core Assists**
* **Description:** The LSP must provide the following `textDocument/codeAction` responses:
  1. `fill_match_arms`: Given an incomplete `match` expression, generate arms for all missing variants of the matched enum.
  2. `add_derive`: Suggest adding `#[derive(...)]` to a struct/enum based on available traits in scope.
  3. `extract_function`: Select a block of code and extract it into a new `fn`.
  4. `unwrap_or_else`: Convert `.unwrap()` calls on `Result`/`Option` into `.unwrap_or_else(|e| ...)`.

### 3.5 Non-Functional Requirements (NFRs)

**NFR-PERF-001 (Input Latency):**
* The time from a keystroke event arriving at the LSP to the LSP sending the updated syntax highlighting/line changes must be **<50ms** at p99 on a standard M-series MacBook Pro for a 100k LOC workspace.

**NFR-PERF-002 (Memory Overhead):**
* The LSP server process must not consume more than **2.0 GB** of RSS memory when indexing a workspace containing 500k LOC of Glyim code, including all macro expansions in Salsa caches.

**NFR-ROBUST-001 (Span Accuracy):**
* In a test suite of 1,000 complex macro invocations (up to 5 levels of nesting), "Go to Definition" must resolve to the correct user source file location in **>99.5%** of cases.

## Document 4: Architecture & Design Specification (Level 3)

**Document ID:** GLYIM-V110-ARCH-001  

### 4.1 Context & System Boundary (C4 Level 1)
v0.11.0 is a major architectural rewrite of the `glyim-lsp` server component. It transitions from a synchronous, compiler-wrapping model to a highly asynchronous, incremental Salsa-driven model with a distinct LSP-side analysis layer separate from the main `glyimc` compilation pipeline.

### 4.2 Architecture Decision Records (ADRs)

**ADR-015: LSP 3.17 Pull Diagnostics & Salsa Lazy Queries**
* **Context:** The v0.7.0 LSP pushes diagnostics via `publishDiagnostics` on every `textDocument/didChange`. For a 5,000-line file, type-checking takes ~200ms, blocking the main thread and dropping keystrokes.
* **Decision:** Implement `textDocument/diagnostic`. The LSP will no longer trigger analysis on `didChange`. Instead, it marks the file as "dirty" in the Salsa database. When the client explicitly requests diagnostics, the Salsa query runs. Furthermore, Salsa queries for type-checking will be split into "local" (function body) and "global" (cross-crate trait resolution). Local queries run synchronously and instantly; global queries run asynchronously and yield results when ready.
* **Consequences:**
  * *(+)* Zero keystroke lag for local edits (typing inside a function doesn't trigger cross-crate trait solving).
  * *(+)* Significant battery life improvement on laptops.
  * *(-)* Diagnostic appearance might be delayed by ~100-200ms compared to push model if the user stops typing and waits, though perceived lag is eliminated.

**ADR-016: MIR-Based Span Hygiene Chain**
* **Context:** v0.7.0 maps spans during AST expansion. Nested procedural macros (e.g., a derive macro calling `quote!` which calls another derive) corrupt the span map because intermediate synthetic files overwrite the original span context.
* **Decision:** Implement a persistent `SpanContext` linked list. When a proc-macro expands, the generated AST nodes do not receive the span of the macro definition; they receive a new `SpanContext` whose parent pointer is the invocation span. This chain is preserved through HIR (High-Level IR) lowering into MIR (Mid-Level IR). When the LSP resolves a hover or definition request on a MIR location, it walks the chain *backwards* to find the root user-facing span.
* **Consequences:**
  * *(+)* Perfect accuracy for arbitrarily deep macro expansions.
  * *(+)* Enables "Go to Definition" across macro boundaries.
  * *(-)* Increases memory usage per macro expansion (one pointer per AST node).

**ADR-017: LSP-Side Mini-SSA for Semantic Features**
* **Context:** To provide "Dead Code" semantic highlighting and "Inline Values" for constants, the LSP needs data-flow analysis. Running the full compiler's MIR is too slow for real-time typing.
* **Decision:** Build a lightweight, LSP-specific SSA (Static Single Assignment) form directly on the HIR. This mini-SSA only tracks local variable assignments and basic reachability. It does not handle drop elaboration or complex borrow checking. It is strictly used to drive `semanticTokens` (dead code) and `inlineValue` (const folding) requests.
* **Consequences:*
  * *(+)* Dead code highlighting updates instantly without blocking the compiler.
  * *(-)* Might show false positives for dead code that the full borrow checker would later reject (acceptable trade-off for IDE speed).

## Document 5: Behavioral Specification & Test Verification (Level 4/5)

**Document ID:** GLYIM-V110-BEHAV-001  
**Verification Method:** Automated Test (LSP Integration tests via `tower-lsp` mock client, Performance benchmarks via `criterion`).

### 5.1 Specification by Example (SbE): Diagnostic Pull Model

**Feature:** Zero-lag typing with Pull Diagnostics

```gherkin
Scenario: Typing inside a function does not trigger cross-crate type checking
  Given a 5,000-line Glyim file with a complex trait constraint relying on an external crate
  And the LSP server has fully indexed the workspace
  When the user types a new local variable assignment inside a function body
  Then the LSP server shall NOT execute the cross-crate trait resolution query
  And the client shall receive an updated syntax highlight response within 50ms
  And the client shall NOT receive a `publishDiagnostics` notification

Scenario: Diagnostics appear only when requested or on focus
  Given a Glyim file containing a type error in a background tab
  When the user switches to the tab containing the type error
  Then the client sends a `textDocument/diagnostic` request
  And the server responds with the type error diagnostic
```

### 5.2 Specification by Example (SbE): Macro Span Mapping

**Feature:** MIR-Based Span Hygiene Chain

```gherkin
Scenario: Go to definition through nested procedural macros
  Given a procedural macro `route_builder!` that generates code calling another macro `handler_map!`
  And `handler_map!` expands to `fn handle(req: Request) { internal_logic(req) }`
  Where `internal_logic` is defined in the user's original source file
  When the user invokes "Go to Definition" on `internal_logic` inside the expanded macro view
  Then the cursor shall jump to the definition of `internal_logic` in the user's source file
  And the cursor shall NOT jump to the `route_builder!` macro definition file
  And the hover tooltip shall display the documentation of `internal_logic`
```

### 5.3 Specification by Example (SbE): Assists & Semantic Highlighting

**Feature:** Fill Match Arms Assist

```gherkin
Scenario: User triggers assist on incomplete match expression
  Given a source file containing:
    """
    enum Color { Red, Green, Blue }
    let c = Color::Red;
    match c {
      Color::Red => println!("red"),
    }
    """
  When the user places their cursor on the `match` keyword and triggers "Code Action"
  And the user selects the "Fill missing match arms" assist
  Then the source text is updated to:
    """
    match c {
      Color::Red => println!("red"),
      Color::Green => todo!(),
      Color::Blue => todo!(),
    }
    """
```

**Feature: Semantic Syntax Highlighting (Dead Code)**

```gherkin
Scenario: Unused variable is visually suppressed
  Given a source file containing "let unused_var = 42;"
  And the variable `unused_var` is never read or mutated
  When the LSP computes semantic tokens for the file
  Then the token representing `unused_var` shall include the semantic modifier "deprecated" or "deadCode"
  And the editor shall render the token as visually faded or struck-through
```

### 5.5 Test Strategy & Edge Cases
* **Macro Span Stress Testing:** We will maintain a corpus of real-world Glyim proc-macros (e.g., an HTTP router generator, a serialization derive macro). The LSP test suite will run "Go to Def" on every identifier in the expanded output of these macros and assert a 0% failure rate on mapping back to user source.
* **Cancellation Storms:** A dedicated benchmark will simulate a "rapid typist" (10 keystrokes per second) on a 10,000-line file. The test asserts that the LSP processes 100% of cancellation requests and never falls behind by more than 1 queued request, ensuring the UI thread never blocks.
* **Mini-SSA Fallbacks:** Test that if the Mini-SSA engine panics or fails on malformed code (e.g., unresolvable control flow), it silently degrades to standard TextMate highlighting without crashing the LSP server.
