# Glyim v0.12.0 Specification Suite: The Scale & Ecosystem Release

This document suite constitutes the fully fledged specification for Glyim v0.12.0. It applies evidence-based requirements engineering, architecture documentation, and behavioral specification practices, structured across five specification levels to ensure strategic alignment, stakeholder satisfaction, technical feasibility, and verifiable delivery.

---

## Document 1: Vision & Strategic Alignment (Level 0)

**Document ID:** GLYIM-V120-VISION-001  
**Version:** 1.0  
**Status:** Approved  
**Author:** Glyim Core Team  

### 1.1 Vision & Elevator Pitch
**Vision:** To transform Glyim from a high-performance language for individual crates into a foundational platform for massive, multi-team monorepos, while opening the compiler's intelligence to the ecosystem.  
**Elevator Pitch (Moore Template):** For engineering managers and platform teams frustrated by slow, linear CI times and fragmented tooling in Rust/Go monorepos, Glyim provides native, graph-aware distributed caching and a strictly versioned Compiler SDK. Unlike Cargo, which relies on brittle third-party caching bolt-ons, Glyim caches at the MIR level natively, and unlike internal Rust compiler crates, Glyim exposes its HIR/Salsa layers as a stable API—eliminating build bottlenecks and enabling custom enterprise linters without forking the compiler.

### 1.2 Problem & Business Context
Glyim v0.11.0 delivered a world-class individual developer experience (IDE). However, empirical evidence from enterprise Rust adoptions (e.g., Android, Microsoft) shows that languages hit a "monorepo wall" at ~500k-1M lines of code. The current `glyim build` system treats every crate as an isolated compilation unit. CI times scale linearly, and build caches (if used) operate at the wrong abstraction layer (LLVM IR only), missing expensive trait resolution and MIR generation steps. Furthermore, companies cannot enforce domain-specific architectural rules (e.g., "no raw FFI outside this module") without writing fragile regex parsers, because the compiler's internal AST is unstable.

### 1.3 Target Users (Target Group)
1. **The Platform Engineer:** Responsible for CI/CD pipelines. Needs builds to scale sub-linearly and caches to be hermetic and secure.
2. **The Tooling Author:** Internal developer productivity (DevProd) engineer tasked with writing custom linters, mutators, or refactoring tools.
3. **The Staff Architect:** Manages a massive codebase. Needs to visualize dependency graphs to find bottlenecks and eliminate duplicate transitive dependencies.

### 1.4 Desired Outcomes & Success Metrics (OKRs)
* **Objective 1: Achieve sub-linear CI scaling.**
  * **KR 1:** Clean CI build times for a synthetic 1M LOC monorepo (10k crates) must not exceed 5 minutes on a 64-core machine utilizing remote caching.
  * **KR 2:** A 1-line change in a leaf dependency must trigger a rebuild of only the dependent crates, taking <10 seconds total via cache hits.
* **Objective 2: Establish a thriving tooling ecosystem.**
  * **KR 1:** Release `glyim-syntax`, `glyim-hir`, and `glyim-salsa` SDK crates as stable v1.0.0 with a strict backward compatibility guarantee.
  * **KR 2:** At least 5 third-party linter/refactoring tools are built using the SDK within 6 months of release.
* **Objective 3: Provide build graph transparency.**
  * **KR 1:** `glyim build --timings` generates a dependency DAG visualization that loads in <2 seconds and accurately identifies the top 5 slowest trait resolution steps.

### 1.5 Goals and Non-Goals (Scope / Anti-scope)
**Goals (P0/P1):**
* Implement `glyim-workspace` (native monorepo support with shared lockfiles).
* Implement native MIR-level distributed caching (`glyim-cache`).
* Extract and stabilize the Compiler SDK (`glyim-syntax`, `glyim-hir`).
* Implement Build Profiler (`glyim build --timings` HTML output).
* Implement Dependency Tree Visualizer (`glyim tree`).
* Update LSP to support Virtual Workspaces.
* Release official `rules_glyim` for Bazel.

**Non-Goals (Anti-scope):**
* **No Build System Replacement:** `glyim build` remains the default; we are not building a competitor to Buck2 or Pants. We are making `glyim build` scalable and exposing a Bazel integration.
* **No Distributed Compilation:** We are caching *artifacts*, not distributing the execution of the compiler across multiple machines (like `distcc`).
* **No Package Registry Rewrite:** The registry UI/API remains as of v0.9.0; this release focuses purely on the local build graph and SDK.

## Document 2: Business & Stakeholder Requirements (Level 1)

**Document ID:** GLYIM-V120-BRS-001  

### 2.1 Stakeholders and User Classes
| User Class | Characteristics | Primary Needs | Disfavored Behaviors |
| :--- | :--- | :--- | :--- |
| **Platform Engineer** | Manages CI runners, S3 buckets, and build clusters. | Cache hits must be deterministic; cache uploads must be authenticated and compressed; no cache poisoning. | Relying on `sccache` which often invalidates caches incorrectly due to macro state. |
| **Tooling Author** | Writes Rust/TypeScript -> Glyim transpilers, or custom lint rules. | Stable APIs that don't break on minor compiler updates; easy way to hook into the LSP (v0.11.0). | Reading `glyimc` internal crates directly, which change without warning. |
| **Staff Architect** | Owns system architecture across 50+ teams. | Visualizing "dependency hell"; finding which team owns a slow-to-compile crate. | Counting lines of code to estimate build impact. |

### 2.2 Stakeholder Needs & Value Proposition
* **SN-001 (Platform Engineer):** "I need CI to fetch compiled dependencies instead of recompiling them, without writing complex wrapper scripts." -> *Value Prop: Native `glyim-cache` protocol with built-in S3/GCS backends.*
* **SN-002 (Tooling Author):** "I need to write a linter that warns when an async function lacks a timeout, without parsing macros manually." -> *Value Prop: Stable `glyim-hir` SDK providing type-resolved ASTs.*
* **SN-003 (Staff Architect):** "I need to know if adding this utility crate will blow up our binary size or compile times." -> *Value Prop: `glyim tree --sizes` mapping dependency graph to binary bloat.*

### 2.3 Ubiquitous Language (Glossary Seed)
* **Workspace:** A collection of multiple Glyim packages (crates) managed under a single unified build context and lockfile.
* **MIR-Level Cache:** A build cache that stores artifacts after the Mid-Level IR (MIR) optimization phase, skipping trait resolution and early lowering on cache hits.
* **Virtual Workspace (LSP):** An IDE mode where only the actively focused project is fully analyzed in RAM; out-of-focus projects are represented by pre-computed declaration stubs (`.d.gy` files).
* **Compiler SDK:** The public, stable API surface of the compiler's syntax trees (Rowan) and analysis framework (Salsa).

## Document 3: Software Requirements Specification (Level 2)

**Document ID:** GLYIM-V120-SRS-001  
**Traceability:** Maps to Level 1 SN-001 through SN-003 and Level 0 OKRs.

### 3.1 Functional Requirements: Workspace & Graph Intelligence

**FR-WS-001: Unified Workspace Definition**
* **Description:** The build system shall support a `glyim.workspace.toml` file defining multiple members.
* **Fit Criterion:** A workspace must generate a single, unified `glyim.lock` file encompassing all members. Running `glyim build` at the workspace root must resolve dependencies across all members concurrently.

**FR-WS-002: Dependency Tree Visualization**
* **Description:** A CLI command to inspect the dependency graph.
* **Fit Criterion:** `glyim tree` must output a textual tree. `glyim tree --viz=graph.html` must output an interactive HTML DAG. `glyim tree --duplicates` must highlight transitive dependencies that are pulled in by multiple paths with differing versions (diamond dependency conflicts).

**FR-WS-003: Build Profiler**
* **Description:** A CLI flag to profile compilation steps.
* **Fit Criterion:** `glyim build --timings` must output an HTML report containing a flamegraph-style visualization of the build process, explicitly breaking down time spent in: Parsing, HIR Lowering, Trait Resolution, MIR Optimization, and LLVM Codegen per crate.

### 3.2 Functional Requirements: Native Distributed Caching

**FR-CACHE-001: MIR-Level Content Addressing**
* **Description:** The compiler shall hash the HIR/MIR state of a crate to determine cache keys.
* **Fit Criterion:** The cache key must be a SHA-256 hash of the crate's HIR (including all resolved generic types and trait bounds) plus the compiler version hash. It must *not* include absolute filesystem paths (to ensure cache sharing across different CI machines).

**FR-CACHE-002: Remote Cache Backend**
* **Description:** The `glyim-cache` daemon shall support remote object storage.
* **Fit Criterion:** The daemon must natively support AWS S3 and Google Cloud Storage protocols via standard environment variables (`AWS_*`, `GOOGLE_APPLICATION_CREDENTIALS`). It must upload compiled artifacts (`.gyo` binaries, metadata) as compressed zstd archives.

**FR-CACHE-003: Cache Authentication & Signing**
* **Description:** Remote cache entries must be tamper-proof.
* **Fit Criterion:** Before loading a remote artifact, `glyim-cache` must verify an attached Sigstore signature (using the compiler's private key, verified against a public key configured in `glyim.toml`). Unverifiable artifacts must be discarded and recompiled locally.

### 3.3 Functional Requirements: Compiler-as-a-SDK

**FR-SDK-001: Stable Syntax Crate (`glyim-syntax`)**
* **Description:** Expose the Rowan-based syntax tree as an independent crate.
* **Fit Criterion:** `glyim-syntax` v1.x must guarantee that any code parsing a valid v0.12.0 Glyim source file will continue to parse it identically under v1.y. The API must provide lossless string-to-AST-to-string conversion.

**FR-SDK-002: Stable HIR Crate (`glyim-hir`)**
* **Description:** Expose the type-resolved High-Level IR.
* **Fit Criterion:** `glyim-hir` must allow users to iterate over a file's functions, resolve types, and inspect trait implementations *without* invoking the full compiler pipeline. It must handle macro-expanded code transparently using the v0.11.0 MIR-based span hygiene chains.

**FR-SDK-003: LSP Tooling Hook API**
* **Description:** Provide a standard mechanism for SDK tools to integrate with `glyim-lsp`.
* **Fit Criterion:** A developer must be able to define a custom diagnostic in a standalone binary, and configure `glyim-lsp` via `glyim.toml` to spawn this binary as a subprocess, forwarding LSP diagnostics natively.

### 3.4 Functional Requirements: LSP & Bazel Integration

**FR-LSP-001: Virtual Workspaces**
* **Description:** The LSP must handle massive workspaces without loading everything into RAM.
* **Fit Criterion:** When opening a workspace with 10,000 crates, the LSP must generate "Declaration Files" (`.d.gy`, analogous to TypeScript `.d.ts`) for out-of-focus crates during the initial workspace build. The LSP RAM usage must not exceed 4GB regardless of workspace size.

**FR-BAZEL-001: Official Bazel Rules (`rules_glyim`)**
* **Description:** Provide a Starlark ruleset for compiling Glyim.
* **Fit Criterion:** `rules_glyim` must support `glyim_library`, `glyim_binary`, and `glyim_test` rules. It must integrate with Bazel's remote caching natively, mapping Bazel action keys to Glyim's MIR-level cache keys.

### 3.5 Non-Functional Requirements (NFRs)

**NFR-PERF-001 (Cache Latency):**
* Fetching a cache hit from a remote S3 bucket over a 1Gbps network must add **<500ms** of overhead to the total build time of a single crate.

**NFR-STAB-001 (SDK Semantic Versioning):**
* The `glyim-syntax`, `glyim-hir`, and `glyim-salsa` crates must strictly adhere to Semantic Versioning 2.0.0. A minor version bump (1.x -> 1.y) must not break compilation of downstream SDK consumers.

**NFR-SEC-001 (Cache Isolation):**
* The `glyim-cache` daemon must run in a hermetic sandbox. It must not leak secrets from the CI environment into the cached artifacts, and must strip all environment variables except an explicit allowlist from the cache key.

## Document 4: Architecture & Design Specification (Level 3)

**Document ID:** GLYIM-V120-ARCH-001  

### 4.1 Context & System Boundary (C4 Level 1)
v0.12.0 introduces a parallel infrastructure layer that sits alongside the compiler: the `glyim-cache` daemon, the Compiler SDK extraction boundary, and the Workspace graph manager.

### 4.2 Architecture Decision Records (ADRs)

**ADR-018: MIR-Level Content-Addressable Caching over LLVM-Only Caching**
* **Context:** Tools like `sccache` cache the final LLVM IR/CodeGen step. However, in Glyim, trait resolution (HIR) and MIR optimization often take as long as LLVM codegen. If a struct changes, dependent crates re-run trait resolution even if the machine code is ultimately identical.
* **Decision:** Implement caching at the boundary between MIR optimization and LLVM lowering. We serialize the post-optimized MIR into a binary blob, hash the pre-optimization MIR + HIR context, and use that as the cache key. On a hit, we skip HIR, trait resolution, and MIR optimization entirely, passing the serialized MIR directly to LLVM.
* **Consequences:**
  * *(+)* Massive CI time reductions for large dependency trees (often 60-70% of compile time skipped).
  * *(+)* Cache is immune to LLVM version upgrades (MIR is stable, LLVM IR is not).
  * *(-)* Requires a stable, deterministic MIR serialization format.

**ADR-019: Extracting the Compiler SDK via API Sealing & Facade Crates**
* **Context:** The compiler internals change constantly. We cannot stabilize internal crates without freezing compiler development. However, tooling authors need stable APIs.
* **Decision:** Create "Facade Crates" (`glyim-syntax`, `glyim-hir`). These crates will re-export specific, vetted types from the internal compiler crates. Internally, the compiler will use the raw crates; externally, users use the facades. We will use a combination of API sealing (hiding internal modules) and strict `#[doc(hidden)]` boundaries.
* **Consequences:**
  * *(+)* Compiler team can refactor internals freely as long as the Facade API remains unchanged.
  * *(+)* Clear contract for third-party tooling authors.
  * *(-)* Maintenance burden of maintaining the facade mapping layer.

**ADR-020: Declaration Generation (`.d.gy`) for Virtual Workspaces**
* **Context:** Loading 10,000 crates into the v0.11.0 Salsa database causes OOM. We need a way for the LSP to understand types from out-of-focus crates without fully type-checking them.
* **Decision:** During the workspace build (`glyim build`), the compiler will emit `.d.gy` files for every crate. These files contain *only* public type signatures, trait definitions, and constants—no function bodies. The LSP will parse `.d.gy` files into a lightweight "Stub HIR" that provides type information and goto-def capabilities but skips MIR/SSA analysis.
* **Consequences:*
  * *(+)* LSP RAM usage becomes flat regardless of workspace size.
  * *(-)* Go-to-definition for private items in out-of-focus crates will fall back to text-based search (grep).

## Document 5: Behavioral Specification & Test Verification (Level 4/5)

**Document ID:** GLYIM-V120-BEHAV-001  
**Verification Method:** Automated Test (Integration for cache, API compliance for SDK, Synthetic benchmarks for scaling).

### 5.1 Specification by Example (SbE): Native Distributed Caching

**Feature:** MIR-Level Cache Hit & Security

```gherkin
Scenario: Successful remote cache hit skips trait resolution
  Given a remote S3 cache containing a compiled artifact for crate "utils" (MIR hash: ABC123)
  And the local "utils" source code has not changed
  When the developer runs "glyim build" for a project depending on "utils"
  Then the compiler shall calculate the MIR hash for "utils" as ABC123
  And the compiler shall skip HIR lowering, trait resolution, and MIR optimization for "utils"
  And the build time for "utils" shall be <1 second
  And the compiler shall verify the Sigstore signature of the downloaded artifact against the trusted public key

Scenario: Cache poisoning is rejected
  Given a remote S3 cache where the artifact for "utils" (hash ABC123) has been maliciously altered
  And the altered artifact lacks a valid cryptographic signature
  When the developer runs "glyim build"
  Then the compiler shall discard the downloaded artifact
  And the compiler shall fall back to local compilation
  And a warning shall be printed to stderr: "Cache artifact signature verification failed"
```

### 5.2 Specification by Example (SbE): Compiler SDK Stability

**Feature:** Semantic Versioning Guarantee

```gherkin
Scenario: Downstream linter compiles against minor SDK version bump
  Given a third-party linter "glyim-no-unsafe" that depends on "glyim-hir" version "1.4.0"
  And the Glyim project releases "glyim-hir" version "1.5.0"
  When the developer runs "glyim build" on the "glyim-no-unsafe" project without modifying its Cargo.toml/Glyim.toml
  Then the project shall compile successfully without errors
  And the linter shall be able to parse and inspect function signatures accurately
```

### 5.3 Specification by Example (SbE): LSP Virtual Workspaces

**Feature:** Flat memory usage in massive workspaces

```gherkin
Scenario: LSP handles 10,000 crate workspace without OOM
  Given a synthetic workspace containing 10,000 interdependent Glyim crates
  And a pre-compiled cache of `.d.gy` declaration files for all 10,000 crates
  When the developer opens the workspace in VS Code and the LSP server starts
  Then the LSP server shall parse the 10,000 `.d.gy` files into Stub HIR
  And the RSS memory usage of the LSP process shall not exceed 4.0 GB
  And "Go to Definition" of a type defined in an out-of-focus crate shall successfully navigate to the `.d.gy` file
```

### 5.4 Specification by Example (SbE): Build Profiling & Tree Visualization

**Feature:** Accurate Build Timings DAG

```gherkin
Scenario: Generate HTML timings report
  Given a workspace with 3 crates: A, B, and C
  And compiling A takes 2s, B takes 3s, and C takes 1s (after B is done)
  When the developer runs "glyim build --timings"
  Then an "index.html" file is generated in the target directory
  And opening the HTML file shall display an interactive flamegraph
  And the flamegraph shall explicitly show that B and C ran in parallel (total wall time ~4s, not 6s)
```

### 5.6 Test Strategy & Edge Cases
* **Cache Key Hermeticity Fuzzing:** A fuzzer will mutate environment variables, `PATH` contents, and filesystem layouts to prove that none of these factors alter the MIR hash, ensuring perfect cache sharing across heterogeneous CI runners.
* **SDK API Freeze Testing:** In the CI pipeline, we will maintain a "canary" tooling project that is forcibly held at an older `glyim-sdk` version. Every nightly compiler build must successfully compile this canary project to catch accidental API breakages before they ship.
* **Diamond Dependency DAG Testing:** The `glyim tree` tests will include a known-bad state of a massive diamond dependency graph to ensure the visualization algorithm handles extreme fan-out without stack overflowing or producing unreadable SVGs.
