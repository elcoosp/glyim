# Glyim v0.13.0 Specification Suite: The Frontier Release

This document suite constitutes the fully fledged specification for Glyim v0.13.0. It applies evidence-based requirements engineering, architecture documentation, and behavioral specification practices, structured across five specification levels to ensure strategic alignment, stakeholder satisfaction, technical feasibility, and verifiable delivery.

---

## Document 1: Vision & Strategic Alignment (Level 0)

**Document ID:** GLYIM-V130-VISION-001  
**Version:** 1.0  
**Status:** Approved  
**Author:** Glyim Core Team  

### 1.1 Vision & Elevator Pitch
**Vision:** To break through the abstraction limits of modern systems programming by replacing rigid compiler constructs with user-defined algebraic effects, and replacing heavy containerized microservices with lightweight, type-safe WASM components.  
**Elevator Pitch (Moore Template):** For systems architects dissatisfied with Rust's async "colored functions" and the operational overhead of Dockerized microservices, Glyim is a post-async, WASM-native language. Unlike Rust or Go, Glyim allows developers to define I/O, state, and concurrency as library-level effects (eliminating syntax bloat), and natively load WASM components as type-safe local structs with automatic memory management—bridging the gap between monolithic speed and microservice isolation.

### 1.2 Problem & Business Context
Glyim v0.12.0 secured enterprise adoption at the build-system level. However, at the language design level, we are approaching the theoretical limits of the `async/await` paradigm (introduced in v0.8.0). As codebases grow, developers are forced to wrap async runtimes, logging tracers, and cancellation tokens into deeply nested generic types ("wrapper hell"). Concurrently, teams are over-using microservices not for scale, but simply to achieve independent deployability of modules written in different languages. We must shift the paradigm before Glyim inherits the technical debt of our predecessors.

### 1.3 Target Users (Target Group)
1. **The Modularity Architect:** Wants to deploy a Python ML scoring module or a Rust parsing library inside a core Glyim billing service without spinning up a gRPC sidecar, Docker container, or Kubernetes pod.
2. **The Async Pioneer:** Wants to write a data-processing function that can run synchronously in a deterministic test, but asynchronously in production, without changing the function's signature or adding `async` boundaries.
3. **The AI Tooling Engineer:** Wants to build internal Copilots that understand Glyim's complex macro expansions and trait bounds perfectly, requiring structured compiler data rather than raw text parsing.

### 1.4 Desired Outcomes & Success Metrics (OKRs)
* **Objective 1: Prove the Algebraic Effects paradigm in production.**
  * **KR 1:** 100% of v0.12.0 `async fn` code compiles unchanged in v0.13.0 by implicitly desugaring to the standard `Async` effect handler.
  * **KR 2:** The top 5 most-starred Glyim open-source libraries adopt explicit `effect` blocks for state/context management within 6 months of release.
* **Objective 2: Achieve zero-overhead polyglot modularity.**
  * **KR 1:** A Wasm component function call from Glyim must add **<50 nanoseconds** of overhead compared to a native Glyim function call.
* **Objective 3: Maximize AI Copilot accuracy via compiler streams.**
  * **KR 1:** Third-party AI agents consuming the `$/glyim/astStream` LSP protocol demonstrate a 40% reduction in hallucinated type errors compared to raw-text parsing.

### 1.5 Goals and Non-Goals (Scope / Anti-scope)
**Goals (P0/P1):**
* Implement Algebraic Effects (`effect` keyword, `perform`, handlers).
* Implement backward-compatible async-to-effect desugaring.
* Implement first-class WASM Component Model host (`glyim-component`).
* Implement Wasm Resource to Glyim `Drop` trait mapping.
* Implement LSP AI Context Stream (`$/glyim/astStream`).
* Implement Compile-Time Function Execution (CTFE) for constants.
* Implement Effect-based Structured Concurrency stdlib handler.

**Non-Goals (Anti-scope):**
* **No Native Wasm Compilation Target:** v0.8.0 compiles *to* Wasm. v0.13.0 *hosts* Wasm. We are not changing the output target.
* **No Unikernel OS:** Compiling Glyim directly to a hypervisor payload (P2) is purely experimental and not a supported target.
* **No Visual Effect Debugger:** Stepping through effect resumptions in the DAP is deferred.

## Document 2: Business & Stakeholder Requirements (Level 1)

**Document ID:** GLYIM-V130-BRS-001  

### 2.1 Stakeholders and User Classes
| User Class | Characteristics | Primary Needs | Disfavored Behaviors |
| :--- | :--- | :--- | :--- |
| **Modularity Architect** | Manages large polyglot codebases. | Type-safe boundaries; memory safety across language borders; hot-reloading capabilities. | Maintaining gRPC `.proto` files and Docker compose files for internal modules. |
| **Async Pioneer** | Writes highly concurrent, stateful network services. | Composable control flow; deterministic testing without mocking async runtimes. | Refactoring a sync function to async by adding `async` to 10 downstream callers. |
| **AI Tooling Engineer** | Builds internal DevProd tools. | Lossless, structured representation of code semantics. | Using Regex to parse Glyim macros to feed into an LLM. |

### 2.2 Stakeholder Needs & Value Proposition
* **SN-001 (Modularity Architect):** "I need to load a Python script as a plugin without writing FFI bindings or managing a separate Python process." -> *Value Prop: Wasm Component Host with automatic WIT-to-Glyim trait generation.*
* **SN-002 (Async Pioneer):** "I need my core business logic to be completely agnostic to whether it runs on a single thread, a thread pool, or across the network." -> *Value Prop: Algebraic Effects, where the execution strategy is provided by a handler, not baked into the function.*
* **SN-003 (AI Tooling Engineer):** "I need the LSP to tell my AI agent exactly what generic bounds are in scope at the cursor." -> *Value Prop: AI Context Stream (JSON HIR snapshot).*

### 2.3 Ubiquitous Language (Glossary Seed)
* **Algebraic Effect:** A mechanism to separate the *description* of an operation (e.g., `fetch_data`) from its *implementation* (e.g., sync read, async HTTP, mock).
* **Effect Handler:** A runtime construct that intercepts performed effects and provides their implementation.
* **WIT (Wasm Interface Types):** An IDL used to define the API surface of a WASM Component.
* **Wasm Resource:** A Wasm Component concept representing an object with encapsulated state and a destructor, analogous to a Glyim struct implementing `Drop`.
* **CTFE (Compile-Time Function Execution):** The ability of the compiler to run arbitrary pure Glyim functions at compile time to calculate constants.

## Document 3: Software Requirements Specification (Level 2)

**Document ID:** GLYIM-V130-SRS-001  
**Traceability:** Maps to Level 1 SN-001 through SN-003 and Level 0 OKRs.

### 3.1 Functional Requirements: Algebraic Effects

**FR-EFF-001: Effect Definition Syntax**
* **Description:** The language shall provide an `effect` keyword to define an effect interface.
* **Fit Criterion:** An effect block must define one or more function signatures. It must be compiable as a standard trait type.

**FR-EFF-002: Effect Performance (`perform`)**
* **Description:** The language shall provide a `perform` keyword to invoke an effect operation inside a function.
* **Fit Criterion:** A function using `perform` must not be required to declare `async`, return a `Future`, or specify any execution context. It must compile as a standard, synchronous-looking function signature.

**FR-EFF-003: Effect Handlers**
* **Description:** The standard library shall provide a mechanism to wrap effectful computations with a handler.
* **Fit Criterion:** A handler must intercept `perform` calls. If the handler resolves the effect, execution continues. If the handler aborts, the error must propagate to the `run` caller. Handlers must support deep stacking (a handler for `Logging` wrapping a handler for `Async`).

**FR-EFF-004: Async-to-Effect Desugaring**
* **Description:** Existing `async fn` and `.await` syntax must seamlessly map to the new Effect system.
* **Fit Criterion:** The compiler must internally translate `async fn` into a function performing a standardized `Effect<Async>` operation. The default async runtime must implicitly act as the root handler. No changes to existing v0.12.0 async code are permitted.

### 3.2 Functional Requirements: WASM Component Host

**FR-WASM-001: WIT Binding Generation (`glyim-component bind`)**
* **Description:** A CLI tool to generate Glyim source code from WIT definitions.
* **Fit Criterion:** Given a `world.wit` file, the tool must generate a Glyim module containing a native `struct` that implements traits matching the WIT interfaces, and a `Loader` struct to instantiate the Wasm component.

**FR-WASM-002: Dynamic Component Loading**
* **Description:** The runtime shall support loading compiled WASM components from disk or memory.
* **Fit Criterion:* `Loader::from_file("plugin.wasm")` must parse the component, validate its WIT canonical ABI, and instantiate it in the current process's memory space without spawning a subprocess.

**FR-WASM-003: Wasm Resource to Glyim `Drop` Mapping**
* **Description:** Wasm Resources must be mapped safely to Glyim structs.
* **Fit Criterion:** When a Wasm function returns a `resource`, the generated Glyim code must wrap it in a native Glyim struct. When this struct goes out of scope and the Glyim `Drop` trait runs, the generated binding must invoke the Wasm resource's `[resource-drop]` destructor.

### 3.3 Functional Requirements: AI & CTFE

**FR-AI-001: AI Context Stream (`$/glyim/astStream`)**
* **Description:** The LSP shall expose a streaming endpoint for AI agents.
* **Fit Criterion:* Upon subscription, the LSP must output a JSON object for the active cursor containing: `expected_type`, `visible_traits_in_scope`, `local_variables` (with types), `macro_expanded_hir` (if cursor is inside a macro), and `imported_modules`.

**FR-CTFE-001: Compile-Time Function Execution**
* **Description:** The compiler shall evaluate `const` declarations by executing functions.
* **Fit Criterion:* A function marked `const fn` may contain loops, conditional logic, and standard library calls (e.g., `format!`, basic math), provided they do not perform I/O or heap allocations exceeding a compiler-defined limit (e.g., 1MB). The result must be evaluated at compile time and embedded in the binary as a primitive or byte slice.

### 3.4 Functional Requirements: Concurrency

**FR-CONC-001: Effect-based Structured Concurrency**
* **Description:** The stdlib shall provide a structured concurrency handler using the Effect system.
* **Fit Criterion:** A `Scope::run(|s| { s.spawn(task1); s.spawn(task2); })` must guarantee that if `task1` panics or is cancelled, `task2` is deterministically cancelled before `Scope::run` returns.

### 3.5 Non-Functional Requirements (NFRs)

**NFR-PERF-001 (Wasm Call Overhead):**
* A call from Glyim to a Wasm component function, passing primitives, must take **<50 nanoseconds** longer than an equivalent native Glyim function call.

**NFR-PERF-002 (Effect Indirection):**
* When an Effect handler is statically known at compile time and monomorphized, the overhead of the `perform` mechanism must be **<2%** compared to a direct function call.

**NFR-COMPAT-001 (Zero Breaking Changes for Async):**
* 100% of valid v0.12.0 async codebases must compile and produce identical binary behavior in v0.13.0 without source modifications.

## Document 4: Architecture & Design Specification (Level 3)

**Document ID:** GLYIM-V130-ARCH-001  

### 4.1 Context & System Boundary (C4 Level 1)
v0.13.0 fundamentally alters two core compiler subsystems: the MIR lowering phase (to support Effects and CTFE) and the linking/runtime phase (to support the WASM Component Host).

### 4.2 Architecture Decision Records (ADRs)

**ADR-021: Koka-style Continuation-Passing for Effects**
* **Context:** We need an implementation strategy for `perform` that allows deep stacking of effects (e.g., a custom logging effect wrapping an async effect).
* **Decision:** Adopt a multi-shot continuation-passing style (CPS) at the MIR level. When the compiler sees `perform`, it transforms the current function into a suspended state machine, capturing the continuation. The Effect handler receives this continuation and decides whether to resume it (with a value) or abort it (with an error).
* **Consequences:**
  * *(+)* Composable and stack-safe. Deeply nested effects do not cause stack overflows.
  * *(+)* Makes CTFE trivial, as the compiler's MIR interpreter can natively capture and resume these continuations.
  * *(-)* Breaks standard stack unwinding for panics inside effectful code; requires a custom runtime allocator for continuation frames.

**ADR-022: Wasm Component State via Opaque Typed Handles**
* **Context:** Wasm Components encapsulate state inside "Resources." Glyim cannot just dereference Wasm memory because WASM is linear memory, not garbage-collected objects.
* **Decision:** The generated `glyim-component bind` code represents Wasm resources as opaque `struct ResourceHandle<T>(usize)`. The `usize` is an index into a host-side slab allocator that maps to the Wasm instance's resource table. The `Drop` implementation calls a Wasm export `[resource-drop]`.
* **Consequences:*
  * *(+)* Zero-cost abstraction; the handle is just an integer.
  * *(+)* Completely safe from the Glyim side; the type system prevents forging handles.
  * *(-)* Requires strict lifecycle management in the Wasm binding layer to prevent use-after-free across different Wasm instances.

**ADR-023: CTFE via In-Process MIR Interpreter**
* **Context:** We need to execute `const fn` logic (loops, parsing) at compile time. Writing a separate interpreter is massive duplication of effort.
* **Decision:** Execute the MIR (Mid-Level IR) directly inside the compiler process. We will instantiate a specialized MIR interpreter that uses a bounded, deterministic heap. Standard library functions marked `#[const_eval_safe]` will be linked into this interpreter environment.
* **Consequences:*
  * *(+)* Guarantees that compile-time evaluation exactly matches runtime semantics (same MIR).
  * *(-)* A bug in the MIR interpreter could crash the compiler; requires heavy sandboxing (memory limits, instruction limits).

## Document 5: Behavioral Specification & Test Verification (Level 4/5)

**Document ID:** GLYIM-V130-BEHAV-001  
**Verification Method:** Automated Test (Unit for MIR interpreter, Integration for Wasm runtime, Snapshot for AI stream).

### 5.1 Specification by Example (SbE): Algebraic Effects (Sync/Async Transparency)

**Feature:** Effect definition and handler resolution

```gherkin
Scenario: Function using an effect runs synchronously with a sync handler
  Given an effect definition:
    """
    effect Env { fn get_var(name: str) -> Option<String>; }
    """
  And a function:
    """
    fn get_config() -> String {
      perform Env::get_var("PORT").unwrap_or("8080".to_string())
    }
    """
  When `get_config` is executed inside a synchronous handler `MockEnvHandler`
  Then the function executes completely on the current thread
  And no async state machine is generated in the binary

Scenario: The exact same function runs asynchronously with an async handler
  Given the same function `get_config` from above
  When `get_config` is executed inside an asynchronous handler `RealEnvHandler` (which performs OS I/O)
  Then the function suspends at the `perform` keyword
  And the thread is yielded back to the async runtime
  And the final result is identical to the sync execution
```

### 5.2 Specification by Example (SbE): Wasm Resource Safety

**Feature:** Automatic Wasm Resource destruction via Glyim Drop

```gherkin
Scenario: Wasm resource is destroyed when Glyim handle goes out of scope
  Given a WASM component "db.wasm" defining a WIT resource "Connection"
  And "Connection" has a Wasm-side destructor that prints "Closed"
  And Glyim code generated via `glyim-component bind` maps this to `struct Connection`
  When the Glyim code executes:
    """
    let conn = db::Connection::open("localhost");
    // conn goes out of scope here
    """
  Then the Glyim `Drop` trait for `Connection` is triggered
  And the WASM runtime invokes the `[resource-drop]` export for "Connection"
  And the Wasm-side print statement executes successfully
  And no memory leak occurs in the Wasm instance
```

### 5.3 Specification by Example (SbE): AI Context Stream

**Feature:** LSP provides structured HIR context

```gherkin
Scenario: AI agent requests context stream at a specific cursor
  Given a Glyim file containing:

    fn process<T: Serialize>(data: T) -> Result<Vec<u8>> { ... }
    """
  And the cursor is positioned on the `data` parameter
  When the AI client sends a `$/glyim/astStream` request
  Then the LSP responds with a JSON object containing:

    {
      "cursor": "1:23",
      "expected_type": "T",
      "visible_traits_in_scope": ["Serialize"],
      "local_variables": [],

  And the JSON strictly validates against the published `glyim-ai-stream-schema.json`
```

### 5.4 Specification by Example (SbE): Compile-Time Function Execution (CTFE)

**Feature:** Complex logic evaluated at compile time

```gherkin
Scenario: Parse a string into a struct at compile time
  Given a function:
    """
    const fn parse_config() -> Config {
        let raw = "{\"port\": 8080}"; // Simulated const string
        serde_json::from_str(raw).unwrap()
    }
    """
  And a static assertion: `static CONFIG: Config = parse_config();`
  When the compiler evaluates `parse_config()`
  Then the compiler executes the JSON parser inside the MIR CTFE engine
  And the resulting binary does not contain the JSON parser code for `parse_config`
  And the binary contains only the pre-calculated bytes of the `Config` struct
```

### 5.6 Test Strategy & Edge Cases
* **Effect CPS Stack Depth:** Fuzz test the continuation allocator by generating deeply nested effect handlers (>10,000 levels) to ensure no stack overflows or memory leaks occur in the CPS transformation.
* **Wasm Host Isolation:** Deliberately pass malformed Wasm components (e.g., wrong canonical ABI, missing exports) to the `Loader`. Verify that the Glyim runtime traps the error cleanly and returns a `Err(LoadError::InvalidAbi)` without crashing the host process.
* **CTFE Sandboxing:** Attempt to perform network I/O or allocate >1GB of memory inside a `const fn`. The MIR interpreter must immediately halt compilation with a clear error: "Const eval exceeded memory limit".
