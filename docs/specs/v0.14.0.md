# Glyim v0.14.0 Specification Suite: The Living Documentation Release

This document suite constitutes the fully fledged specification for Glyim v0.14.0. It applies evidence-based requirements engineering, architecture documentation, and behavioral specification practices, structured across five specification levels to ensure strategic alignment, stakeholder satisfaction, technical feasibility, and verifiable delivery.

---

## Document 1: Vision & Strategic Alignment (Level 0)

**Document ID:** GLYIM-V140-VISION-001  
**Version:** 1.0  
**Status:** Approved  
**Author:** Glyim Core Team  

### 1.1 Vision & Elevator Pitch
**Vision:** To eradicate documentation rot by making doc comments a first-class, compilable, and testable element of the language syntax.  
**Elevator Pitch (Moore Template):** For library authors frustrated by Rust's disconnected macro-documentation and slow, brittle external doctest harnesses, Glyim unifies Markdown and code into a single lossless AST. Unlike rustdoc, Glyim natively executes doctests via the compiler's MIR engine and provides IDE error squiggles directly inside doc comments—even for code generated by procedural macros—guaranteeing that if it compiles, it is documented, and if it is documented, it compiles.

### 1.2 Problem & Business Context
Glyim v0.9.0 delivered excellent narrative docs, but the *mechanics* of writing library documentation remain stuck in the past. Empirical evidence from the Rust ecosystem (the primary competitor) shows that as macro usage increases, documentation drifts because generated code cannot easily host doc comments. Furthermore, standard doctest implementations (like Rust's or Python's) operate as separate test harnesses; they are slow, do not leverage modern compiler caching (v0.12.0), and cannot report type errors inline within the IDE. We must leverage our unique architectural advantages—Rowan (v0.2.0), MIR Span Chains (v0.11.0), and CTFE (v0.13.0)—to solve this fundamentally.

### 1.3 Target Users (Target Group)
1. **The Library Maintainer:** Writes complex public APIs and procedural macros. Needs to guarantee that every example in their documentation is syntactically valid and type-checks.
2. **The Open Source Contributor:** Reviewing pull requests. Needs to see red squiggles directly in the doc comments if the author forgot to import a module in an example.
3. **The Consumer/Reader:** Reading docs on `docs.glyim.org`. Needs to run examples instantly in the browser (v0.9.0 Playground) without copy-pasting.

### 1.4 Desired Outcomes & Success Metrics (OKRs)
* **Objective 1: Achieve zero documentation drift.**
  * **KR 1:** CI builds will fail if a doc-test code block fails to compile or panics, enforced natively without third-party tools.
  * **KR 2:** 100% of the Glyim standard library public items will have at least one executing doc-test.
* **Objective 2: Perfect macro documentation.**
  * **KR 1:** Doc comments injected via `quote!` (v0.6.0) into generated code will successfully render in `glyim doc` HTML output, with source links tracing back to the macro invocation.
* **Objective 3: Seamless reading-to-execution flow.**
  * **KR 1:** Every code block on `docs.glyim.org` generated by `glyim doc` will feature a functional "Run on Playground" button.

### 1.5 Goals and Non-Goals (Scope / Anti-scope)
**Goals (P0/P1):**
* Implement Rowan-Markdown parser (Unified AST).
* Implement native doctest execution via the MIR CTFE engine (v0.13.0).
* Implement macro-aware doc generation using MIR span hygiene chains (v0.11.0).
* Implement automatic Playground URL injection in HTML docs.
* Implement compile-time checked intra-doc links (`[MyType]`).
* Implement doc attributes (`#[doc(cfg(...))]`, `#[doc(alias = "...")]`).

**Non-Goals (Anti-scope):**
* **No Custom Markdown Extensions (Beyond Standard):** We support CommonMark + GFM. We will not invent custom markdown tags.
* **No Interactive Tutorial Frameworks:** We generate static HTML with JS hooks for the playground. We are not building a scalable interactive tutorial engine like Swift DocC.
* **No Wasm-based Doc Rendering:** Docs are pre-rendered at build time, not rendered dynamically in the browser via Wasm.

## Document 2: Business & Stakeholder Requirements (Level 1)

**Document ID:** GLYIM-V140-BRS-001  

### 2.1 Stakeholders and User Classes
| User Class | Characteristics | Primary Needs | Disfavored Behaviors |
| :--- | :--- | :--- | :--- |
| **Library Maintainer** | Writes `///` comments above public APIs and inside `quote!` blocks. | Inline IDE errors for broken doc examples; automatic docs for derived structs. | Running `glyim test --doc` in a separate terminal and matching line numbers manually. |
| **OSS Contributor** | Reviews PRs adding new functions. | Seeing a red squiggle directly in the `///` comment if an example is missing an import. | Opening a web browser to check if a code block renders correctly. |
| **Consumer** | Reads `docs.glyim.org` to learn an API. | 1-click execution of examples; visible source links for macro-generated methods. | Copy-pasting a code block into the Playground only to find it's missing a `use` statement. |

### 2.2 Stakeholder Needs & Value Proposition
* **SN-001 (Library Maintainer):** "I need to know if my doc examples break when I refactor a private module." -> *Value Prop: Native doctests run on every `glyim build`, failing the build instantly.*
* **SN-002 (OSS Contributor):** "I need to document structs generated by my derive macro without writing a separate markdown file." -> *Value Prop: Macro-aware docs via v0.11.0 span chains.*
* **SN-003 (Consumer):** "I want to tweak a doc example and run it immediately." -> *Value Prop: "Run" button injecting base64-encoded, compiler-validated code into the Playground.*

### 2.3 Ubiquitous Language (Glossary Seed)
* **Rowan-Markdown:** An extension of the v0.2.0 Rowan parser that generates a lossless Green Tree for Markdown, with Glyim code blocks parsed as child AST nodes.
* **Native DocTest:** A code block inside a `/// ` comment that is type-checked and executed by the compiler's MIR interpreter (CTFE), not an external test harness.
* **Intra-Doc Link:** A Markdown link syntax `[TypeName]` that the compiler resolves to an item ID, failing the build if the item is deleted or private.
* **Macro-Aware Doc:** Documentation attached to code generated by procedural macros, accurately mapped back to the macro invocation site.

## Document 3: Software Requirements Specification (Level 2)

**Document ID:** GLYIM-V140-SRS-001  
**Traceability:** Maps to Level 1 SN-001 through SN-003 and Level 0 OKRs.

### 3.1 Functional Requirements: Unified Parsing & IDE

**FR-MD-001: Rowan-Markdown Parser**
* **Description:** The compiler shall parse Markdown doc comments into a unified AST.
* **Fit Criterion:** When the lexer encounters `///`, it invokes the Rowan-Markdown parser. Text inside ` ```glyim ` blocks must be parsed into standard Glyim AST nodes and attached as children of the Markdown AST.

**FR-MD-002: Inline Diagnostics in Doc Comments**
* **Description:** The LSP (v0.11.0) must report errors inside doc comments.
* **Fit Criterion:** If a ` ```glyim ` block contains a type error, the LSP must emit a diagnostic with a span that exactly matches the character indices *inside the `/// ` comment*. No external tools or file switches required.

### 3.2 Functional Requirements: Native DocTest Execution

**FR-DOC-001: Compiler-Native DocTest Execution**
* **Description:** `glyim test` shall execute doc-tests natively.
* **Fit Criterion:** The compiler must extract ` ```glyim ` blocks, feed them into the v0.13.0 MIR CTFE (Compile-Time Function Execution) engine, and run them. It must not shell out to a separate `glyim-doctest` binary. Failure of a doc-test must result in a non-zero exit code and a standardized error report.

**FR-DOC-002: DocTest Isolation**
* **Description:** Each doc-test block must execute in a clean environment.
* **Fit Criterion:** State mutated in DocTest A must not leak into DocTest B, even if they are in the same file. A default set of stdlib prelude imports must be implicitly provided unless `# ignore` or `# no_run` is specified.

**FR-DOC-003: DocTest Visibility Inheritance**
* **Description:** Doc-tests must respect the visibility of the item they document.
* **Fit Criterion:** A doc-test on a `pub(crate)` function may not access private sibling modules, but may access anything the function itself can access.

### 3.3 Functional Requirements: Macro-Aware Documentation

**FR-MACRO-001: Doc Generation for Macro-Emitted Items**
* **Description:** `glyim doc` must render documentation for items generated by macros.
* **Fit Criterion:** If a procedural macro uses `quote!` to emit `/// My generated docs` above a struct, `glyim doc` must render that text in the HTML output for the generated struct.

**FR-MACRO-002: Source Links for Macro-Generated Docs**
* **Description:** HTML output must show where macro-generated items came from.
* **Fit Criterion:** On the documentation page for a macro-generated struct, the "Source" link must point to the *invocation* of the macro (e.g., `#[derive(MyMacro)]` on line 10), not the internals of the macro definition file.

**FR-MACRO-003: DocTest Errors in Macro-Generated Docs**
* **Description:** Doc-tests inside macro-generated docs must map errors correctly.
* **Fit Criterion:** If a doc-test inside a generated struct fails, the compiler error message must state: "Doc-test failed in macro-generated struct `FooBuilder`, originating from macro invocation at src/main.gy:10:3".

### 3.4 Functional Requirements: HTML Generation & Links

**FR-HTML-001: Playground Injection**
* **Description:** `glyim doc` HTML must include "Run" buttons.
* **Fit Criterion:** Every ` ```glyim ` block in the generated HTML must be wrapped in a UI element with a button. Clicking the button must open `https://play.glyim.org/?code=<base64_valid_code>`.

**FR-LINK-001: Compile-Time Intra-Doc Links**
* **Description:** Markdown links `[TypeName]` must resolve to items.
* **Fit Criterion:** The compiler must parse `[TypeName]` in doc comments. If `TypeName` does not exist, is private, or is ambiguous, the compiler must emit a hard error (not a warning) during `glyim doc --check`.

### 3.5 Non-Functional Requirements (NFRs)

**NFR-PERF-001 (DocTest Overhead):**
* Running the full native doc-test suite must add **<5%** overhead to the total `glyim test` execution time, leveraging the MIR interpreter and caching.

**NFR-PERF-002 (Markdown Parse Time):**
* Parsing a 1,000-line Markdown comment with 5 embedded code blocks must take **<2ms**.

## Document 4: Architecture & Design Specification (Level 3)

**Document ID:** GLYIM-V140-ARCH-001  

### 4.1 Context & System Boundary (C4 Level 1)
v0.14.0 deeply integrates the documentation pipeline into the compiler frontend and the MIR interpreter. The `glyim doc` tool becomes primarily an HTML renderer, while all validation, testing, and type-checking occur inside `glyimc` and `glyim-lsp`.

### 4.2 Architecture Decision Records (ADRs)

**ADR-024: Unified Rowan AST for Markdown and Code**
* **Context:** We need IDE diagnostics inside doc comments. Standard tooling parses markdown to HTML, and code separately. This creates two disconnected ASTs.
* **Decision:** Extend Rowan to natively support Markdown syntax. When Rowan encounters a fenced code block tagged ` ```glyim `, it invokes the Glyim lexer/parser recursively and inserts the resulting Glyim Green Tree as a *child node* of the Markdown AST.
* **Consequences:*
  * *(+)* The Salsa database (v0.11.0) naturally indexes doc-block code. Diagnostics map perfectly to character spans in the comment.
  * *(-)* Increases the size of the Rowan crate.

**ADR-025: DocTests via MIR Interpreter (CTFE Integration)**
* **Context:** Traditional doc-tests compile code to a binary and run it. This is slow and requires linking.
* **Decision:** Reuse the v0.13.0 CTFE (Compile-Time Function Execution) MIR interpreter. When `glyim test` encounters a doc block, it lowers it to MIR and executes it in the sandboxed interpreter, exactly like a `const fn`.
* **Consequences:*
  * *(+)* Doc-tests run in milliseconds, not seconds. No binary linking required.
  * *(+)* Perfect isolation (bounded heap, no filesystem access).
  * *(-)* Doc-tests cannot test platform-specific native FFI (v0.10.0) or complex async runtimes; those must remain traditional integration tests.

**ADR-026: Macro Doc Span Transitivity**
* **Context:** A macro emits a doc comment. The v0.11.0 MIR span chain tracks code origins, but doc comments are usually stripped during parsing.
* **Decision:** Modify the parser to preserve `/// ` comment spans in the HIR. When `quote!` generates a doc comment, the span context of that comment points to the `quote!` invocation site (or the input tokens). `glyim doc` uses this span to render the "Source" link.
* **Consequences:*
  * *(+)* Zero-cost macro documentation. Macro authors just write `///` in their `quote!` calls.
  * *(-)* Requires careful span hygiene to prevent malicious macros from injecting docs that look like they belong to the stdlib.

## Document 5: Behavioral Specification & Test Verification (Level 4/5)

**Document ID:** GLYIM-V140-BEHAV-001  
**Verification Method:** Automated Test (LSP Integration tests, Snapshot tests for HTML, Compiler unit tests for CTFE).

### 5.1 Specification by Example (SbE): Unified AST & Inline Diagnostics

**Feature:** IDE squiggles inside Markdown doc comments

```gherkin
Scenario: Type error inside a doc comment code block
  Given a Glyim source file containing:
    """
    /// Adds two numbers.
    /// ```glyim
    /// let sum = add(1, "two"); // Type error: expected u32, got str
    /// ```
    fn add(a: u32, b: u32) -> u32 { a + b }
    """
  When the LSP calculates diagnostics for the file
  Then a diagnostic of severity "Error" must be emitted
  And the diagnostic range must map exactly to the characters `add(1, "two")` inside the `/// ` comment
  And the IDE must render a red squiggle under the code block in the comment
```

### 5.2 Specification by Example (SbE): Native DocTest Execution

**Feature:** DocTests run natively via MIR Interpreter

```gherkin
Scenario: Successful native doctest execution
  Given a doc comment containing:
    """
    /// ```
    /// let x = 5 + 5;
    /// assert_eq!(x, 10);
    /// ```
    """
  When the user runs `glyim test`
  Then the compiler extracts the code block
  And the code is lowered to MIR and executed in the CTFE sandbox
  And the test is marked as "Passed"
  And no external binary is spawned

Scenario: DocTest isolation prevents state leakage
  Given a file with two functions, each with a doc-test:
    """
    /// ```glyim
    /// static mut STATE: i32 = 0;
    /// STATE += 1;
    /// ```
    fn a() {}
    /// ```glyim
    /// assert_eq!(STATE, 0); // Should be 0, not 1, due to isolation
    /// ```
    fn b() {}
    """
  When the user runs `glyim test`
  Then the doc-test for `a()` passes
  And the doc-test for `b()` passes
```

### 5.3 Specification by Example (SbE): Macro-Aware Docs

**Feature:** Docs and errors for macro-generated code map to invocation site

```gherkin
Scenario: DocTest error in macro-generated code reports invocation location
  Given a procedural macro `derive(Builder)` that generates:
    """
    /// Generated builder.
    /// ```glyim
    /// let b = FooBuilder::new().invalid_method(); // Error
    /// ```
    pub struct FooBuilder { ... }
    """
  And the macro is invoked at `src/main.gy:10:3`
  When the user runs `glyim test`
  Then the doc-test must fail
  And the compiler stderr must contain the text "originating from macro invocation at src/main.gy:10:3"
```

### 5.4 Specification by Example (SbE): Playground Injection & Link Resolution

**Feature:** HTML generation includes valid Playground links and checks intra-doc links

```gherkin
Scenario: Playground button uses valid base64 code
  Given the HTML output of `glyim doc` for a function with a doc-test
  When inspecting the HTML DOM
  Then there must be an element with an `onclick` or `href` attribute
  And the attribute must contain `https://play.glyim.org/?code=`
  And the base64 payload must decode to exactly the code block text (including implicit prelude)

Scenario: Broken intra-doc link fails the build
  Given a doc comment containing:
    """
    /// See also [NonExistentType].
    """
  When the user runs `glyim doc --check`
  Then the compiler must exit with a non-zero code
  And stderr must contain "unresolved link to `NonExistentType`"
```

### 5.6 Test Strategy & Edge Cases
* **Rowan-Markdown Fuzzing:** A fuzzer will generate random byte sequences inside `/// ` blocks. The Rowan-Markdown parser must *never* panic (panic-free guarantee). It must gracefully recover and emit a parsing error diagnostic.
* **CTFE Limits in DocTests:** Deliberately write a doc-test that attempts an infinite loop or allocates 2GB of memory. The MIR interpreter must trap the execution and report: "Doc-test failed: CTFE resource limit exceeded", without crashing the `glyim test` process.
* **HTML Snapshot Testing:** The entire HTML output of `glyim doc` for the `core` library will be snapshot-tested. Any change in CSS classes, Playground URL formatting, or macro-source-link rendering will require updating the snapshot, ensuring visual/doc regressions are caught.
