# Glyiml v0.15.0 Specification Suite: The Oracle Release

This document suite constitutes the fully fledged specification for Glyiml v0.15.0. It applies evidence-based requirements engineering, architecture documentation, and behavioral specification practices, structured across five specification levels to ensure strategic alignment, stakeholder satisfaction, technical feasibility, and verifiable delivery.

---

## Document 1: Vision & Strategic Alignment (Level 0)

**Document ID:** GLYIML-V150-VISION-001  
**Version:** 1.0  
**Status:** Approved  
**Author:** Glyiml Core Team  

### 1.1 Vision & Elevator Pitch
**Vision:** To eradicate false positives and blind spots in static analysis by making linting a native, zero-cost, type-aware compiler pass, and automating the resolution of architectural and security violations.  
**Elevator Pitch (Moore Template):** For security engineers and platform teams dissatisfied with the high false-positive rates of AST-based linters (ESLint/Clippy) and the slow setup of relational security tools (CodeQL), Glyiml provides a native MIR-based linting engine. Unlike external tools, Glyiml lints query the compiler's own Salsa database to guarantee zero false positives on semantic checks, natively track security taint across Algebraic Effect boundaries, and autonomously apply fixes via the IDE Assist API—turning CI from a noisy报警 system into an autonomous repair mechanism.

### 1.2 Problem & Business Context
Glyiml v0.12.0 open-sourced the compiler SDK (`glyim-hir`, `glyim-mir`), and v0.13.0 introduced Algebraic Effects. However, the ecosystem currently relies on external, regex or AST-based linters that cannot understand macro-expanded code, generic monomorphizations, or cross-crate Effect boundaries. Furthermore, security teams are forced to integrate slow, external taint-analysis tools that require writing Datalog queries. The result is noisy CI pipelines that developers ignore. We must leverage our unique internal compiler architecture to make linting omniscient and automated.

### 1.3 Target Users (Target Group)
1. **The Security Engineer:** Needs to ensure untrusted data from the network is sanitized before hitting the database. Requires taint analysis that understands Glyiml's Effect system.
2. **The Platform Engineer:** Manages CI/CD for 500+ developers. Wants trivial style violations auto-fixed without human intervention.
3. **The Library Maintainer:** Writes complex procedural macros. Needs lints to accurately analyze the *generated* code, not the macro invocation syntax.

### 1.4 Desired Outcomes & Success Metrics (OKRs)
* **Objective 1: Achieve zero false positives on semantic lints.**
  * **KR 1:** The built-in `unused_assignment` lint (using v0.11.0 SSA) will exhibit a 0% false-positive rate on a 100k LOC benchmark suite.
* **Objective 2: Automate away trivial CI failures.**
  * **KR 1:** In the official Glyiml repository, >90% of failed CI lint checks must be automatically resolved by `glyim lint --fix` without human edits.
* **Objective 3: Deliver native security taint tracking.**
  * **KR 1:** The `taint_effect_flow` lint will successfully detect 100% of known vulnerable data-flow patterns in the OWASP benchmark suite adapted for Effect-based I/O.

### 1.5 Goals and Non-Goals (Scope / Anti-scope)
**Goals (P0/P1):**
* Implement Native MIR Lint API (`glyim-lint-sdk`).
* Implement Effect-Aware Taint Analysis (Sources and Sinks).
* Implement Macro-Transparent Linting (automatic analysis of expanded code).
* Implement Autonomous Fix Engine (`glyim lint --fix`).
* Implement Lint Trust Profiles (Style, Correctness, Security, Architecture).
* Implement Dataflow-Based Unused Assignments.

**Non-Goals (Anti-scope):**
* **No Graph Database Query Language:** We are not building a Datalog/query language (like CodeQL). Lints are written in native Glyiml using the SDK API.
* **No External Linter Plugin System (v0.15.0):** Third-party linters must currently be compiled as standalone binaries that consume the SDK, not dynamically loaded WASM plugins.

## Document 2: Business & Stakeholder Requirements (Level 1)

**Document ID:** GLYIML-V150-BRS-001  

### 2.1 Stakeholders and User Classes
| User Class | Characteristics | Primary Needs | Disfavored Behaviors |
| :--- | :--- | :--- | :--- |
| **Security Engineer** | Writes secure APIs; defines trust boundaries. | Accurate tracking of untrusted data; zero false alarms on safe code paths. | Writing complex CodeQL Datalog scripts that take 20 minutes to run per PR. |
| **Platform Engineer** | Optimizes developer velocity and CI pipeline health. | CI automatically fixing formatting/style issues; clear severity grouping. | CI failing on 500 "missing doc comments" that devs just locally suppress. |
| **Library Maintainer** | Authors `derive` macros and procedural codegen. | Lints catching logic errors inside the *generated* code, pointing to the macro invocation. | Linters warning about the macro definition file instead of where it is used. |

### 2.2 Stakeholder Needs & Value Proposition
* **SN-001 (Security Engineer):** "I need to enforce that any data fetched via the `Network` Effect must pass through a `Sanitize` Effect before reaching the `Database` Effect." -> *Value Prop: Native MIR Taint Engine tracking Effect boundaries.*
* **SN-002 (Platform Engineer):** "I want CI to automatically fix 50 trailing whitespace errors and apply the correct `#[derive]` traits." -> *Value Prop: Autonomous Fix Engine bridging v0.11.0 LSP Assists.*
* **SN-003 (Library Maintainer):** "I want a lint to check that a generated `Clone` impl doesn't leak file handles, and if it does, point to the struct using my macro." -> *Value Prop: Macro-Transparent MIR analysis utilizing v0.11.0 Span Chains.*

### 2.3 Ubiquitous Language (Glossary Seed)
* **Semantic Lint:** A lint rule that queries the compiler's type-resolved MIR (Mid-Level IR) or SSA (Static Single Assignment) graph, rather than just the syntax tree.
* **Effect Taint Tracking:** A security analysis that marks data returned by specific Algebraic Effects (Sources) as "tainted" and prevents it from flowing into parameters of other Effects (Sinks) without passing through a "sanitizer" Effect.
* **Macro-Transparent Linting:** The ability of a lint to automatically analyze the fully expanded code generated by procedural macros, rather than just the macro invocation syntax.
* **Lint Profile:** A named grouping of lints with a predefined severity baseline (e.g., `Style`, `Security`).
* **Autonomous Fix:** A compiler operation that applies an IDE Assist (Code Action) automatically without interactive user confirmation.

## Document 3: Software Requirements Specification (Level 2)

**Document ID:** GLYIML-V150-SRS-001  
**Traceability:** Maps to Level 1 SN-001 through SN-003 and Level 0 OKRs.

### 3.1 Functional Requirements: Semantic Lint Engine

**FR-LINT-001: Native MIR Lint SDK (`glyim-lint-sdk`)**
* **Description:** A first-party crate providing the framework for writing lints.
* **Fit Criterion:** The SDK must expose a `LintPass` trait. Implementations must be able to query the Salsa database (v0.11.0) to iterate over functions, structs, and macro-expanded MIR bodies. Lints must be incrementally computed via Salsa.

**FR-LINT-002: Macro-Transparent Execution**
* **Description:** Lints must automatically analyze macro-generated code.
* **Fit Criterion:** When a lint queries the MIR of a function generated by a procedural macro, it must evaluate the *expanded* MIR. Any diagnostic emitted must map its span back to the macro invocation site (using v0.11.0 hygiene chains) if the expanded span is synthetic.

**FR-LINT-003: Dataflow-Based Unused Assignment**
* **Description:** A built-in lint utilizing SSA for 100% accuracy.
* **Fit Criterion:** The `unused_assignment` lint must use the v0.11.0 Mini-SSA graph. If a variable is assigned but its SSA definition is never used by any other SSA use-site, it must emit a diagnostic. It must not false-positive on variables used only in macro-generated drops.

### 3.2 Functional Requirements: Effect-Aware Taint Analysis

**FR-TAINT-001: Taint Source/Sink Attributes**
* **Description:** The SDK must provide attributes to mark Algebraic Effects.
* **Fit Criterion:** Developers must be able to mark an Effect operation as `#[taint_source]` and another as `#[taint_sink(require_sanitized)]`.

**FR-TAINT-002: MIR Taint Propagation**
* **Description:** The compiler must track tainted data across MIR basic blocks.
* **Fit Criterion:** If a variable is assigned the return value of a `#[taint_source]` Effect, all downstream assignments to that variable are marked tainted. If a tainted variable is passed as an argument to a `#[taint_sink]` Effect, a critical security diagnostic must be emitted.

**FR-TAINT-003: Sanitizer Interception**
* **Description:** Taint must be cleared by explicitly defined functions or Effects.
* **Fit Criterion:** If a tainted variable is passed to a function annotated with `#[taint_sanitizer]`, the taint flag is removed for subsequent assignments. Passing a sanitized variable to a `#[taint_sink]` must not emit a diagnostic.

### 3.3 Functional Requirements: Autonomous Fix Engine

**FR-FIX-001: CLI Auto-Fix (`glyim lint --fix`)**
* **Description:** The CLI must automatically apply applicable fixes.
* **Fit Criterion:** `glyim lint --fix` must iterate over all emitted diagnostics. For diagnostics that have an associated v0.11.0 Assist (Code Action), the CLI must apply the text edit to the file system.

**FR-FIX-002: Fix Verification and Commit**
* **Description:** Auto-fixes must be validated before finalizing.
* **Fit Criterion:** After applying text edits, `glyim lint --fix` must run `glyim test`. If tests pass, it must automatically generate a git commit with the message `chore: apply automated lint fixes`. If tests fail, it must roll back the edits.

### 3.4 Functional Requirements: Profiles and Suppression

**FR-PROF-001: Trust Profiles**
* **Description:** The SDK must support grouping lints into profiles.
* **Fit Criterion:** `glyim.toml` must support a `[lints.profile]` table (e.g., `style`, `security`). Enabling a profile enables all lints within that group with predefined severities.

**FR-SUPP-001: Scoped Suppression**
* **Description:** Developers must be able to suppress lints granularly.
* **Fit Criterion:** The attribute `#[allow(lint_id)]` must suppress a lint for a specific function. `#[expect(lint_id)]` must suppress the lint but emit a warning if the lint *would not* have triggered (catching useless suppressions).

### 3.5 Non-Functional Requirements (NFRs)

**NFR-PERF-001 (Semantic Lint Overhead):**
* Running the full suite of native semantic lints on a 100k LOC crate must add **<3%** overhead to a cold `glyim build`, leveraging Salsa caching.

**NFR-ACCURACY-001 (Zero False Positives on SSA):**
* The built-in SSA lints (e.g., unused assignment, dead code) must exhibit a **0.0% false positive rate** on the official Glyiml test suite.

## Document 4: Architecture & Design Specification (Level 3)

**Document ID:** GLYIML-V150-ARCH-001  

### 4.1 Context & System Boundary (C4 Level 1)
v0.15.0 introduces a major new subsystem: the Analysis Engine. It sits directly atop the Salsa database (v0.11.0) and the MIR/SSA graphs, completely bypassing traditional AST-based text scanning.

### 4.2 Architecture Decision Records (ADRs)

**ADR-027: Salsa-Native Lint Queries vs. AST Walkers**
* **Context:** External linters (like Clippy) walk the AST or use raw token streams. This causes O(N) traversal per lint per file, leading to combinatorial explosion and no access to type information.
* **Decision:** Implement lints as Salsa queries. A lint function requests a specific Salsa key (e.g., `all_functions_with_unused_mir_assignments`). Salsa computes this incrementally, re-running only when the HIR/MIR of a specific function changes.
* **Consequences:**
  * *(+)* Massive performance gains on large codebases; lints are computed lazily.
  * *(+)* Lints have perfect type information because they run post-HIR lowering.
  * *(-)* Lint authors must learn the Salsa query paradigm.

**ADR-028: Effect-Driven Taint State Machines**
* **Context:** Traditional taint analysis tracks variable pointers through memory. This is fragile in systems languages with moves, borrows, and destructors.
* **Decision:** Implement taint tracking as an extension of the MIR SSA graph. We do not track memory locations; we track SSA definition IDs. When a `#[taint_source]` Effect returns, its SSA result ID is marked `Tainted`. When a `#[taint_sink]` Effect consumes an argument, the checker looks at the SSA ID of the argument. If it resolves to a `Tainted` ID, it fails.
* **Consequences:**
  * *(+)* Completely immune to borrow-checker complexities and pointer aliasing.
  * *(+)* Naturally integrates with Algebraic Effects (v0.13.0), as Effects are just function calls in the MIR.
  * *(-)* Cannot track taint through raw pointers or unsafe transmutes (by design, accepted limitation).

**ADR-029: Assist-to-Fix CI Pipeline**
* **Context:** The v0.11.0 LSP provides Code Actions (Assists) as JSON payloads. We need to apply these in a headless CI environment.
* **Decision:** Extract the core logic of the LSP Assist engine into a headless library `glyim-assist-core`. Both the LSP server and the `glyim lint --fix` CLI consume this library.
* **Consequences:**
  * *(+)* Guaranteed consistency: The fix applied by `Alt+Enter` in the IDE is byte-for-byte identical to the fix applied in CI.
  * *(-)* Requires refactoring the v0.11.0 LSP codebase to decouple from the VS Code JSON-RPC layer.

## Document 5: Behavioral Specification & Test Verification (Level 4/5)

### 5.1 Specification by Example (SbE): Effect-Aware Taint Analysis

**Feature:** Tracking untrusted data from Network Effect to Database Effect

```gherkin
Scenario: Tainted data reaches sink without sanitization (Fail)
  Given an Effect `Network` with a method `read()` marked `#[taint_source]`
  And an Effect `Database` with a method `query(raw: Vec<u8>)` marked `#[taint_sink(require_sanitized)]`
  And Glyiml code:
    """
    let data = perform Network::read();
    perform Database::query(data);
    """
  When the taint analysis lint runs
  Then a critical diagnostic must be emitted on the line `perform Database::query(data)`
  And the diagnostic message must contain "Tainted data from 'Network::read' reaching 'Database::query' without sanitization"

Scenario: Tainted data is sanitized before reaching sink (Pass)
  Given the same Effects as above
  And a function `sanitize(data: Vec<u8>) -> Vec<u8>` marked `#[taint_sanitizer]`
  And Glyiml code:
    """
    let data = perform Network::read();
    let clean_data = sanitize(data);
    perform Database::query(clean_data);
    """
  When the taint analysis lint runs
  Then NO diagnostic must be emitted
```

### 5.2 Specification by Example (SbE): Macro-Transparent Linting

**Feature:** Linting code generated by a procedural macro

```gherkin
Scenario: Lint catches unsafe block inside macro-generated code
  Given a procedural macro `derive(UnsafeBuilder)` that generates:
    """
    impl Foo {
      pub fn build(&self) -> Foo {
        unsafe { /* do something dangerous */ }
      }
    }
    """
  And a lint rule `unsafe_block_in_safe_fn`
  And Glyiml code applying the macro: `#[derive(UnsafeBuilder)] struct Foo;`
  When the semantic lint runs
  Then a diagnostic must be emitted
  And the diagnostic span must point to the macro invocation: `#[derive(UnsafeBuilder)]`
  And the diagnostic span must NOT point to the macro definition file
```

### 5.3 Specification by Example (SbE): Autonomous Fix Engine

**Feature:** CI applies fixes and creates a commit

```gherkin
Scenario: Auto-fix applies style fixes, tests pass, commits
  Given a repository with 5 files containing trailing whitespace
  And a lint rule `trailing_whitespace` with an associated Assist
  When the CI pipeline runs `glyim lint --fix`
  Then the CLI must modify the 5 files to remove the whitespace
  And the CLI must run `glyim test` implicitly
  And the CLI must exit with code 0
  And a git commit must exist with the message "chore: apply automated lint fixes"

  And the CLI must run `glyim test` and observe a failure
  And the CLI must exit with code 1
  And the git working tree must be restored to its original state (no commit made)
```

### 5.6 Test Strategy & Edge Cases
* **Taint Engine Fuzzing:** A fuzzer will generate random MIR graphs with random taint source/sink annotations and SSA permutations to ensure the taint state machine never enters an invalid state (e.g., infinite loops in cyclic SSA graphs).
* **Auto-Fix Idempotency:** A test will apply `glyim lint --fix` on a repository, then run it again. The second run must result in 0 changes and exit 0, proving fixes are idempotent.
* **Macro Span Chain Exhaustiveness:** A corpus of real-world Glyiml proc-macros (up to 10 levels of nested `quote!` expansions) will be linted. Every diagnostic must trace back to a human-readable source location. If a diagnostic points to a compiler internal span, the test fails.
