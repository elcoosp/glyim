# Glyim v0.16.0 Specification Suite: The Metaprogramming Mastery Release

This document suite constitutes the fully fledged specification for Glyim v0.16.0. It applies evidence-based requirements engineering, architecture documentation, and behavioral specification practices, structured across five specification levels to ensure strategic alignment, stakeholder satisfaction, technical feasibility, and verifiable delivery.

---

## Document 1: Vision & Strategic Alignment (Level 0)

**Document ID:** GLYIM-V160-VISION-001  
**Version:** 1.0  
**Status:** Approved  
**Author:** Glyim Core Team  

### 1.1 Vision & Elevator Pitch
**Vision:** To transform macro authoring from opaque text manipulation into a tangible, type-aware, projectional sculpting process, deeply integrated into the IDE.  
**Elevator Pitch (Moore Template):** For framework authors frustrated by the "compile-pray" loop of writing macros in Rust or Scala, Glyim provides a projectional macro environment. Unlike existing tools, Glyim’s LSP provides full type inference and autocomplete *inside* `quote!` blocks, features a live Virtual Expansion View that updates as you type, and allows bidirectional editing—letting developers directly modify generated code while the compiler automatically rewrites the macro definition.

### 1.2 Problem & Business Context
Glyim v0.15.0 achieved macro-transparent *linting* by analyzing fully expanded MIR. However, the act of *writing* macros remains trapped in the 1970s paradigm: the IDE treats `quote!` blocks as opaque strings. Developers cannot hover over spliced variables (`#ident`) to see their types, cannot set breakpoints inside macro logic, and cannot easily visualize the AST they are building. We must leverage our foundational advantages—Rowan (v0.2.0), Salsa (v0.11.0), and Span Hygiene (v0.11.0)—to elevate the IDE experience to match the theoretical power of the macro system itself.

### 1.3 Target Users (Target Group)
1. **The Macro Architect:** Writes complex, multi-stage procedural macros. Needs to understand the exact shape of the generated AST without mental gymnastics.
2. **The Framework Adopter:** Uses third-party macros. Needs to understand what a macro does, and tweak its generated output without forking the macro's source code.
3. **The Debugging Engineer:** Tracks down why a macro generates invalid MIR. Needs to step through the macro execution line-by-line inside the IDE.

### 1.4 Desired Outcomes & Success Metrics (OKRs)
* **Objective 1: Eliminate macro context blindness.**
  * **KR 1:** Hovering over a spliced variable (`#var`) inside a `quote!` block will successfully resolve and display its inferred type in 100% of non-recursive cases.
* **Objective 2: Achieve seamless projectional editing.**
  * **KR 1:** 80% of active macro authors will use the "Edit Expansion -> Sync Definition" workflow to modify macro output at least once per session.
* **Objective 3: Native IDE macro debugging.**
  * **KR 1:** Debugging a procedural macro via DAP will require zero CLI tools; all stepping and AST inspection will happen natively inside VS Code/Neovim.

### 1.5 Goals and Non-Goals (Scope / Anti-scope)
**Goals (P0/P1):**
* Implement Type-Aware `quote!` blocks (LSP autocomplete/hover inside macros).
* Implement Virtual Expansion View (live read-only mirror of expanded output).
* Implement Bidirectional AST Sync (editing expansion updates the macro definition).
* Implement LSP-Native Macro Stepping (DAP integration).
* Implement Macro Call-Site Diffing (Sketch-n-Sketch for macros).

**Non-Goals (Anti-scope):**
* **No Visual Node-Based Editors:** We are not building a drag-and-drop node editor. We remain text-first, using virtual text documents as the projectional surface.
* **No Cross-Language Macro Stepping:** Stepping into Wasm components (v0.13.0) from inside a macro is out of scope; we only step through native Glyim macro logic.
* **No Automatic Macro Generation (AI):** The LSP will not use AI to write macros; all projectional features are deterministic compiler operations.

## Document 2: Business & Stakeholder Requirements (Level 1)

**Document ID:** GLYIM-V160-BRS-001  

### 2.1 Stakeholders and User Classes
| User Class | Characteristics | Primary Needs | Disfavored Behaviors |
| :--- | :--- | :--- | :--- |
| **Macro Architect** | Writes `quote!` spanning hundreds of lines. | Seeing the inferred type of `#ast_node`; autocomplete for AST builder methods. | Keeping a mental map of what `#var` resolves to; running CLI to check syntax. |
| **Framework Adopter** | Uses a derive macro but needs to tweak its output. | Modifying the generated `impl` block without learning the macro's source code. | Copy-pasting generated code into a file, editing it, and manually maintaining the divergence. |
| **Debugging Engineer** | Tracks down invalid MIR generated by complex logic. | Pausing execution inside a procedural macro and inspecting the Rowan tree state. | Adding `println!` statements to the macro definition and recompiling. |

### 2.2 Stakeholder Needs & Value Proposition
* **SN-001 (Macro Architect):** "I need to know what methods are available on the variable I'm splicing." -> *Value Prop: Type-Aware `quote!` Salsa queries providing full IDE intelligence inside macro blocks.*
* **SN-002 (Framework Adopter):** "I want to add a method to a derived builder without touching the macro's crate." -> *Value Prop: Bidirectional AST Sync modifying the virtual expansion to rewrite the macro definition.*
* **SN-003 (Debugging Engineer):** "I need to see what AST node the macro is currently constructing." -> *Value Prop: DAP integration exposing the live Rowan Green Tree during macro execution.*

### 2.3 Ubiquitous Language (Glossary Seed)
* **Type-Aware Quoting:** The ability of the LSP to resolve the types of variables and AST nodes spliced into a `quote!` block using the Salsa database.
* **Virtual Expansion View:** A temporary, read-only text document generated by the LSP representing the expanded output of a macro for a specific call site.
* **Bidirectional AST Sync:** The mechanism by which edits made to the Virtual Expansion View are diffed against the current AST, and the resulting delta is mathematically applied to the `quote!` token stream in the macro definition.
* **Macro Call-Site Diffing:** A Sketch-n-Sketch feature that instantly calculates and displays the git-style diff of all macro expansions across a workspace when the macro definition changes.

## Document 3: Software Requirements Specification (Level 2)

**Document ID:** GLYIM-V160-SRS-001  
**Traceability:** Maps to Level 1 SN-001 through SN-003 and Level 0 OKRs.

### 3.1 Functional Requirements: Type-Aware Macro Authoring

**FR-MACRO-001: Salsa Context Isolation**
* **Description:** The LSP shall create an isolated analysis context for procedural macro definitions.
* **Fit Criterion:** When parsing a `fn derive_macro`, the LSP must instantiate a child Salsa database containing the macro's parameters and body. This context must resolve the types of `#ident` and `#ast` spliced variables without requiring the macro to actually execute.

**FR-MACRO-002: Intra-Macro Autocomplete**
* **Description:** The LSP shall provide autocomplete inside `quote!` blocks.
* **Fit Criterion:* Triggering autocomplete after `#ident.` must show methods available on the `Ident` type. Triggering autocomplete inside a block must show AST builder methods (e.g., `append`, `node`).

**FR-MACRO-003: Intra-Macro Hover**
* **Description:** Hovering over spliced variables must show type information.
* **Fit Criterion:* Hovering over `#input` in `quote! { #input.clone() }` must display the resolved type of `input` (e.g., `proc_macro2::TokenStream`).

### 3.2 Functional Requirements: Projectional Expansion View

**FR-PROJ-001: Virtual Expansion Generation**
* **Description:** The LSP shall generate a Virtual Expansion View for a given macro invocation.
* **Fit Criterion:* Triggering "Show Expansion" on a macro call site must generate a virtual document containing the pretty-printed, fully expanded Rowan AST resulting from that invocation.

**FR-PROJ-002: Live Update (Sketch-n-Sketch)**
* **Description:** The Virtual Expansion View must update live as the user types.
* **Fit Criterion:* If the user changes the input struct passed to a derive macro, the Virtual Expansion View must update within 500ms without requiring a manual "refresh" action.

**FR-PROJ-003: Diagnostics in Expansion View**
* **Description:** Compiler errors in the expanded code must be shown in the virtual view.
* **Fit Criterion:* If the macro generates invalid Glyim code (e.g., missing trait impl), the Virtual Expansion View must display red squiggles on the exact lines of the generated code.

### 3.3 Functional Requirements: Bidirectional AST Sync

**FR-SYNC-001: Forward Edit Propagation**
* **Description:** Edits in the Virtual Expansion View must update the macro definition.
* **Fit Criterion:* If a user adds a method `fn foo() {}` in the Virtual Expansion View, the LSP must diff the expanded AST against the new AST, calculate the token delta, and insert a corresponding `quote! { fn foo() {} }` block into the macro definition's source code.

**FR-SYNC-002: Sync Conflict Resolution**
* **Description:** The sync mechanism must handle ambiguous AST diffs.
* **Fit Criterion:* If a change in the expansion view maps to multiple valid locations in the `quote!` block, the LSP must present the user with a quick-pick list of valid insertion points rather than failing.

**FR-SYNC-003: Sync Scope Limitation**
* **Description:** Bidirectional sync is restricted to deterministic, pure-AST modifications.
* **Fit Criterion:** The sync mechanism must refuse to apply changes that rely on runtime values (e.g., logic inside an `if` statement based on a string length). It must emit an error: "Cannot sync modification: expression depends on runtime state."

### 3.4 Functional Requirements: IDE-Native Debugging & Diffing

**FR-DBG-001: DAP Suspension at AST Nodes**
* **Description:** The compiler must support suspending execution inside a procedural macro.
* **Fit Criterion:* When a breakpoint is set on a line inside a macro definition, the DAP adapter must pause execution, serialize the current state of the Rowan Green Tree being constructed, and send it to the IDE.

**FR-DIFF-001: Workspace Call-Site Diffing**
* **Description:** Changing a macro definition must show its impact across the workspace.
* **Fit Criterion:* When a macro definition is saved, the LSP must calculate the expanded output diff for all call sites in the workspace and surface a summary diagnostic: "Macro `X` changes expanded output at 4 call sites. View Diffs."

### 3.5 Non-Functional Requirements (NFRs)

**NFR-PERF-001 (Expansion Latency):**
* Generating the Virtual Expansion View for a macro generating a 500-line struct must take **<100ms**.

**NFR-PERF-002 (Sync Calculation Time):**
* The Bidirectional AST diff calculation must take **<50ms** for typical structural changes.

**NFR-MEM-001 (Memory Overhead):**
* Maintaining the Virtual Expansion AST for 10 open macro call sites must consume **<200MB** of additional RAM in the LSP process.

## Document 4: Architecture & Design Specification (Level 3)

**Document ID:** GLYIM-V160-ARCH-001  

### 4.1 Context & System Boundary (C4 Level 1)
v0.16.0 introduces a new subsystem to the LSP: The Projectional Engine. It sits directly on top of the Salsa database and Rowan parser, extending the v0.11.0 architecture to treat macro definitions as first-class, editable entities.

### 4.2 Architecture Decision Records (ADRs)

**ADR-030: Isolated Salsa Scopes for Macro Context**
* **Context:** To provide type-awareness inside `quote!`, the LSP needs to type-check the macro's internal logic without compiling a real binary or waiting for a call site.
* **Decision:* When the LSP encounters a `fn derive_macro`, it creates a "Macro Definition Database" (a child Salsa scope). It artificially satisfies the macro's input parameters (e.g., creates dummy `TokenStream` and `Ident` types) and runs HIR type-checking purely on the macro body. This allows the LSP to resolve the types of `#ident` without executing the macro.
* **Consequences:*
  * *(+)* Zero false positives on type inference inside macros.
  * *(+)* Leverages existing Salsa infrastructure perfectly.
  * *(-)* Cannot infer types that strictly depend on the actual input AST (a highly acceptable edge case).

**ADR-031: Bidirectional Tree-Diffing via Rowan**
* **Context:** Mapping edits from an expanded text document back to a `quote!` block is mathematically complex.
* **Decision:** When the user edits the Virtual Expansion View, the LSP parses the edit into a new Rowan Green Tree. It then runs a custom diffing algorithm (similar to `TreeDiff`) between the old expanded tree and the new one. The diff algorithm outputs a sequence of `Insert` and `Delete` operations on specific AST node types. The LSP maps these AST operations back to the specific `quote!` invocation spans and rewrites the macro source text.
* **Consequences:**
  * *(+)* Handles complex reorderings and multiline insertions gracefully.
  * *(-)* If the user completely rewrites the logic flow, the diff might become ambiguous, triggering the manual resolution UI (FR-SYNC-002).

**ADR-032: DAP Suspension at AST Nodes**
* **Context:** Standard debuggers suspend at machine instructions or source lines. Macros operate on AST nodes.
* **Decision:** Extend the MIR interpreter (v0.13.0 CTFE) with DAP hooks. When a breakpoint is hit inside a proc-macro, the compiler halts, takes the partially constructed Rowan Green Tree, serializes it to JSON, and sends a custom DAP `customEvent` to the IDE. The IDE renders this tree in a dedicated "Macro AST Inspector" panel.
* **Consequences:*
  * *(+)* Developers see exactly what tree is being built, matching their mental model of `quote!`.
  * *(-)* Serializing a 10,000-node Rowan tree to JSON might pause the IDE for a few hundred milliseconds.

### 5.1 Specification by Example (SbE): Type-Aware `quote!`

**Feature:** IDE provides intelligence inside procedural macros

```gherkin
Scenario: Autocomplete inside a quote block
  Given a procedural macro parameter `input: TokenStream`
  And the macro body contains: `quote! { #input.is_empty() ... }`
  When the developer triggers autocomplete after `#input.`
  Then the LSP must suggest methods available on the `TokenStream` type (e.g., `is_empty`, `parse`, `peek`)
  And the suggestion must include doc comments from the `TokenStream` type

Scenario: Hover over spliced variable shows resolved type
  Given a macro body: `let ty = parse_type(#input); quote! { fn foo() -> #ty {} }`
  When the developer hovers over `#ty`

```

### 5.2 Specification by Example (SbE): Bidirectional AST Sync

**Feature:** Editing expanded output updates the macro definition

```gherkin
Scenario: Adding a method in the Expansion View updates the macro definition
  Given a macro that generates:
    """
    pub struct MyStruct { name: String }
    """
  And the user opens the Virtual Expansion View
  When the user edits the Virtual Expansion View to add `fn greet(&self) {}`
  And the LSP applies the Bidirectional Sync
  Then the source code of the macro definition must be updated
  And the macro definition must now contain `quote! { fn greet(&self) {} }`
  And the LSP must emit an informational diagnostic: "Synced 1 addition from Expansion View to macro definition"
```

### 5.3 Specification by Example (SbE): Workspace Call-Site Diffing

**Feature:** Changing a macro shows impact across the workspace

```gherkin
Scenario: Modifying a macro definition triggers workspace-wide diffs
  Given a macro `#[derive(Builder)]` used in 3 files in the workspace
  And the macro definition is updated to remove the generation of a `new()` method
  When the developer saves the macro definition file
  Then the LSP must emit a workspace-wide summary diagnostic: "Macro `Builder` expansion changed at 3 call sites"
  And the developer must be able to click the diagnostic to see a unified diff of the removed `new()` methods across all 3 files
```

### 5.4 Specification by Example (SbE): LSP-Native Macro Debugging

### 5.6 Test Strategy & Edge Cases
* **Tree-Diff Fuzzing:** A fuzzer will generate pairs of random Rowan trees (representing old and new macro expansions) and feed them to the Tree-Diff engine. The engine must never crash, hang, or produce a diff that results in an invalid `quote!` syntax tree.
* **Sync State Rejection:** Deliberately attempt a Bidirectional Sync that relies on a macro's internal `if` condition. The LSP must reject it with FR-SYNC-003 ("depends on runtime state") and *not* modify the source code.
* **DAP Serialization Limits:** Create a macro that builds a deeply nested AST (100k nodes). Verify that the DAP `customEvent` serialization truncates gracefully or streams in chunks, preventing the LSP from running out of memory.
