# Glyim v0.18.0 Specification Suite: The Polished Perfection Release

This document suite constitutes the fully fledged specification for Glyim v0.18.0. It applies evidence-based requirements engineering, architecture documentation, and behavioral specification practices, structured across five specification levels to ensure strategic alignment, stakeholder satisfaction, technical feasibility, and verifiable delivery.

---

## Document 1: Vision & Strategic Alignment (Level 0)

**Document ID:** GLYIM-V180-VISION-001  
**Version:** 1.0  
**Status:** Approved  
**Author:** Glyim Core Team  

### 1.1 Vision & Elevator Pitch
**Vision:** To eliminate developer friction and cognitive load by transforming compiler diagnostics from hostile logs into actionable, inline teaching moments, and completely removing context-switching between the IDE and the terminal.  
**Elevator Pitch (Moore Template):** For developers dissatisfied with Rust's dense, wall-of-text compiler errors and the constant context-switching to a terminal, Glyim provides a polished, zero-friction development loop. Unlike existing tools, Glyim natively renders cognitive-load-aware error callouts in the editor, streams test results directly into the IDE problem pane via the Debug Adapter Protocol, and provides browser-native project scaffolding that gets a "Hello World" running in under 30 seconds without touching a CLI.

### 1.2 Problem & Business Context
Glyim v0.17.0 delivered an "Artisan Stdlib" with type-state patterns and declarative builders. However, empirical UX research (cognitive load theory, flow state psychology) indicates that technical excellence is wasted if the user experience is friction-heavy. Studies show that presenting more than 3-4 distinct concepts in a single error message causes developer disengagement, and that context-switching to a terminal breaks the "flow state," slowing iteration speed. We must apply UX research principles to the compiler frontend and IDE integration.

### 1.3 Target Users (Target Group)
1. **The Flow-State Seeker:** Experienced developer who hates breaking focus to open a terminal. Needs to run tests and view errors without looking away from the code.
2. **The Confused Migrator:** Developer upgrading Glyim editions or migrating from Rust/Go. Needs to see exactly what code changes are required without guessing.
3. **The Junior Onboarder:** Needs to see a successful build in 30 seconds to build confidence in the language.

### 1.4 Desired Outcomes & Success Metrics (OKRs)
* **Objective 1: Achieve zero-error hostility.**
  * **KR 1:** 100% of compiler errors must feature a 1-line cognitive-load-appropriate summary at the top of the inline callout.
* **Objective 2: Eliminate terminal context-switching.**
  * **KR 1:** 90% of daily development tasks (test, build, publish) must be executable via IDE-native UI buttons. Target: <1 terminal context switch per session.
* **Objective 3: Achieve instant onboarding.**
  * **KR 1:** Creating and running a new project from the IDE must take < 30 seconds end-to-end.

### 1.5 Goals and Non-Flags (Scope / Anti-scope)
**Goals (P0/P1):**
* Implement Layered Error UX (1-line summary, code fix, expandable technical details).
* Implement IDE-Native Task Runner (DAP terminal streaming, status bar actions).
* Implement Browser-Native Project Initialization (IDE modal -> Playground instantiation).
* Implement Semantic Migration UI (Side-by-side edition diffing).
* Implement Universal Quick Fix Guarantee (every diagnostic paired with a valid Code Action).
* Implement Error Severity Funnels (Warning -> Error escalation).

**Non-Goals (Anti-scope):**
* **No New Compiler Checks:** We are not adding new semantic lints or type checks. We are only changing *how* existing errors are presented.
* **No Custom Terminal Emulators:** We are wrapping the native terminal, not replacing it with an in-editor terminal emulator.
* **No Changes to `glyim test` Logic:** We are changing the *invocation* and *output routing*, not how tests are executed.

## Document 2: Business & Stakeholder Requirements (Level 1)

**Document ID:** GLYIM-V180-BRS-001  

### 2.1 Stakeholders and User Classes
| User Class | Characteristics | Primary Needs | Disfavored Behaviors |
| :--- | :--- | :--- | :--- |
| **Flow-State Seeker** | High-velocity typist, hates context switches. | Run tests without moving hands from the keyboard. | Opening a terminal, running a command, reading output, switching back. |
| **Confused Migrator** | Upgrading Glyim editions or migrating a Rust crate. | Visual diff of exact line changes required. | Reading a 20-page migration guide and guessing what breaks. |

### 2.2 Stakeholder Needs & Value Proposition
* **SN-001 (Flow-State Seeker):** "I want to see my test output in the IDE, not a raw terminal buffer." -> *Value Prop: DAP-streamed inline terminal views.*
* **SN-002 (Confused Migrator):** "I want to know exactly which of my 50 files will break when I change one setting." -> *Value Prop: Semantic Migration UI with "Accept All" capability.*

### 2.3 Ubiquitous Language (Glossary Seed)
* **Layered Error UX:** An error presentation paradigm where a 1-line cognitive summary is shown first, followed by a suggested fix, and technical details are collapsed by default.
* **DAP Terminal Streaming:** Routing the standard output/error streams of the compiler directly into the IDE's built-in terminal panel via the Debug Adapter Protocol.
* **Semantic Diffing:** Calculating the exact structural differences between the current codebase and a target state (e.g., a new Edition), rendered as an inline diff.
* **Context Switching:** The mental disruption caused by changing windows/panes from the code editor to an external tool.

## Document 3: Software Requirements Specification (Level 2)

**Document ID:** GLYIM-V180-SRS-001  
**Traceability:** Maps to Level 1 SN-001 through SN-003 and Level 0 OKRs.

### 3.1 Functional Requirements: Layered Error UX

**FR-UX-001: 1-Line Summary Generation**
* **Description:** The compiler must generate a short, cognitive-load-appropriate summary for every diagnostic.
* **Fit Criterion:** Every error message must include a 1-line summary string (e.g., "Missing Trait Implementation"). If no obvious summary exists, the LSP must generate one based on the error code and splices.

**FR-UX-002: Inline Code Fix Display**
***Description:** Errors must display a valid code fix directly in the callout.
* **Fit Criterion:** The inline callout must feature a clickable `[Apply Fix]` button that applies the associated LSP Code Action. The fix must be verified by the compiler before being presented.

**FR-UX-003: Expandable Technical Details**
* **Description:** The technical details of an error must be collapsed by default.
***Fit Criterion:* The callout must show a `[Details ▶]` chevron. Clicking it must expand to show the full raw error message (similar to v0.17.0) and the v0.16.0 Macro Expansion View if the error originates from macro-generated code.

**FR-UX-004: Error Severity Funnels**
* **Description:** Errors must escalate from warnings to hard errors based on frequency.
* **Fit Criterion:** A specific diagnostic (e.g., "unused variable") must begin as a Warning. If the LSP detects the same unused variable in 5 different files in the workspace, it escalates to an Error.

### 3.2 Functional Requirements: IDE-Native Workflow

**FR-IDE-001: Status Bar Task Actions**
* **Description:** The IDE status bar must contain native "Run" and "Test" buttons.
* **Fit Criterion:** Clicking "Run" must invoke `glyim build` via the LSP. Clicking "Test" must invoke `glyim test` via the LSP. The output must stream to the IDE's integrated terminal panel, not a separate system terminal.

**FR-IDE-002: DAP Terminal Streaming**
***Description:** The LSP must route all stdout/stderr to the IDE via DAP.
***Fit Criterion:** The LSP server must act as a DAP backend, creating virtual processes for `build` and `test` and forwarding `Output` events to the IDE client. The IDE must render this in a panel that looks and behaves exactly like a local terminal.

**FR-IDE-003: Context-Switch Metrics**
* **Description:** The LSP must track and report IDE context switches.
***Fit Criterion:* If the developer switches to a system terminal instead of using the native IDE buttons, the LSP may emit a low-severity "UX" diagnostic: "Using external terminal for 'glyim test'. Try using the status bar Run button for inline output."

### 3.3 Functional Requirements: Frictionless Integration

**FR-INT-001: Browser-Native Project Initialization**
* **Description:** The IDE must provide a native UI dialog for creating new projects.
***Fit Criterion:** A dialog must allow selecting the project name, Glyim Edition, and a list of dependencies. Upon clicking "Create", a background process must run the CLI commands. When finished, the IDE must open the generated `main.gy` file and automatically launch it in the v0.9.0 WASM Playground.

**FR-INT-002: "One-Click Publish" Action**
***Description:** A single action to build, test, and publish.
***Fit Criterion:* A "Publish" button in the status bar must invoke `glyim build -> glyim test -> glyim publish` in sequence. If a step fails, the action stops, and the error is rendered in the Problem pane.

**FR-INT-003: Dependency Quick-Add**
***Description:** Adding a dependency should be a 1-step IDE action.
***Fit Criterion:* Right-clicking the `glyim.toml` file and selecting "Add Dependency" must open a modal to search the registry and inject the dependency line, updating the file via the v0.11.0 Assist API.

### 3.4 Functional Requirements: Semantic Migration UI

**FR-MIG-001: Semantic Diff Calculation**
* **Description:** Changing the Edition in `glyim.toms` must trigger a structural diff.
***Fit Criterion:* The LSP must calculate the exact AST differences between the current codebase and the target Edition's expectations. It must group changes by module and function signature.

**FR-MIG-002: Side-by-Side Migration Pane**
***Description:** The IDE must render the Semantic Diff in a split pane.
***Fit Criterion:* A side pane must open showing the current code on the left and the proposed changes on the right. Lines should be syntax-highlighted. Clicking a change must navigate to the source file.

**FR-MIG-003: "Accept All Changes" Action**
***Description:** A mechanism to apply all proposed migration changes at once.
***Fit Criterion:* The side pane must have an "Accept All" button that applies all proposed text edits to the workspace using the v0.15.0 Autonomous Fix Engine.

### 3.5 Non-Functional Requirements (NFRs)

**NFR-UX-001 (Error Resolution Time):**
* Resolving an error by applying the suggested Quick Fix must take **<5 seconds** of user interaction time (navigating to the fix, clicking apply).

**NFR-PERF-001 (DAP Streaming Latency):**
* stdout/stderr output must appear in the IDE terminal panel within **<100ms** of generation.

**NFR-PERF-002 (Migration Diff Calculation):**
* Calculating the Semantic Diff for a 100k LOC workspace must take **<50ms** of CPU time in the LSP Salsa database.

**NFR-PERF-003 (Scaffold Latency):**
* The Browser-Native Init process must open a working Playground instance in **<30 seconds**.

## Document 4: Architecture & Design Specification (Level 3)

**Document ID:** GLYIM-V180-ARCH-001  

### 4.1 Context & System Boundary (C4 Level 1)
v0.18.0 is entirely an IDE/LSP release. It modifies the LSP server protocol, introduces a specific "Presentation Layer" that intercepts compiler diagnostics, and creates a DAP backend wrapper around CLI commands. It heavily utilizes the v0.14.0 Doc Engine and v0.15.0 Autonomous Fix Engine.

### 4.2 Architecture Decision Records (ADRs)

**ADR-036: State-Machine for Layered Errors**
* **Context:** Errors need state (warning vs. error) and state transitions (expanded vs. collapsed).
* **Decision:** Implement an in-memory state machine in the LSP for each diagnostic ID. State transitions: `Pending (warning)` -> `Active (warning)` -> `Resolved (fixed)`. The LSP only renders the 1-line summary if the state is `Active`, and expands technical details if the user explicitly clicks Expand.
* **Consequences:**
  * *(+)* Ensures UI does not flicker as the compiler emits multiple related warnings.
  * *(-)* Requires careful state synchronization between the LSP main loop and the diagnostic generation pass.

**ADR-037: DAP Virtual Terminal Streaming Architecture**
***Context:** Standard DAP terminals are system-level and difficult to securely embed inside an IDE extension.
***Decision:** The LSP acts as the DAP `Launch` target. It creates pseudo-processes. When the CLI runs, the LSP intercepts `stdout/stderr`, wraps it in DAP `Output` events, and sends them to the client. The IDE client maps these to a visual terminal emulator that renders the text.
* **Consequences:*
  * *(+)* Completely secure and sandboxed; no shell injection risks.
  * *(-)* Terminal escape sequences (ANSI colors, interactive prompts) might not render perfectly in a canvas-based terminal emulator (accepted trade-off for a v0.18.0 release).

**ADR-038: Scaffold Engine via CLI Subprocess & WASM Playground**
***Context:** Creating a project requires running CLI commands, but we want to avoid making the user install a toolchain.
* **Decision:** The IDE spawns the `glyim` CLI in an isolated, pre-bundled WASM environment (leveraging the v0.13.0 Component Model). It sends a `create_project` command. It reads the output JSON and generates the file tree locally using the LSP TextEdits API. Finally, it requests the v0.9.0 Playground to load the generated `main.gym` via a custom protocol.
* **Consequences:**
  * *(+)* Zero local dependencies for the user.
  * *(+)* Works identically on WebIDE, VS Code Web, and Desktop.
  * *(-)* Requires bundling the CLI into a 15MB+ WASM binary inside the IDE extension.

**ADR-039: Semantic Diffing via Salsa AST Diffing**
***Context:** Standard text diffing fails on macro-generated code and type inference shifts.
***Decision:** The LSP generates the *current* AST for a module and the *target* AST for that module (based on Edition rules). It runs a structural diff (insertions/deletions) at the AST node level. It renders this diff in the side pane as text.
* **Consequences:*
  * *(+)* Ignores formatting changes, focusing entirely on semantic differences.
  * *(+)* Works perfectly with v0.16.0 macro-generated code.
  * *(-)* Requires generating a target-state AST, which requires running type-checking for the target Edition.

### 5.1 Specification by Example (SbE): Layered Error UX

**Feature:** Error callouts provide cognitive-load-aware summaries and guaranteed fixes

```gherkin
Scenario: Error displays a 1-line summary and a guaranteed fix
  Given a file containing a struct `Foo` missing the `Clone` trait
  When the compiler emits the diagnostic
  Then the IDE inline callout must display:
    """
    [!]. Missing Trait Implementation
    The struct `Foo` requires `Clone`. Deriving it takes 1 line of code.
    [Fix: #[derive(Clone)] [Apply Fix] [Details ▶]
    """
  And clicking "Apply Fix" must insert `#[derive(Clone)]` above `pub struct Foo`
  And the code must compile successfully
```

**Scenario: Technical details are collapsed by default
  Given an active error callout showing the 1-line summary
  When the user looks at the callout
  Then the raw compiler error text must NOT be visible
  And a chevron icon (`▶`) must be visible
  And clicking the chevron must expand the callout to show the raw compiler output
```

### 5.6 Test Strategy & Edge Cases

* **State Machine Fuzzing:** Send 1,000 rapid diagnostic events to the LSP and measure if the UI thread blocks or drops events. The state machine must handle out-of-order arrivals and cancellation without crashing.
* **DAP Terminal Emulation Fidelity:** Deliberately execute code in the emulated terminal that uses complex ANSI escape codes, raw control characters, and interactive prompts. The emulator must render them as closely as possible, falling back gracefully to text.
* **Scaffold State Recovery:** Kill the IDE process during the "New Project" scaffold. Reopen the project. The LSP must detect the incomplete scaffold state and offer a "Resume Scaffold" action rather than starting from scratch.
* **Diff Ambiguity Resolution:** Deliberately generate a diff where a 20-line macro expansion changes one line but the type system resolves it to the same MIR. The Semantic Diff must recognize this as a no-op and not show the line in the diff.
