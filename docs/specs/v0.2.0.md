# Glyim v0.2.0 Architecture & Design Specification

## 1. Context & Scope

### What v0.1.0 Delivered
A working end-to-end pipeline: `.g` source → lexer → Pratt parser → HIR → LLVM IR → native executable. The compiler can compile `main = () => 42` and run it. The architectural runway (11 crates, strict DAG, CAS traits, macro context traits, hygiene) is in place.

### The Problem: v0.1.0 Is a Toy
Users cannot:
- Declare mutable variables (`let mut x = 5; x = x + 1`)
- Branch (`if x > 0 { ... } else { ... }`)
- Print anything to stdout
- Work with strings
- Get helpful error messages (errors say "expected ) at bytes 5..7" with no source snippet)
- See more than one error per compilation (parser stops on first error)
- Scaffold a project without manual file creation

Internally, the parser skips Rowan entirely (parses to a raw enum), which blocks LSP and formatter work. The `glyim run` command requires `cc`/`gcc` on PATH, which is a friction point for first-time users.

### The Solution: v0.2.0 — "Feels Like a Real Language"
This spec adds the minimal feature set so a user can write a 50-line program, get helpful errors, and have it feel like a real development experience. It also shores up the internal testing foundation so the compiler doesn't rot.

### Scope of v0.2.0
**Included:**
- `let` / `let mut` bindings with proper stack allocation
- `if` / `else` as expressions with LLVM branching
- String literals as borrowed `&str` (fat pointer: `{ i8*, i64 }`)
- `println` and `assert` as compiler built-in intrinsics
- Beautiful error messages via `ariadne` with source snippets
- Error recovery in the parser (multiple errors per compile)
- Rowan CST rewrite (parser builds Rowan trees first, then lowers to AST)
- JIT execution via MCJIT (removes `cc`/`gcc` dependency for `glyim run`)
- UI test framework (`.g` + `.stderr` snapshot tests)
- `glyim init` project scaffolding
- `use` parsing (no resolution — skeleton only)
- Fuzz testing setup for lexer and parser

**Excluded:**
- Struct types (v0.3.0)
- Generic types / type inference (v0.3.0)
- Pattern matching (v0.3.0)
- Real macro execution (v0.3.0)
- LSP / formatter (v0.4.0)
- Distributed CAS (v0.5.0)
- `@rust()` FFI beyond built-in intrinsics (v0.3.0)

## 2. Goals and Non-Goals (ASRs)

### Goals

| ID | Statement |
|----|-----------|
| **ASR-010** | Users can declare immutable and mutable local bindings with `let` / `let mut`. |
| **ASR-011** | Users can branch with `if` / `else` as expressions that produce values. |
| **ASR-012** | Users can write string literals (`"hello"`) and pass them to built-ins. Strings are borrowed `&str` fat pointers. |
| **ASR-013** | `println` is a compiler built-in that writes to stdout. It accepts `&str` or `i64`. |
| **ASR-014** | `assert` is a compiler built-in that aborts on failure with an optional message. |
| **ASR-015** | Every compiler error includes the source text, line/column, `^^^` underline, and a help suggestion where applicable. |
| **ASR-016** | The parser reports all errors it can find in a single compilation, not just the first. |
| **ASR-017** | The parser builds Rowan CST (Green/Red trees) as its primary output; the AST enum is derived via lowering. |
| **ASR-018** | `glyim run` executes via LLVM JIT (MCJIT) without requiring a C compiler on PATH. |
| **ASR-019** | Every diagnostic is tested via a snapshot-based UI test framework (`.g` + `.stderr`). |
| **ASR-020** | `glyim init <name>` scaffolds a project with `glyim.toml` and `src/main.g`. |

### Non-Goals
- **No heap allocation:** Strings are global constants referenced by fat pointers. No `String` type, no concatenation.
- **No type checker:** The compiler uses structural inference at codegen time (HIR variant → LLVM type). No named types, no type errors.
- **No closures:** Lambdas exist but cannot capture variables from enclosing scope.
- **No unicode handling beyond UTF-8 pass-through:** String operations are byte-level.
- **No `glyim build` via JIT:** `glyim build` still emits object files for AOT compilation. JIT is for `glyim run` only.

## 3. The Design

### 3.1 Statement/Expression Distinction

v0.1.0's HIR had only expressions. v0.2.0 introduces statements:

```
HirStmt
├── Let { name: Symbol, mutable: bool, value: HirExpr }
├── Assign { target: Symbol, value: HirExpr }
└── Expr(HirExpr)
```

A block becomes `HirExpr::Block(Vec<HirStmt>)`. The last statement in a block, if it is an `Expr` statement, becomes the block's value. This matches Rust's semantics.

**Codegen for `let`:**
```
let x = 42        →  %x = alloca i64
                    store i64 42, i64* %x

let mut y = 10     →  %y = alloca i64
                    store i64 10, i64* %y

y = y + 1          →  %0 = load i64, i64* %y
                    %1 = add i64 %0, 1
                    store i64 %1, i64* %y
```

### 3.2 String Representation

Strings are **borrowed fat pointers**: `{ i8* data, i64 len }` in LLVM.

```
println("hello")
```
Codegen:
```llvm
@str.0.data = private constant [5 x i8] c"hello"
@str.0 = private constant { i8*, i64 } { i8* getelementptr ([5 x i8], [5 x i8]* @str.0.data, i32 0, i32 0), i64 5 }

; println calls the runtime shim:
call void @glyim_println_str({ i8*, i64 } @str.0)
```

**Runtime shim** (`glyim_println_str`): emitted by the compiler as LLVM IR, not as a separate Rust file. It calls the `write` syscall directly:
```llvm
define void @glyim_println_str({ i8*, i64 } %s) {
  %ptr = extractvalue { i8*, i64 } %s, 0
  %len = extractvalue { i8*, i64 } %s, 1
  call i64 @write(i32 1, i8* %ptr, i64 %len)   ; stdout fd=1
  call i64 @write(i32 1, i8* getelementptr([2 x i8], [2 x i8]* @newline, i32 0, i32 0), i64 1)
  ret void
}
@newline = private constant [2 x i8] c"\0A"
```

For `println(42)` (integer overload), the compiler detects the HIR variant and calls a separate shim:
```llvm
define void @glyim_println_int(i64 %val) {
  ; Convert i64 to decimal string on the stack, then call write
  ; ... (straightforward but tedious LLVM IR — ~30 instructions)
}
```

**Declaration: `write` is an external function:**
```llvm
declare i64 @write(i32, i8*, i64)
```

### 3.3 If/Else Expressions

```llvm
; let x = if cond { 1 } else { 2 }
  %cond_val = ... ; evaluate condition
  %cond_bool = icmp ne i64 %cond_val, 0
  br i1 %cond_bool, label %then, label %else

then:
  br label %merge         ; (with 1 as the value)

else:
  br label %merge         ; (with 2 as the value)

merge:
  %result = phi i64 [ 1, %then ], [ 2, %else ]
  store i64 %result, i64* %x
```

**If without else:** the else branch produces `i64 0` (unit value). The block's value is still a phi node.

### 3.4 Rowan-First Parser Architecture

```
Source → Lexer → TokenStream → Parser → GreenNode → SyntaxNode (CST) → AST Lower → Ast
```

The parser uses Rowan's `GreenNodeBuilder` to construct the tree incrementally. After parsing, a separate pass walks the `SyntaxNode` (red tree) and produces the typed `Ast` enum.

**Key benefit:** Error recovery can leave "error nodes" in the tree. The AST lowerer skips error nodes but the tree structure is preserved for diagnostics (exact span of the bad subtree).

**Changes to `glyim-parse`:**
- New module `cst_builder.rs`: wraps `GreenNodeBuilder`, provides `start_node(kind)`, `token(kind, text)`, `finish_node()`
- New module `ast_lower.rs`: `fn lower_cst_to_ast(root: &SyntaxNode) -> Result<Ast, Vec<ParseError>>`
- Existing `parser.rs` rewritten to call `CstBuilder` instead of building `ExprNode` directly
- Existing `ast.rs` stays the same (the output type doesn't change)

### 3.5 Error Recovery Strategy

**Sync points** — tokens where the parser can safely resume after an error:
- `fn` keyword (start of function definition)
- `let` keyword (start of binding)
- `}` (end of block)
- EOF

**Algorithm:**
1. On parse error, emit a `ParseError` diagnostic
2. Insert an `Error` node into the Rowan tree (with the consumed tokens as children)
3. Discard tokens until a sync point is found
4. Resume parsing from the sync point

This means a file with 3 typos produces 3 errors in a single compilation.

### 3.6 JIT Execution

Replace the `cc` linker path in `glyim run` with Inkwell's ORC JIT:

```
Source → Parse → HIR → LLVM IR (in memory) → ORC JIT → call main() → exit code
```

**Implementation:**
```rust
use inkwell::execution_engine::ExecutionEngine;

let context = Context::create();
let module = context.create_module("glyim_jit");
// ... codegen into module ...
let engine = module.create_jit_execution_engine(OptimizationLevel::None)?;
let main_fn: JitFunction<unsafe extern "C" fn() -> i32> =
    unsafe { engine.get_function_value("main").unwrap().get_function() };
let exit_code = unsafe { main_fn() };
```

**Runtime functions** (`glyim_println_str`, `glyim_println_int`, `glyim_assert_fail`) are added to the module as IR *before* JIT compilation. The JIT resolves `write` and `abort` as external symbols, which the OS dynamic linker provides.

**`glyim build`** continues to use the AOT path (write object file → invoke `cc` linker). Only `glyim run` switches to JIT.

### 3.7 `ariadne` Diagnostic Rendering

```rust
// In glyim-diag, new module: render.rs
use ariadne::{Report, ReportKind, Label, Source};

pub fn render_diagnostics(
    source: &str,
    file_path: &str,
    diagnostics: &[Diagnostic],
) -> String {
    let mut out = Vec::new();
    for diag in diagnostics {
        let kind = match diag.severity {
            Severity::Error => ReportKind::Error,
            Severity::Warning => ReportKind::Warning,
            Severity::Note => ReportKind::Advice,
        };
        let span_start = diag.span.map_or(0, |s| s.start);
        let span_end = diag.span.map_or(0, |s| s.end);
        Report::build(kind, file_path, span_start)
            .with_message(&diag.message)
            .with_label(Label::new((file_path, span_start..span_end)))
            .finish()
            .write((file_path, Source::from(source)), &mut out)
            .unwrap();
    }
    String::from_utf8(out).unwrap()
}
```

## 4. Alternatives Considered

### 4.1 String Representation: Null-Terminated vs Fat Pointer
- **Alternative A:** C-style `"hello"` → `i8*` null-terminated. Simpler codegen, compatible with C `printf`.
- **Alternative B:** Rust-style `{ i8*, i64 }` fat pointer. No null-termination needed, length is O(1), matches future `&str` semantics.
- **Decision: B (Fat Pointer).** Aligns with the borrowed string design goal. Slightly more codegen complexity (extractvalue instructions) but avoids null-termination bugs and sets up v0.3.0 for slicing/substring support.
- *Traceability: ASR-012.*

### 4.2 `println`: FFI vs Compiler Built-In
- **Alternative A:** Define `println` via `@rust("libc::printf")` FFI. Validates the FFI architecture.
- **Alternative B:** Emit `println` as hardcoded LLVM IR. No FFI dependency.
- **Decision: B (Built-In).** The FFI architecture (`@rust()`) is complex (opaque types, wrapper generation). Hardcoding `println` is 10x faster to implement and still achieves the user-facing goal. The `write` syscall is declared as an external LLVM function, which is the minimal possible FFI.
- *Traceability: ASR-013.*

### 4.3 Execution: Linker vs JIT
- **Alternative A:** Keep `cc` linker for `glyim run`. Works, proven.
- **Alternative B:** Switch to MCJIT/ORC JIT. Removes `cc` dependency, enables REPL, faster for dev.
- **Decision: B (JIT).** Removing the `cc` requirement is a significant UX improvement for first-time users. JIT also enables the v0.3.0 REPL with minimal additional work. The AOT path (`glyim build`) remains available for production builds.
- *Traceability: ASR-018.*

### 4.4 Rowan Rewrite: Now vs Later
- **Alternative A:** Defer Rowan to v0.3.0 when LSP work begins.
- **Alternative B:** Rewrite parser now, before more syntax is added.
- **Decision: B (Now).** Every new syntax feature (let, if, strings, use) makes a Rowan rewrite harder. Doing it now means all new features are built on the correct foundation. The AST types don't change — only the parser internals change.
- *Traceability: ASR-017.*

### 4.5 Error Recovery: Panic vs Sync-Point Recovery
- **Alternative A:** On error, `?` propagate immediately (current behavior). Simple but one-error-per-compile.
- **Alternative B:** Emit error, insert error node in tree, skip to sync point, continue.
- **Decision: B (Sync-Point Recovery).** The "fix one error, recompile, see the next" loop is the #1 complaint about new-language compilers. Sync-point recovery is well-understood and matches what rustc and TypeScript do.
- *Traceability: ASR-016.*

## 5. Cross-Cutting Concerns

### 5.1 Runtime Functions
The compiler emits three runtime shim functions as LLVM IR:

| Function | Signature | Behavior |
|----------|-----------|----------|
| `glyim_println_str` | `void ({ i8*, i64 })` | `write(1, ptr, len)` then `write(1, "\n", 1)` |
| `glyim_println_int` | `void (i64)` | Convert to decimal on stack, `write(1, buf, len)`, `write(1, "\n", 1)` |
| `glyim_assert_fail` | `void (i8*, i64)` | `write(2, "assertion failed: ", 18)` then `write(2, msg, len)` then `abort()` |

These are declared as external: `declare i64 @write(i32, i8*, i64)` and `declare void @abort()`.

### 5.2 Platform Considerations
- `write` syscall and `abort` are POSIX. Windows uses `WriteFile` and `ExitProcess`. For v0.2.0, we target **Linux and macOS only**. Windows support is deferred.
- Fat pointer size: `{ i8*, i64 }` assumes 64-bit. 32-bit support is deferred.
- JIT: Inkwell's ORC JIT works on Linux and macOS. Windows support is deferred.

### 5.3 Backward Compatibility
v0.2.0 is **not** backward compatible with v0.1.0 syntax:
- `main = () => 42` still works (lambda binding)
- But `let main = () => 42` is the new preferred form
- The old `name = expr` binding syntax continues to work (no breaking change)

## 6. Architecture Decision Records

### ADR-010: Borrowed String Fat Pointers
- **Context:** Need a string representation for `println` that aligns with future `&str` semantics.
- **Decision:** String literals compile to a global constant pair `{ i8* data, i64 len }`. No heap allocation, no null terminator.
- **Consequences:** + Matches Rust mental model. + Length is O(1). + No buffer overflow risk. – Slightly more complex codegen than null-terminated. – No concatenation until heap strings exist.

### ADR-011: Compiler Built-In Intrinsics
- **Context:** Need `println` and `assert` without implementing the full `@rust()` FFI.
- **Decision:** The compiler recognizes `println` and `assert` as built-in names. It emits calls to compiler-provided runtime shim functions (written as LLVM IR). These shims use `write`/`abort` as external POSIX functions.
- **Consequences:** + Zero FFI complexity. + Full control over behavior. – Not extensible by users. – Runtime shims must be maintained in lockstep with the compiler.

### ADR-012: JIT Execution via ORC
- **Context:** `glyim run` requires `cc`/`gcc`, creating a friction point.
- **Decision:** Use Inkwell's ORC JIT for `glyim run`. Runtime functions are added to the module before JIT compilation. AOT path (`glyim build`) remains separate.
- **Consequences:** + No external dependency for `glyim run`. + Enables future REPL. + Faster dev cycle. – JIT has higher memory usage. – Slightly different optimization behavior vs AOT.

### ADR-013: Statement/Expression Distinction in HIR
- **Context:** `let` and `x = expr` are statements, not expressions. v0.1.0's HIR had only expressions.
- **Decision:** Introduce `HirStmt` enum (`Let`, `Assign`, `Expr`). Blocks become `Vec<HirStmt>`. The last `Expr` statement's value is the block's value.
- **Consequences:** + Matches Rust semantics. + Enables mutable variables. – Changes block codegen (must track "last expr" for phi nodes).

### ADR-014: Rowan-First Parser
- **Context:** v0.1.0 parser skips Rowan, parsing directly to an AST enum. This blocks LSP and loses formatting info.
- **Decision:** Rewrite parser to build Rowan GreenNodes. Add a separate AST lowering pass. Error nodes are inserted into the tree for recovery.
- **Consequences:** + Lossless tree available for LSP/formatter. + Error recovery is natural (error nodes in tree). – Significant parser refactor. – Two-phase parsing (CST → AST) adds conceptual overhead.

### ADR-015: UI Test Framework with Snapshots
- **Context:** Need to test every diagnostic message. Manual testing doesn't scale.
- **Decision:** `tests/ui/` directory with `.g` + `.g.stderr` pairs. Use `insta` for initial snapshot generation, then lock as text files. Test harness compiles source, captures stderr, compares.
- **Consequences:** + Every error message is a tested contract. + Easy to review diagnostic changes in PRs. – Test file count grows quickly. – Snapshot updates require `cargo insta review` discipline.

### ADR-016: Sync-Point Error Recovery
- **Context:** Users need to see all errors in one compile pass.
- **Decision:** Define sync points (`fn`, `let`, `}`, EOF). On error: emit diagnostic, insert error node, skip tokens to next sync point, resume.
- **Consequences:** + Multiple errors per compile. + Error tree structure is preserved. – May produce cascading errors (acceptable — better than one-error-per-compile). – Sync point list must be maintained as syntax grows.

## 8. Behavioral Specification (BDD)

### 8.1 Let Bindings

```gherkin
Feature: Let Bindings

  Scenario: Immutable let binding
    Given the source "let x = 42"
    When compiled and run
    Then the program compiles without errors

  Scenario: Mutable let binding
    Given the source "let mut x = 10\nx = x + 1"
    When compiled and run
    Then the program compiles without errors

  Scenario: Reassigning immutable binding is a compile error
    Given the source "let x = 5\nx = 10"
    When compiled
    Then the compiler emits an error: "cannot assign to immutable binding 'x'"
    And the error points to the second occurrence of 'x'

  Scenario: Let binding in a block is scoped
    Given the source "let x = 1\n{ let x = 2 }\nprintln(x)"
    When compiled and run
    Then the output is "1"

  Scenario: Old-style binding still works
    Given the source "main = () => 42"
    When compiled and run
    Then the exit code is 42
```

    And the source 'main = () => { println("jit") }'
    When "glyim run" is executed

    And the exit code is 0

  Scenario: Run with println still works via JIT
    Given the source:

## 11. The 60-Second Test (Acceptance Criteria)

v0.2.0 is complete when a user can do all of this, with zero errors, in under 60 seconds:

```bash
$ glyim init hello
Created hello/
  hello/glyim.toml
  hello/src/main.g

$ cd hello
$ cat > src/main.g << 'EOF'
let name = "Glyim"
let version = 2

if version > 1 {
  println("Hello from " + name)   # Note: concatenation NOT in v0.2 — this will error
} else {
  println("old version")
}

let mut count = 0
count = count + 1
assert(count == 1, "count should be 1")
EOF

$ glyim check
# (should show error about string concatenation not being supported, with nice span)

$ cat > src/main.g << 'EOF'
let name = "Glyim"
let version = 2

if version > 1 {
  println("Hello from Glyim!")
} else {
  println("old version")
}

let mut count = 0
count = count + 1
assert(count == 1, "count should be 1")
println(count)
EOF

$ glyim run
Hello from Glyim!
1

$ glyim ir
# (prints valid LLVM IR with alloca, br, phi, string constants)
```
