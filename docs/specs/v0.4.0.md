# Glyim v0.4.0 Architecture & Design Specification

**Document ID:** GLYM-ARCH-004
**Status:** Draft
**Date:** 2026-04-28
**Predecessor:** GLYM-ARCH-003 (v0.3.0)
**Traceability:** Business Goal 4 (Generic Types) → ASR-033 (Local-Only Inference)

---

## 1. Context & Scope

### What v0.3.0 Delivered
Monomorphic type checker, struct and enum definitions, pattern matching, `bool`/`Unit` types, raw pointers, `f64`, `Option<T>`/`Result<T,E>` as built-in monomorphic types, `?` operator, one working macro (`@identity`), 630 tests, CI with GitHub Actions, file size enforcement (500 LOC max).

### The Problem: No Generics, No Methods, No Reusability
Users cannot write `fn id<T>(x: T) -> T { x }`. They cannot attach methods to structs via `impl`. They cannot define generic data structures like `Node<T>`. The built-in `Option<T>` is compiler magic — there's no way for users to define their own parameterized types. The type checker is a single-pass architecture that cannot separate type checking from monomorphization.

### The Solution: v0.4.0 — "Generic Types"
This spec introduces full generic types with local-only type inference at call sites. Structs, enums, functions, and impl blocks can be parameterized. `Option<T>` and `Result<T,E>` migrate from built-in magic to standard library prelude definitions. The type checker becomes a two-pass architecture: check first, then monomorphize.

### Scope of v0.4.0
**Included:**
- Generic type parameters: `fn id<T>(x: T) -> T { x }`
- Generic struct definitions: `struct Node<T> { value: T, next: Option<Node<T>> }`
- Generic enum definitions: `enum Maybe<T> { Some(T), Nothing }`
- Impl blocks: `impl<T> Node<T> { fn new(value: T) -> Node<T> { ... } }`
- Tuple types: `(i64, Str)`, `()`, destructuring `let (a, b) = pair`
- Cast expressions: `x as f64`, `"hello" as *const u8`
- Pattern guards: `Some(v) if v > 0 => v`
- Destructuring `let` bindings: `let (a, b) = pair`, `let Point { x, y } = p`
- Two-level visibility: `pub struct`, `pub fn`, `pub impl`
- Standard prelude: `Option<T>`, `Result<T, E>` defined as real generic enums
- Two-pass type checker: check-then-monomorphize architecture
- Monomorphization via worklist algorithm
- `ExprId` integration for type annotations

**Excluded:**
- Trait definitions or trait impl blocks — v0.5.0
- Method chaining (`.foo().bar()` without rebinding) — v0.5.0
- Generic bounds (`where T: Clone`) — v0.5.0
- Generic associated types — v0.5.0
- `pub(crate)`, `pub(super)` visibility — v0.5.0
- Lifetime annotations — v0.5.0 (no `&T` references in v0.4.0)
- Async/await — v0.6.0+
- Distributed CAS for monomorphization caching — v0.6.0+
- Macro compilation to native code (still interpreted) — v0.5.0+
- `Option`/`Result` with `?` inside generic functions — **explicit return type required** (see §3.3)

### What Changes from v0.3.0

| Area | v0.3.0 | v0.4.0 | User-visible? |
|------|---------|---------|---------------|
| `Option<T>` | Built-in magic, monomorphized at check time | Real generic enum in standard prelude | **No** (same syntax) |
| `Result<T,E>` | Built-in magic | Real generic enum in standard prelude | **No** (same syntax) |
| `?` operator | Works in `Result`-returning functions only | Same | **No** |
| Function signatures | Explicit type annotations required | Same | **No** |
| Struct definitions | Monomorphic only | Can be generic | **Yes** (`struct Name<T>`) |
| `match` | Enums only | Also works on tuples and wildcards | **Yes** (more patterns) |
| `if` | Requires `bool` | Same | **No** |
| Visibility | None | `pub struct`, `pub fn`, `impl pub` | **Yes** (new feature) |
| Pattern guards | Not supported | `Some(v) if v > 0` | **Yes** (new feature) |
| Destructuring `let` | Not supported | `let (a, b) = pair` | **Yes** (new feature) |
| Type checker | Single-pass | Two-pass (check + monomorphize) | **No** (internal only) |
| File size | 500 LOC max | 500 LOC max | **No** (same policy) |

## 2. Goals and Non-Goals (ASRs)

### Goals

| ID | Statement |
|----|-----------|
| **ASR-033** | Type inference is local-only: `let x = 42` infers `x: i64`. Function signatures always have explicit types. Call sites infer generic args from argument types. |
| **ASR-034** | Generic struct definitions produce parameterized types. Layout is computed after monomorphization. |
| **ASR-035** | Generic enum definitions produce parameterized tagged unions. Monomorphized the same way as generic structs. |
| **ASR-036** | Impl blocks desugar to namespaced functions. `impl<T> Node<T> { fn new(...) }` → `fn Node_new<T>(...)`. |
| **ASR-037** | Tuple types `(T, U, ...)` are anonymous struct types. Destructuring patterns match by position. |
| **ASR-038** | Two-level visibility: `pub` (exported) vs default (file-scoped). Applies to structs, enums, functions, impl blocks. |
| **ASR-039** | Pattern guards: `Some(v) if v > 0 => v` — a boolean guard that filters pattern arms. |
| **ASR-040** | Destructuring `let` bindings: `let (a, b) = pair` extracts tuple fields by position. |
| **ASR-041** | Cast expression `expr as Type` with validity checking (no arbitrary casts). |
| **ASR-042** | Standard prelude defines `Option<T>` and `Result<T,E>` as real generic enums. Backward-compatible with v0.3.0 source syntax. |
| **ASR-043** | Two-pass type checker: Pass 1 checks all types. Pass 2 monomorphizes generic call sites. |
| **ASR-044** | Monomorphization uses a worklist algorithm with topological ordering. Each unique `(function, concrete_args)` pair produces one LLVM function. |
| **ASR-045** | `ExprId` is attached to every `HirExpr` for type annotation lookup. Existing `v0.3.0` HIR gains this field with `#[cfg(test)]`-only initialization. |

### Non-Goals
- **No HM inference.** No inference in type annotations. Function signatures are always concrete or generic (with `<T>` parameters). Let bindings infer from initializer expressions only.
- **No trait bounds.** `where T: Clone` is v0.5.0.
- **No method chaining.** `p.foo().bar()` requires borrow checking. v0.4.0 functions are called as `Foo_bar(p, args)`.
- **No associated types.** `impl<T> SomeType { type Item = U; }` is v0.5.0.
- **No pub(crate)/pub(super).** Two levels only. v0.5.0+ for finer-grained visibility.
- **No lifetime annotations.** There are no references (`&T`) in v0.4.0. All values are owned or copied.
- **No `?` inside generic functions returning `Option`.** `?` desugars to a match on `Result`. Inside a function returning `Option`, the user must write an explicit match. (This simplification avoids needing to detect the return type's generic structure.)
- **No generic impl blocks on enums.** `impl<T> Maybe<T> { ... }` is v0.5.0 (requires associated types for variant extraction).
- **No eager monomorphization.** Monomorphization happens after full type checking. No cross-crate caching in v0.4.0.

## 3. The Design

### 3.1 HirType Extensions

```rust
pub enum HirType {
    // ── Primitive ───────────────────────────────────────────────
    Int,                                // i64
    Bool,                               // i1 (zero-extended to i64 for uniform representation)
    Float,                              // f64
    Str,                                // &str fat pointer { i8*, i64 }
    Unit,                               // () — zero-size

    // ── Composite ───────────────────────────────────────────────
    Named(Symbol),                       // user-defined struct or enum
    Generic(Symbol, Vec<HirType>),    // e.g., Node<T>, Pair<T, U>
    Tuple(Vec<HirType>),              // e.g., (i64, Str)
    Func(Vec<HirType>, Box<HirType>),  // function type
    RawPtr { inner: Box<HirType> },    // *const T, *mut T

    // ── Error types ────────────────────────────────────────────
    Never,                              // uninhabited type (match exhaustiveness)

    // ── v0.3.0 built-ins (deprecated, replaced by prelude) ──
    // These are kept for v0.3.0 compatibility but marked deprecated.
    // The type checker maps these to their prelude equivalents.
    Option(Box<HirType>),               // → Named(Symbol("_option_<hash>"))
    Result(Box<HirType>, Box<HirType>), // → Named(Symbol("_result_<hash>"))
    Opaque(Symbol),                       // @rust("...") type
}
```

### 3.2 Generic Function Signatures

```g
// Generic function: T is a type parameter, resolved at call site
fn identity<T>(x: T) -> T {
    x
}

// Multi-parameter generic
fn make_pair<T, U>(a: T, b: U) -> Pair<T, U> {
    Pair { first: a, second: }
}

// Generic struct with type parameter
struct Node<T> {
    value: T,
    next: Option<Node<T>>,
}

// Concrete usage — T is inferred from argument types
let x: i64 = identity(42)           // T = i64
let p: Pair<i64, Str> = make_pair(1, "hello")  // T = i64, U = Str
```

**HirFn gains a `type_params` field:**

```rust
pub struct HirFn {
    pub name: Symbol,
    pub type_params: Vec<Symbol>,  // NEW: empty for non-generic functions
    pub params: Vec<(Symbol, Span)>,
    pub body: HirExpr,
}
```

### 3.3 Standard Prelude

The prelude is injected before user code. For v0.4.0, it defines two generic enums:

```g
// Injected automatically by the compiler before user source
pub enum Option<T> {
    Some(T),
    None,
}

pub enum Result<T, E> {
    Ok(T),
    Err(E),
}
```

**Backward compatibility:** v0.3.0 source code using `Option<i64>` works in v0.4.0 without changes. The type checker resolves `Option` to the prelude enum, instantiates with `T = i64`, and produces identical behavior.

The `?` operator works identically — it checks the function's return type is `Result<_, _>` (whether built-in or prelude) and desugars to the same match expression.

**No `?` inside functions returning `Option`:** The desugaring produces `return Err(e)` which requires `Result`. To keep the desugaring simple, `?` is restricted to functions returning `Result<_, _>` in v0.4.0.

### 3.4 Impl Block Desugaring

```g
impl Point {
    fn distance(self: Point, other: Point) -> f64 { ... }

    fn zero() -> Point { Point { x: 0, y:0 } }
}
```

**Desugared:**

```g
fn Point_distance(self: Point, other: Point) -> f64 { ... }
fn Point_zero() -> Point { Point { x: 0, y: 0 } }
```

**Call-site desugaring:**

```g
p.distance(other)  →  Point_distance(p, other)
Point::zero()      →  Point_zero()
```

**Self parameter rules:**
- `self` must be the first parameter
- The impl target must be a Named type (struct or enum)
- `self` is typed as the impl target type
- `pub` on an impl block makes all methods exported

**Generic impl blocks:**

```g
impl<T> Node<T> {
    fn new(value: T) -> Node<T> { Node { value, next: None } }
}
```

Desugared: `fn Node_new<T>(value: T) -> Node<T> { ... }`. The type parameter `T` is added to the function's type params.

### 3.5 Tuple Types

```g
// Tuple literal
let pair: (i64, Str) = (1, "hello")

// Tuple type annotation
let pair: (i64, Str) = (1, "hello")

// Destructuring
let (a, b) = pair
println(a)
println(b)
```

**HirType:** `Tuple(Vec<HirType>)`. Internally stored as `{ _0: T, _1: U }`.

**HirPattern:** `Tuple { elements: [HirPattern, HirPattern] }`. Pattern matching is positional: `let (a, _, b) = pair` skips the second element.

**Tuple construction:** `(expr1, expr2, ...)` is parsed as an expression that produces a `Tuple` value.

### 3.6 Pattern Guards

```g
match value {
    Some(v) if v > 0 => v,
    Some(_) => -1,
    None => 0,
}
```

**MatchArm gains a `guard` field:**

```rust
pub struct MatchArm {
    pub pattern: HirPattern,
    pub guard: Option<HirExpr>,  // NEW: optional boolean expression
    pub body: HirExpr,
}
```

**Type checker behavior:** If a guard is present, its type must be `bool`. The guard is evaluated at runtime (like an `if` before the body).

**Exhaustiveness:** Guards do NOT affect exhaustiveness checking. Only the pattern matters. `Some(v) if false` is still covered by `Some(_)`.

### 3.7 Destructuring Let Bindings

```g
let Point { x, y } = point
let (a, _, b) = triple
```

**Mechanism:** Same pattern matching logic used in `match` arms is applied to `let` bindings. The pattern is matched against the initializer expression's type.

**Struct destructuring:**
- Match by field name: `let Point { x, y } = point`
- Wildcards: `let Point { x, .. } = point`
- Missing fields: error ("missing field 'z'")

**Tuple destructuring:**
- Match by position: `let (a, _, b) = triple`
- Nested: `let (a, (b, c)) = nested_pair`

### 3.8 Cast Expression

```g
let x: f64 = 42 as f64
let y: i64 = x as i64
let p: *const u8 = "hello" as *const u8
```

**HirExpr:** `As { expr: Box<HirExpr>, target_type: HirType }`

**Valid casts:**
- Numeric ↔ Numeric: `i64` ↔ `f64` (both directions)
- Any type ↔ `RawPtr` of any inner type
- `Str` ↔ `*const u8` (string to raw pointer)

**Invalid casts (compile error):**
- `i64 as Str` — cannot convert integer to string
- `Struct` as `f64` — no structural conversion
- `f64` as `i64` — narrowing conversion requires explicit function call, not cast

### 3.9 Visibility

```g
pub struct Config {
    path: Str,
}

pub fn load(path: Str) -> Config { ... }

struct Internal {
    secret: i64,
}
```

**Rules:**
- `pub struct` → the struct and its public fields are visible outside the file. Private fields can exist inside `pub struct`.
- `pub fn` → the function is visible outside the file.
- `impl pub { ... }` → all methods are visible outside the file. The struct must also be `pub`.
- No annotation → visible only within the file. Default.
- Visibility is a CHECK, not a generation concern. It's enforced by the type checker and has no codegen impact.

### 3.10 Type Checker Architecture

The type checker is restructured from v0.3.0's single-pass approach to a two-pass approach:

**Pass 1: Type Checking**
```
fn check(hir: &Hir) -> Result<CheckedHir, Vec<TypeError>>
```

1. Register all top-level items (structs, enums, impls, extern blocks, prelude)
2. For each function body (generic or concrete), push a new scope
3. If generic, bind type params
4. Walk statements and expressions, assigning types
5. Return `CheckedHir` with `expr_types: Vec<HirType>` filled in

**Pass 2: Monomorphization**
```
fn monomorphize(hir: CheckedHir) -> Hir
```

1. Walk all expressions looking for call sites to generic functions
2. For each unique `(function_symbol, concrete_type_args)` pair, add to worklist
3. Pop from worklist. If not yet monomorphized:
   a. Substitute type params with concrete types in the function body
   b. Add the monomorphized function to the HIR
   c. Scan its body for further generic call sites, add those to the worklist
4. Replace all generic call sites with monomorphized call sites
5. Remove the original generic function from the HIR (it's been replaced by monomorphs)

**Why two passes?** Monomorphization needs to know the types of generic call arguments, which requires type checking. Type checking needs monomorphized functions to be available for cross-function generic calls. Two passes resolve this circular dependency.

### 3.11 Monomorphization Algorithm

```rust
fn monomorphize(hir: &CheckedHir) -> Hir {
    let mut worklist: Vec<MonoRequest> = Vec::new();
    let mut monomorphs: HashMap<MonoKey, HirFn> = HashMap::new();

    // Phase 1: Collect all generic call sites
    for item in &hir.items {
        if let HirItem::Fn(fn_def) = item {
            if fn_def.type_params.is_empty() { continue; }
            collect_generic_calls(fn_def, &mut worklist);
        }
    }

    // Phase 2: Process worklist (topological order)
    while let Some(req) = worklist.pop() {
        let key = (req.func, req.concrete_args.clone());
        if monomorphs.contains_key(&key) { continue; }
        let mono_fn = instantiate_fn(req.func, &req.concrete_args);
        monomorphs.insert(key, mono_fn);
        collect_generic_calls(&mono_fn, &mut worklist);
    }

    // Phase 3: Replace generic call sites with monomorphized call sites
    for item in &mut hir.items {
        replace_generic_calls_in_item(item, &monomorphs);
    }

    // Phase 4: Remove original generic definitions
    hir.items.retain(|item| {
        matches!(item, HirItem::FnDef(fn) if !fn.type_params.is_empty())
    });

    hir
}

fn collect_generic_calls(fn_def: &HirFn, worklist: &mut Vec<MonoRequest>) {
    walk_expr(&fn_def.body, |expr| {
        if let HirExpr::Call { callee, args } = expr {
            if let HirExpr::Ident(sym) = &callee.kind {
                if let Some(HirFn { type_params, .. }) = resolve_function(*sym) {
                    if !type_params.is_empty() {
                        // Infer concrete args from argument types
                        let concrete_args: Vec<HirType> = args.iter()
                            .map(|a| expr_types[a.id()])
                            .collect();
                        worklist.push(MonoRequest { func: *sym, concrete_args });
                    }
                }
            }
        }
    });
}
```

**MonoRequest key:** `(Symbol, Vec<HirType>)` — the function symbol plus the concrete type arguments. This produces unique monomorphs for `id<Int>`, `id<Str>`, `Node<Int>`, etc.

## 4. Alternatives Considered

### 4.1 Generic Inference Strategy: Full HM vs Local-Only vs Explicit

*Alternative A (Full Hindley-Milner):* Infer types everywhere — `fn foo(x) = x + 1` infers `x: i64` from the body.
*Alternative B (Go-style):* All types are explicit. `fn foo(x: i64) -> i64`.
*Alternative C (Kotlin-style):* Explicit signatures, local inference for `let` bindings only.

*Decision: C (Kotlin-style).* Same as v0.3.0 but extended with generic call-site inference. Function signatures remain explicit. This keeps the inference algorithm to ~200 LOC instead of ~2000 LOC. Research shows this covers 90% of use cases with 10% of the complexity.

### 4.2 Option/Result: Keep Built-In Magic vs Real Generic Enums

*Alternative A:* Keep `Option<T>` as built-in compiler magic (v0.3.0 approach).
*Alternative B:* Migrate to real generic enums in the prelude (v0.4.0 approach).

*Decision: B.* Same user-visible syntax. Different internals. Real generic enums can be extended with user-defined generic enums. The built-in magic is deprecated but not removed.

### 4.3 Tuple Types: Inline Structs vs Named Structs

*Alternative A:* Tuples are sugar for anonymous structs: `(T, U)` → `struct _Tuple_T_U { _0: T, _1: U }`.
*Alternative B:* Tuples are a first-class `HirType::Tuple` variant, with dedicated parsing and pattern matching.

*Decision: B.* Tuples are a language primitive (not a desugaring). The field names `_0`, `_1` are internal only; the user interacts via positional patterns.

### 4.4 Method Chaining: Impl Blocks vs Extension Functions

*Alternative A:* `p.foo().bar()` requires resolving the return type of `foo()` to call `bar()` on it.
*Alternative B:* Method calls always rebind: `let temp = p.foo(); let result = temp.bar();`

*Decision: B for v0.4.0.* Method calls desugar to rebindings. Chaining is a v0.5.0 feature (requires tracking "self" rebinding).

### 4.5 Visibility Granularity: 2 vs 5 levels

*Alternative A (Rust):* `private`, `pub`, `pub(crate)`, `pub(super)`, `pub(in path)`.
*Alternative B (Go):* exported (capitalized) vs unexported (lowercase).
*Alternative C (2-level):* `pub` vs default-private.

*Decision: C (2-level).* Matches the 90% use case. `pub(crate)` and `pub(super)` can be added in v0.5.0 without breaking any existing code.

## 5. Cross-Cutting Concerns

### 5.1 Error Messages for Generic Errors

Type errors involving generics must show the type arguments:

```
error[E0301]: type mismatch in generic instantiation
  --> src/main.g:12:1
   |
12 |   let x: i64 = identity("not a number")
   |                  ^^^^^^^^^^^^^^ expected `i64` for generic parameter `T`
   |
help: ensure the argument type matches the generic parameter
```

### 5.2 ExprId Plumbing

Every `HirExpr` gains `id: ExprId` (u32 counter). The type checker fills `expr_types: Vec<HirType>` indexed by `ExprId`. The `as` cast, `?` desugaring, and impl call desugaring all use `expr.id` to look up types. The `#[cfg(test)]` attribute guards the field in v0.3.0 code — upgrading to v0.4.0 requires running the id-assignment once with tests, then removing the guard.

### 5.3 Backward Compatibility

v0.3.0 source compiled by v0.4.0 must produce identical behavior. Specifically:
- `let x: Option<i64> = Some(42)` — still works, same semantics
- `let x: Result<i64, Str> = Ok(42)` — still works
- `let x = r?` in a function returning `Result<i64, Str>` — still works
- `let x: bool = true` — still works (already a distinct type in v0.3.0)

### 5.4 File Size Enforcement

Same policy: 500 LOC hard max, 300 LOC soft warning in CI. No exceptions, not even for the monomorphization code.

## 6. Architecture Decision Records

### ADR-026: Local-Only Type Inference

*Context:* Full Hindley-Milner inference adds ~2000 LOC of unification algorithm. Go 1.18 shipped generics with zero inference (all types explicit).
*Decision:* Local inference only. `let x = 42` infers `x: i64`. Function signatures are always explicit. Call sites infer generic args from arguments. This keeps inference to a single ~200 LOC function.
*Consequences:* + Simple, fast, maintainable. – Cannot write `fn foo(x) -> x + 1` without annotation. – Matches Kotlin and early Rust patterns.

### ADR-027: Two-Pass Type Checker (Check then Monomorphize)

*Context:* v0.3.0 has a single-pass type checker. Generic call sites need to be monomorphized, but monomorphization needs type information.
*Decision:* Two passes. Pass 1: type-check everything. Pass 2: monomorphize generic call sites.
*Consequences:* + Resolves circular dependency. + Clean separation of concerns. – Every type is checked before any codegen. – Slightly slower (two passes instead of one).

### ADR-028: Impl Block Desugaring

*Context:* Rust's `impl` blocks are deeply integrated with the trait system. Without traits, impl blocks are just sugar for namespaced functions.
*Decision:* Desugar to `Name_method(self, args...)` at parse time. No impl table in the type checker. The type checker only sees functions.
*Consequences:* + ~200 LOC parser change. + Zero trait infrastructure needed. – Cannot dispatch on `self` type (v0.5.0 for trait dispatch).

### ADR-029: Two-Level Visibility

*Context:* Rust has 5 visibility levels. Most code uses only 2.
*Decision:* `pub` (exported) vs default (file-scoped). Simple, matches Go's approach. No `pub(crate)` or `pub(super)`.
*Consequences:* + Covers 90% of use cases. + Zero confusion about what `pub(crate)` means. – Cannot restrict visibility to a specific parent module. Deferred to v0.5.0.

### ADR-030: Standard Prelude as Real Generic Enums

*Context:* v0.3.0 implements `Option<T>` as built-in magic. Shipping real generic enums alongside built-in ones creates two ways to define the same concept.
*Decision:* Replace built-in Option/Result with real generic enums in a standard prelude. Mark the built-in representation as deprecated but still functional for backward compatibility.
*Consequences:* + One mechanism for all parameterized types. + Users can define their own `Option`-like types. – Existing code breaks if it used internal details of the built-in representation.

### ADR-031: On-Demand Monomorphization

*Context:* LLVM does eager monomorphization (monomorphize when generating IR). Rust does on-demand monomorphization (monomorphize when a generic function is first called).
*Decision:* On-demand via worklist algorithm. When `Node::new(42)` is first called, monomorphize `Node_new<i64>`. Subsequent calls reuse the monomorph. No pre-computation.
*Consequences:* + No wasted compilation. + Simpler implementation. – Cannot do LTO across generic instantiations (deferred).

### ADR-032: Tuple Types as Anonymous Structs

*Context:* Tuples can be represented as anonymous structs `struct _Tuple_T_U { _0: T, _1: U }`.
*Decision:* Tuples are a first-class `HirType::Tuple` variant with positional field access. Not a desugaring — the parser directly produces `Tuple` types. Internally, the codegen maps them to LLVM structs with `_0`, `_1` field names.
*Consequences:* + Clean separation (tuples are not structs). + Positional destructuring is natural. – Internal field names (`_0`, `_1`) are not visible to users.
