# Revised v0.6.0 Architecture & Design Specification

## Executive Analysis

After thorough review of the current codebase (v0.5.0, commit `elcoosp-glyim-8a5edab282632443`), the original v0.6.0 specification is **unrealistically ambitious** given the current macro system architecture. The existing macro pipeline is entirely string‑based and identity‑only; there is no HIR construction API, no `quote!` engine, and no way for macros to inspect or generate typed AST nodes. Achieving the full set of features (quote/splice, built‑in derives, stepper, etc.) within a single release cycle would require rewriting the entire macro system, parser, HIR, and codegen, a task more appropriate for a major version.

This revised specification defers the large‑scale rewrite to **v0.6.0** (or later) and instead focuses on **foundational improvements** to the string‑based macro system that can be delivered incrementally, build trust, and enable a smooth transition toward the full vision later.

## Revised Goals for v0.6.0

| Goal | ID | Rationale |
|------|----|-----------|
| **String‑based macro execution with Wasm** (improved) | ASR-031r | Strengthen the current `MacroExecutor` to support multiple macros, proper error handling, and deterministic caching. |
| **Attribute argument parsing** | ASR-032r | Parse structured key‑value attribute arguments (`@serde(rename_all = "camelCase")`) and make them available to macros. |
| **Extended `MacroContext` (read‑only reflection)** | ASR-033r | Expose `get_variants`, `get_methods`, `caller_location`, and basic module info from the type checker to string‑based macros. |
| **Macro expansion inspection tool** | ASR-034r | Add a `glyim macro-inspect` command that shows the string‑level macro expansions step‑by‑step. |
| **Basic caching improvement** | ASR-035r | Use the existing ContentStore to cache macro expansion results per input hash; no dependency tracking yet. |
| **Parser support for `quote!` and `${}` (syntax only)** | ASR-036r | Add token/parse support for `quote! { ... }` and `${expr}` so that source files can be parsed without error, but the expansion is deferred. |

### Deferred to v0.6.0

All items that require HIR construction, typed macro execution, or built‑in derives are postponed:

- `quote!` engine and `${}` splicing as HIR builders
- Built‑in derive macros (`Debug`, `Clone`, `Serialize`, …)
- Macro stepper with AST diffs and LSP integration
- Macro composability protocol (`requires`, conflict detection)
- Incremental expansion with dependency tracking (per‑invocation)
- Full `MacroContext` reflection with sandboxed FS

## Revised Design: Incremental String‑Based Enhancements

### 1. Attribute Arguments

**Parser change** (doable today): extend the existing attribute parser to support `key = value` and nested arguments.

```ebnf
attribute_arg = ident ["=" (string | ident | integer)] ["," attribute_arg]
```

The parsed `AttributeArg` list is attached to the `Item::FnDef`, `Item::StructDef`, etc. and passed to the macro call site via the `MacroContext`–no HIR changes needed; just string lookups.

### 2. Macro Executor and Registry Improvements

- **Multi‑macro support**: the `MacroRegistry` already exists; we will add the ability to load user‑provided `.wasm` macros from the project’s `glyim.lock`.
- **Error handling**: catch Wasm traps and return structured errors that include the macro name and a simple span of the invocation.
- **Caching**: use the existing `MacroExpansionCache` with input hash; on cache hit, skip execution.

All of this is a direct evolution of the current `MacroExecutor`.

### 3. String‑Based `MacroContext` Extension

Because macros run in Wasm, we cannot expose rich HIR objects. Instead, we pre‑compute the necessary reflection data **at compile time** and serialise it into the Wasm linear memory as a structured byte buffer.

The `MacroContext` trait gains:

```rust
fn serialize_type_info(&self, type_name: &str) -> Vec<u8>;
```

The host (non‑Wasm) side will, before invoking the macro, look up the target type (struct or enum) using the type checker’s data and pack its fields, variants, and methods into a binary format that the Wasm module can parse.

The Wasm macro can then decode this buffer and use it to generate output strings.

This is much simpler than a full HIR API and sufficient for many use‑cases (e.g., custom `Debug`-like formatters).

### 4. Parser Syntax for `quote!` and `${}`

We add **syntactic recognition** of `quote!` macros and `${...}` splices without implementing the HIR‐building semantics.

```glyim
// Parsed successfully, but not yet expanded
macro my_macro() {
    quote! {
        fn ${name}() { ... }
    }
}
```

The tokeniser and parser are extended so that these constructs do not cause parse errors. The actual expansion will be deferred to a future version.

### 5. `glyim macro-inspect` (String Stepper)

The existing `cmd_macro_inspect` already performs string‑level expansion and shows before/after. We enhance it to support **multiple macros**, **multiple steps**, and an optional `--interactive` mode where the user can step through each expansion one at a time.

This provides immediate debugging value without requiring HIR diffs.

## Revised Architecture (Container View)

No new crates. Changes are limited to:

| Crate | Change |
|-------|--------|
| `glyim-macro-core` | Enhanced `MacroExecutor` with error codes, `MacroContext` serialization stub, attribute arg parsing helpers. |
| `glyim-compiler` | Updated `macro_expand` to use attribute args and reflection data. |
| `glyim-parse` | Parse attribute arguments (`key=value`), parse `quote!{}`/`${}` as unknown-but-valid tokens. |
| `glyim-typeck` | Implement `serialize_type_info` on the type checker. |
| `glyim-cli` | Extend `macro-inspect` with interactive stepping. |
| `glyim-diag` | Support for simple macro‑expansion error messages with invocation location. |

## Roadmap (Revised Execution Order)

1. **Attribute Arguments** – parser and `MacroContext` integration.
2. **Type Info Serialization** – expose field/variant lists to macros.
3. **Enhanced Macro Executor** – multi‑macro loading, error trapping.
4. **`macro-inspect` Step‑by‑Step** – interactive CLI.
5. **`quote!` / `${}` Syntax Parsing** – token/parse acceptance.
6. **Basic Macro Error Messages** – attach invocation span.

## 60‑Second Test (Revised Acceptance)

```bash
# 1. Parse attribute arguments
$ cat > src/main.g << 'EOF'
@serde(rename_all = "camelCase")
struct User { name: String }
fn main() { 0 }
EOF
$ glyim check src/main.g
✓ No errors. Attribute args parsed.

# 2. Use a custom Wasm macro with type info
$ glyim macro-inspect --steps src/main.g
Step 1: @my_debug(User)
After @my_debug:
struct User { ... }
impl Debug for User { ... }  # generated by string macro using serialized fields

# 3. Interactive macro stepping
$ glyim macro-inspect --interactive src/main.g
Step 1/2: @serde(rename_all = "camelCase")
[e]xpand > e
Step 2/2: After expansion: struct User { name: String }
✓ Done.
```
