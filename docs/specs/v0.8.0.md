# Glyim v0.8.0 Architecture & Design Specification

## 1. Context & Scope

### What v0.1.0–v0.7.0 Delivered

| Version | Theme | Key Deliverables |
|---------|-------|-----------------|
| **v0.1.0** | Architectural Runway | Pipeline, CAS traits, MacroContext |
| **v0.2.0** | Feels Like a Real Language | `let`/`mut`, `if`/`else`, strings, Rowan |
| **v0.3.0** | Language Core | Structs, enums, generics, pattern matching, FFI |
| **v0.4.0** | Developer Experience | LSP skeleton, formatter, lint |
| **v0.5.0** | Ecosystem & Production | Package manager, stdlib, DWARF, cross-compilation |
| **v0.6.0** | Macros That Don't Suck | Quote/splice, macro stepper, rich errors, incremental expansion |
| **v0.7.0** | The IDE Experience | Salsa DB, go-to-def through macros, DAP, completion, rename |

### The Problem: Glyim is Powerful but Unsafe

After v0.7.0, Glyim has a world-class macro system, a world-class IDE, and a full ecosystem. But it is fundamentally **unsafe for production software**:

- **No memory safety:** `Vec::push` can OOM and abort. No destructors. Every `malloc` without a matching `free` leaks. Aliasing bugs cause undefined behavior.
- **No async I/O:** Every I/O operation blocks. There is no way to write a concurrent web server or a responsive GUI.
- **No error handling ergonomics:** `Result<T, E>` exists but there is no `?` operator. Every fallible call requires manual `match`.
- **No dynamic dispatch:** No trait objects. No heterogeneous collections. No plugin systems.
- **No WASM target:** Can't run Glyim in the browser or at the edge.

### The Solution: v0.8.0 — "Safe, Concurrent, Everywhere"

**Tagline:** *"Memory safe. Concurrent. Runs anywhere."*

This specification adds the three pillars that make a systems language viable for production: safe memory management (ownership and destructors), concurrent I/O (async/await with structured concurrency), and universal deployment (WASM).

### Scope of v0.8.0

**Included:**
- Ownership and Borrowing system with gradual adoption
- Destructors and RAII (`Drop` trait)
- Error Propagation with `?` operator and `From` trait
- Async/Await with Effect Types (`!IO`, `!Async`) and Structured Concurrency
- OS Threads and Channels
- WASM target (`wasm32-wasi` and `wasm32-unknown-unknown`)
- Trait Objects (`dyn Trait`) with vtables
- Ownership-Aware Macro Errors (mapping borrow errors to macro invocations)
- Mutex, RWLock, and Atomics
- Scoped Threads
- `#[no_std]` embedded support
- C Interop improvements

**Excluded:**
- Full algebraic effects (v0.9.0+)
- Green threads / lightweight tasks (v0.9.0+)
- Hot reloading (v0.9.0+)
- Comptime reflection (v0.9.0+)
- Parallel iterators / Rayon-style (v0.8.1)
- WASI Component Model (v0.9.0+)

## 2. Goals and Non-Goals (ASRs)

### Goals

| ID | Statement |
|----|-----------|
| **ASR-049** | Glyim enforces ownership and borrowing rules at compile time. Code without annotations still compiles but may use runtime checks for gradual adoption. |
| **ASR-050** | Every type can implement a `Drop` trait, and the compiler automatically inserts destructor calls at all scope exit points. |
| **ASR-051** | The `?` operator propagates errors concisely, automatically converting error types via the `From` trait. |
| **ASR-052** | Async functions declare effects (`!IO`, `!Async`), avoiding the colored function problem, and `TaskGroup` enforces structured concurrency. |
| **ASR-053** | Users can spawn OS threads, communicate via channels, and share state via `Mutex` and `RWLock`. |
| **ASR-054** | `glyim build --target wasm32-wasi` produces a WASM binary that runs in Wasmtime or any WASI-compatible runtime. |
| **ASR-055** | `dyn Trait` enables dynamic dispatch via fat pointers and vtables for heterogeneous collections and plugin systems. |
| **ASR-056** | Borrow checker errors inside macro-generated code map back to the macro invocation site, utilizing v0.6.0's `ExpansionMap`. |

### Non-Goals

| What | Why |
|------|-----|
| **Full algebraic effects** | Research territory; `!IO`/`!Async` are sufficient. v0.9.0+. |
| **Green threads** | Complex runtime. v0.9.0+. |
| **Hot reloading** | Requires deep runtime introspection. v0.9.0+. |
| **WASI Component Model** | Ecosystem not mature yet. v0.9.0+. |

## 3. The Design

### 3.1 Ownership and Borrowing System

#### 3.1.1 The Rules

Glyim adopts Rust's ownership model:

1. Each value has exactly one **owner**.
2. When the owner goes out of scope, the value is **dropped**.
3. Values can be **borrowed** immutably (`&T`) or mutably (`&mut T`).
4. At any time, you can have either **one mutable reference** or **any number of immutable references**.
5. References must always point to valid data.

#### 3.1.2 Gradual Adoption

To avoid the "borrow checker wall" that blocks Rust beginners, Glyim supports **gradual adoption**:

- **Annotated code** (with `&`, `&mut`, lifetimes): Enforced at compile time, zero-cost.
- **Unannotated code** (plain `let`): Compiles with **runtime borrow checks** (similar to `RefCell<T>` in Rust). If a rule is violated at runtime, the program panics with a clear trace.

```glyim
// Annotated: compile-time enforcement
fn annotated(data: &mut Vec<i64>) {
    data.push(1);
}

// Unannotated: runtime enforcement
fn unannotated(data: Vec<i64>) {
    // data is moved here; caller no longer owns it
    // Runtime tracks the borrow state
}
```

The compiler flag `--strict` disables runtime checks and requires full annotation, ensuring zero-cost abstractions for production builds.

#### 3.1.3 Borrow Checker Implementation

A new pass is added to `glyim-typeck`:

```rust
pub struct BorrowChecker {
    // Maps each variable to its borrow state
    borrow_states: HashMap<Symbol, BorrowState>,
}

pub enum BorrowState {
    Owned,
    ImmutableBorrowed(usize),  // Number of active immutable borrows
    MutableBorrowed,
    Moved,
}
```

The borrow checker walks the HIR and validates each assignment, function call, and reference creation against the ownership rules.

### 3.2 Destructors and RAII

#### 3.2.1 The `Drop` Trait

```glyim
pub trait Drop {
    fn drop(&mut self);
}
```

When a type implements `Drop`, the compiler automatically inserts a call to `drop` at every scope exit point.

#### 3.2.2 LLVM IR Generation

```llvm
; Glyim source:
; {
;   let f = File::open("data.txt")
;   let x = 42
; }

; Generated LLVM IR:
define i32 @main() {
entry:
  %f = alloca %File
  call void @File_open(%File* %f, i8* getelementptr("data.txt"))
  %x = alloca i64
  store i64 42, i64* %x
  call void @File_drop(%File* %f)  ; Inserted by compiler
  ret i32 0
}
```

For blocks with early returns (`if`/`else`), the compiler inserts `drop` calls in *every* branch:

```llvm
  br i1 %cond, label %then, label %else

then:
  call void @File_drop(%File* %f)  ; Drop before early return
  ret i32 1

else:
  br label %merge

merge:
  call void @File_drop(%File* %f)  ; Drop at normal exit
  ret i32 0
```

**Drop order:** Fields are dropped in declaration order. Local variables are dropped in reverse declaration order (same as Rust and C++).

### 3.3 Error Propagation with `?` Operator

#### 3.3.1 The `?` Operator

```glyim
fn read_config(path: &str) -> Result<Config, AppError> {
    let text = fs::read_to_string(path)?     // ? propagates Err automatically
    let config = parse_config(&text)?        // ? propagates Err automatically
    Ok(config)
}
```

Desugars to:

```glyim
fn read_config(path: &str) -> Result<Config, AppError> {
    let text = match fs::read_to_string(path) {
        Ok(v) => v,
        Err(e) => return Err(From::from(e)),
    };
    let config = match parse_config(&text) {
        Ok(v) => v,
        Err(e) => return Err(From::from(e)),
    };
    Ok(config)
}
```

#### 3.3.2 The `From` Trait

```glyim
pub trait From<T> {
    fn from(value: T) -> Self;
}
```

Automatically implemented for `Result<T, E>` when `E: From<OtherE>`, allowing the `?` operator to convert error types on the fly.

### 3.4 Async/Await with Effect Types

#### 3.4.1 Avoiding the Colored Function Problem

Instead of Rust's colored functions (`async fn` vs `fn`), Glyim uses **effect types**:

```glyim
fn fetch_url(url: &str) -> String !IO {
    // This function performs I/O (declared with !IO)
    let response = http::get(url)    // Also !IO
    response.body
}

fn process(data: String) -> Result<Output, ParseError> {
    // This function is pure (no effects declared)
    parse(data)
}
```

**Key principles:**

1. Functions declare their effects (`!IO`, `!Async`, `!Panic`).
2. Callers must handle effects (propagate or suppress).
3. `async` is syntactic sugar for a function returning a `Future`.

```glyim
async fn fetch_url(url: &str) -> String !IO {
    // Desugars to:
    // fn fetch_url(url: &str) -> impl Future<Output = String> !IO { ... }
    let response = http::get(url).await
    response.body
}
```

#### 3.4.2 Structured Concurrency

```glyim
fn main() !IO {
    let results = TaskGroup::new(|tg| {
        tg.spawn(fetch_url("https://api1.example.com"))
        tg.spawn(fetch_url("https://api2.example.com"))
        tg.spawn(fetch_url("https://api3.example.com"))
    })
    // All tasks complete before TaskGroup exits (structured)
    
    for result in results {
        println(result)
    }
}
```

If any task panics, the `TaskGroup` cancels the remaining tasks and propagates the panic.

#### 3.4.3 Async Executor

The runtime is provided by `glyim-std::runtime`:

- **Executor:** A work-stealing thread pool.
- **Reactor:** `epoll` (Linux), `kqueue` (macOS), `IOCP` (Windows), and WASI poll for WASM.
- **Timer:** For timeouts and intervals.

### 3.5 OS Threads and Channels

#### 3.5.1 Thread Spawning

```glyim
let handle = thread::spawn(|| {
    expensive_computation()
})
let result = handle.join()
```

#### 3.5.2 Channels

```glyim
let (tx, rx) = channel::unbounded()

thread::spawn(move || {
    tx.send("hello from thread 1")
})

println(rx.recv())  // "hello from thread 1"
```

#### 3.5.3 Scoped Threads

```glyim
let mut data = vec![1, 2, 3, 4, 5];

thread::scope(|s| {
    s.spawn(|| {
        // Can borrow `data` because the scope waits for all threads
        println(data[0]);
    });
    s.spawn(|| {
        println(data[1]);
    });
}); // All threads complete here; data is no longer borrowed
```

#### 3.5.4 Mutex and RWLock

```glyim
let counter = Mutex::new(0)

thread::scope(|s| {
    for _ in 0..10 {
        s.spawn(|| {
            let mut guard = counter.lock()
            *guard += 1
        });
    }
})

println(*counter.lock())  // 10
```

### 3.6 WASM Target

#### 3.6.1 Target Triples

```bash
glyim build --target wasm32-wasi               # WASI (server-side WASM)
glyim build --target wasm32-unknown-unknown     # Browser WASM
```

#### 3.6.2 WASM-Specific Considerations

1. **No OS threads** (WASM is single-threaded; `SharedArrayBuffer` is deferred).
2. **No direct file I/O** (WASI provides file access).
3. **No raw pointers** (WASM memory is linear but bounds-checked).
4. **No DWARF** (WASM uses its own debug format).

#### 3.6.3 WASI Polyfill for Standard Library

`glyim-std` provides WASM-compatible I/O via WASI:

```glyim
// glyim-std/io/wasi.glyim

fn read_to_string(path: &str) -> Result<String, IoError> !IO {
    // Uses WASI fd_read under the hood
}
```

#### 3.6.4 `#[no_std]` for Embedded and WASM

```glyim
#![no_std]

fn main() -> i64 {
    42  // No allocator, no runtime, bare metal
}
```

### 3.7 Trait Objects (`dyn Trait`)

#### 3.7.1 Dynamic Dispatch

```glyim
trait Shape {
    fn area(&self) -> f64
}

fn print_area(shape: &dyn Shape) {
    println(shape.area())
}

fn main() {
    let shapes: Vec<dyn Shape> = vec![
        Circle { radius: 5.0 },
        Rectangle { width: 3.0, height: 4.0 },
    ]
    for shape in &shapes {
        println(shape.area())
    }
}
```

#### 3.7.2 Fat Pointers

`dyn Trait` is a fat pointer: `{ void* data, void* vtable }`

The vtable contains:
- Pointers to each method
- Size, alignment, and drop function

```llvm
%ShapeVTable = type {
    i64 (%Shape*)*,       ; area
    void (%Shape*)*,      ; drop
    i64,                  ; size
    i64                   ; align
}

%ShapeFatPtr = type {
    i8*,                  ; data
    %ShapeVTable*         ; vtable
}
```

### 3.8 Ownership-Aware Macro Errors

#### 3.8.1 The Problem

When a macro generates code that the borrow checker rejects, the error points to the generated code, not the macro invocation. This is Rust's #1 macro pain point.

#### 3.8.2 The Solution

Glyim leverages v0.6.0's `ExpansionMap` to map borrow checker errors back to macro invocations:

```
error: cannot borrow `user` as mutable because it is also borrowed as immutable
  --> src/main.xyz:5:1 (macro invocation)
   |
 5 | @derive(CustomMutate)
   | ^^^^^^^^^^^^^^^^^^^^^ macro invoked here
   |
   = note: the macro generated code that borrows `user` immutably
           while also borrowing it mutably
   = note: this is a bug in the `CustomMutate` macro, not your code
  --> <CustomMutate macro> line 12 (internal)
   |
   = error: mutable borrow of `user.name` conflicts with existing immutable borrow
   |
help: the macro author should ensure borrows don't overlap
```

## 4. Alternatives Considered

### 4.1 Ownership: Strict vs. Gradual

- **Alternative A (Strict, like Rust):** Full borrow checking at compile time. No exceptions. Zero-cost but steep learning curve.
- **Alternative B (Gradual):** Unannotated code uses runtime checks. Annotated code is checked at compile time. `--strict` mode enforces compile-time checking everywhere.
- **Decision (Chosen: B — Gradual).** The borrow checker wall is Rust's biggest adoption barrier. Gradual adoption lets users start quickly and tighten the screws incrementally. `--strict` ensures production builds are zero-cost.
- *Traceability: ASR-049.*

### 4.2 Async: Colored Functions vs. Effects

- **Alternative A (Rust-style colored functions):** `async fn` returns a `Future`. Sync functions can't await. Requires an executor to bridge.
- **Alternative B (Effect-typed async):** Functions declare `!IO`/`!Async`. Callers propagate effects. No red/blue distinction.
- **Decision (Chosen: B — Effect Types).** The colored function problem is Rust async's #1 complaint. Effect types allow pure functions to call effectful ones by propagating the effect.
- *Traceability: ASR-052.*

### 4.3 Error Handling: `?` vs. Exceptions

- **Alternative A (Rust-style `?`):** Explicit, typed, zero-cost.
- **Alternative B (Swift-style typed throws):** More ergonomic but requires runtime unwinding.
- **Decision (Chosen: A — `?` Operator).** Every modern systems language is converging on explicit, typed error propagation. The `?` operator is the sweet spot of ergonomics and zero-cost.
- *Traceability: ASR-051.*

### 4.4 Trait Objects: Fat Pointers vs. Type Erasure

- **Alternative A (Rust-style fat pointers):** `dyn Trait` is a `{ data, vtable }` pair. Explicit, well-understood.
- **Alternative B (Go-style implicit interfaces):** All interfaces are dynamic. Simpler but always pays the cost.
- **Decision (Chosen: A — Fat Pointers).** Systems languages should only pay for dynamic dispatch when explicitly requested.
- *Traceability: ASR-055.*

## 5. Cross-Cutting Concerns

### 5.1 Runtime Size for WASM

The Glyim runtime (executor, reactor, allocator) adds ~50KB to WASM binaries. For `#[no_std]` builds, the runtime is excluded, and the binary can be < 10KB.

### 5.2 Async Executor Design

The default executor is a multi-threaded work-stealing pool. For WASM and embedded targets, a single-threaded executor is provided.

### 5.3 Pin API for Self-Referential Types

Async state machines often contain self-referential pointers. The `Pin<P>` wrapper prevents the pointed-to data from being moved, ensuring safety:

```glyim
struct SelfRef {
    data: i64,
    pointer: *const i64,  // Points to `data`
}

fn use_self_ref(s: Pin<&mut SelfRef>) {
    // Safe because `Pin` guarantees `s` won't be moved
}
```

### 5.4 C Interop

`extern "C"` blocks allow calling C functions safely:

```glyim
extern "C" {
    fn malloc(size: usize) -> *mut u8
    fn free(ptr: *mut u8)
}
```

`#[repr(C)]` ensures Glyim structs have C-compatible layouts.

## 6. Architecture Decision Records

### ADR-034: Gradual Ownership and Borrowing

- **Context:** Rust's strict borrow checker is the #1 adoption barrier. Go's garbage collector is the #1 performance barrier.
- **Decision:** Implement gradual ownership: unannotated code uses runtime checks, annotated code uses compile-time checks. `--strict` disables runtime checks.
- **Consequences:** + Easier onboarding. + Zero-cost for production. – Runtime checks add overhead in development. – Two "modes" to test.

### ADR-035: Effect-Typed Async

- **Context:** Rust's colored functions (async vs sync) are a major pain point.
- **Decision:** Functions declare effects (`!IO`, `!Async`). Callers propagate effects. No colored function problem.
- **Consequences:** + Better ergonomics. + Pure functions can call effectful ones. – Effect inference adds compiler complexity. – Runtime must track effects.

### ADR-036: WASM/WASI as Tier 1 Target

- **Context:** WASM is critical for modern deployment (browser, edge, serverless).
- **Decision:** Add `wasm32-wasi` and `wasm32-unknown-unknown` as Tier 1 targets. Provide WASI-compatible stdlib.
- **Consequences:** + Broad deployment options. + WASI ecosystem compatibility. – No threads in WASM (yet). – WASM debug format differs from DWARF.

### ADR-037: Fat Pointer Trait Objects

- **Context:** Dynamic dispatch is needed for plugin systems and heterogeneous collections.
- **Decision:** `dyn Trait` uses fat pointers (`{ data, vtable }`), following Rust's model.
- **Consequences:** + Opt-in dynamic dispatch. + Well-understood ABI. – Fat pointers are 2× pointer size. – VTable generation adds codegen complexity.

## 8. Behavioral Specification (BDD)

### 8.1 Ownership and Borrowing

```gherkin
Feature: Ownership and Borrowing

  Scenario: Compile-time borrow check
    Given the source:
      """
      fn main() {
        let mut v = Vec::new()
        v.push(1)
        let first = &v[0]
        v.push(2)
        println(first)
      }
      """
    When "glyim check --strict" is run
    Then an error is reported: "cannot borrow `v` as mutable because it is also borrowed as immutable"

  Scenario: Runtime borrow check
    Given the same source without --strict
    When the program is run
    Then the program panics at runtime: "already mutably borrowed"

  Scenario: Ownership transfer
    Given the source:
      """
      fn take(v: Vec<i64>) { }

        let v = Vec::new()
        take(v)
        println(v.len())  // Use after move
      }

    Then an error is reported: "use of moved value: `v`"
```

## 10. Execution Roadmap

**Phase 1: Ownership & Borrowing**
1. Implement `BorrowChecker` pass in `glyim-typeck`

8. Parse `?` operator

**Phase 4: Trait Objects**
11. Add `dyn Trait` syntax
12. Generate vtables in codegen
13. Implement fat pointer ABI
