# Glyim v0.9.0 Specification Suite: The Onboarding Release

This document suite constitutes the fully fledged specification for Glyim v0.9.0, applying evidence-based requirements engineering, architecture documentation, and behavioral specification practices. It is structured across five specification levels to ensure strategic alignment, stakeholder satisfaction, technical feasibility, and verifiable delivery.

---

## Document 1: Vision & Strategic Alignment (Level 0)

**Document ID:** GLYIM-V090-VISION-001  
**Version:** 1.0  
**Status:** Approved  
**Author:** Glyim Core Team  

### 1.1 Vision & Elevator Pitch
**Vision:** To make Glyim the most accessible systems language in the world, ensuring any developer can learn it in an afternoon and love it for a lifetime.  
**Elevator Pitch (Moore Template):** For systems programmers who are dissatisfied with Rust's steep learning curve, C++'s safety deficits, and Go's lack of expressive generics, Glyim is a modern, safe, and concurrent systems language. Unlike its competitors, Glyim pairs world-class compiler technology with an intuitive, narrative-driven onboarding experience—The Glyim Book, interactive examples, and a zero-install browser playground—eliminating the barrier to entry.

### 1.2 Problem & Business Context
Glyim v0.8.0 delivered a complete, production-ready systems language (ownership, macros, async, WASM). However, empirical evidence from language adoption (e.g., Rust's success tied to "The Book") shows that compiler completeness is necessary but insufficient for adoption. The current state is a "ghost town" ecosystem: no narrative docs, empty API references, no migration paths, and a >30s compile-time friction for first-time users. Without v0.9.0, Glyim cannot capture early-majority developers who require low-friction onboarding.

### 1.3 Target Users (Target Group)
1. **The Curious Explorer:** Developer who wants to try Glyim in <30 seconds without installing a toolchain.
2. **The System Migrant:** Rust/Go/C++ developer looking for mental-model mapping guides.
3. **The Macro Author:** Advanced user seeking to learn Glyim's differentiating macro system without reading source code.
4. **The Ecosystem Builder:** Early adopter looking for essential crates (HTTP, CLI) to build production apps.

### 1.4 Desired Outcomes & Success Metrics (OKRs)
* **Objective 1: Eliminate the first-run friction.**
  * **KR 1:** 90% of users complete a "Hello World" in the browser playground in under 60 seconds.
  * **KR 2:** Playground maintains >95% uptime and <5s cold compile time for standard examples.
* **Objective 2: Establish Glyim as a learnable language.**
  * **KR 1:** "The Glyim Book" reaches 10,000 unique readers within 3 months of release.
  * **KR 2:** Time to complete "Chapter 2: Guessing Game" averages <15 minutes for a target user test group.
* **Objective 3: Prepare the ecosystem for growth.**
  * **KR 1:** 100% of stdlib public APIs have doc comments with runnable examples by v0.9.0 ship date.
  * **KR 2:** Registry contains at least 5 "essential" packages (HTTP, logging, CLI, serde, testing).

### 1.5 Goals and Non-Goals (Scope / Anti-scope)
**Goals (P0/P1):**
* Ship "The Glyim Book" (narrative tutorial).
* Ship "Glyim by Example" (searchable, runnable snippets).
* Implement zero-install Browser Playground via WASM.
* Define and implement the Glyim 2026 Edition.
* Document 100% of stdlib API.
* Publish Migration Guides for Rust, Go, and C++.

**Non-Goals (Anti-scope):**
* **No new compiler features:** v0.9.0 is strictly polish, docs, and tooling. No new syntax or type system changes (except edition boundaries).
* **No visual IDE extensions:** We provide the LSP protocol; building VS Code extensions is deferred.
* **No official video courses:** Community-driven video content is P2; v0.9.0 delivers text/interactive docs only.
* **No green threads:** Async/await is the concurrency model; Go-style goroutines are explicitly out of scope.

## Document 3: Software Requirements Specification (Level 2)

**Document ID:** GLYIM-V090-SRS-001  
**Traceability:** Maps to Level 1 SN-001 through SN-004 and Level 0 OKRs.

### 3.1 Functional Requirements: Documentation Ecosystem

**FR-DOC-001: The Glyim Book**
* **Description:** The system shall provide a unified, free, web-hosted narrative tutorial named "The Glyim Programming Language".
* **Fit Criterion:** The book shall contain at least 20 chapters, progressing from "Hello World" to "Building an Async TCP Server". Chapter 2 must be a guided project (e.g., a guessing game or CLI tool). Every code block must be syntactically valid for the `Glyim 2026` edition.

**FR-DOC-002: Glyim by Example**
* **Description:** The system shall provide a searchable, page-per-concept documentation site.
* **Fit Criterion:** The site must cover at least 50 core concepts (variables, control flow, structs, enums, pattern matching, traits, macros, async). Each page must contain exactly one primary runnable code example (max 30 lines) and output.

**FR-DOC-003: Stdlib API Documentation**
* **Description:** The `glyim doc` CLI tool shall generate HTML documentation from source code doc comments.
* **Fit Criterion:** 100% of public `pub fn`, `pub struct`, `pub trait`, and `pub enum` items in the `core`, `alloc`, and `std` crates must have a `///` or `//!` doc comment containing at least one code example. Generated HTML must include a search bar returning results in <100ms.

**FR-DOC-004: Migration Guides**
* **Description:** The documentation site shall host dedicated "Coming from X" guides for Rust, C++, and Go.
* **Fit Criterion:** Each guide must contain at least 10 side-by-side comparisons of common tasks (e.g., error handling, string manipulation, concurrency), highlighting syntactic and semantic differences.

### 3.2 Functional Requirements: Tooling & Infrastructure

**FR-TOOL-001: Browser Playground**
* **Description:** A web application allowing users to write, compile, and execute Glyim code in the browser without local installation.
* **Fit Criterion:** The frontend shall use Monaco Editor. The backend shall compile to WASM. Users must be able to share code via URL parameters (base64 encoded or shortened link). Execution sandbox must prevent infinite loops (via WASM fuel/interruptions).

**FR-TOOL-002: Edition System**
* **Description:** The compiler shall support an "Edition" manifest field (`glyim.edition = "2026"`) in `glyim.toml`.
* **Fit Criterion:** Code compiled without an explicit edition defaults to `Glyim 2024` (v0.8.0 semantics). Code compiled with `Glyim 2026` opts into new syntax/semantics defined for v0.9.0. The compiler must reject code that mixes editions across a single crate boundary.

**FR-TOOL-003: Edition Migration Linter**
* **Description:** The compiler shall provide a CLI subcommand `glyim fix --edition 2026` to automatically migrate a codebase.
* **Fit Criterion:** The tool must safely rewrite syntax (e.g., updating keyword usage) and output a diff. It must not delete comments or change formatting unless related to the edition change.

### 3.3 Non-Functional Requirements (NFRs)

**NFR-PERF-001 (Playground Latency):**
* Under normal load (100 concurrent users), the Playground shall compile and execute a standard 50-line Glyim program within **5 seconds** (p95), measured at the browser boundary.

**NFR-SEC-001 (Playground Isolation):**
* Playground WASM execution shall be strictly sandboxed. It shall have no access to the host network, file system, or `localStorage`. Memory usage shall be capped at 256MB per execution.

**NFR-SEO-001 (Discoverability):**
* All documentation HTML output must conform to Google's Core Web Vitals (LCP < 2.5s, FID < 100ms, CLS < 0.1) and include structured data markup (JSON-LD) for tutorials to ensure search engine indexability.

## Document 4: Architecture & Design Specification (Level 3)

**Document ID:** GLYIM-V090-ARCH-001  

### 4.1 Context & System Boundary (C4 Level 1)
The v0.9.0 release introduces two primary new bounded contexts to the existing compiler ecosystem:
1. **Docs Ecosystem:** Static site generation pipeline.
2. **Playground Context:** Client-side WASM execution environment.
*(Note: The core compiler context remains unchanged from v0.8.0).*

### 4.2 Architecture Decision Records (ADRs)

**ADR-009: Playground Execution via WASM Threads + Fuel**
* **Context:** We need safe, fast execution of Glyim code in the browser. Raw WASM can hang the browser tab (infinite loops).
* **Decision:** Compile the Glyim compiler to WASM. Execute generated code on a Web Worker using WASM threads. Implement a "Fuel" mechanism (instruction counter) that aborts execution after $N$ instructions.
* **Consequences:** 
  * *(+)* Zero server cost; highly scalable.
  * *(+)* Complete isolation from host OS.
  * *(-)* WASM threading support is inconsistent in older mobile browsers (requires fallback to main thread with `setTimeout` yielding).
  * *(-)* Fuel limit is arbitrary; complex but valid programs might hit it.

**ADR-010: Static Site Generation for Docs (mdBook fork)**
* **Context:** We need a unified look for The Book, By Example, and Stdlib Docs. Rust uses `mdBook`.
* **Decision:** Fork `mdBook` to create `glyim-book`. It will parse standard Markdown but use a custom Glyim syntax highlighter (leveraging the existing Tree-sitter grammar) and a unified search index (MiniSearch).
* **Consequences:**
  * *(+)* Familiar UX for Rust migrants.
  * *(+)* Full control over theming and search.
  * *(-)* Maintenance burden of a forked repo.

**ADR-011: Editions as AST-level Feature Gates**
* **Context:** How to implement editions without duplicating the compiler.
* **Decision:** Introduce an `Edition` enum in the `Session` globals. The parser will check this enum when encountering ambiguous syntax (e.g., a new keyword in 2026 that is an identifier in 2024) and route to the correct AST node type.
* **Consequences:**
  * *(+)* Single compiler binary supports all editions.
  * *(-)* Parser complexity increases; "edition bridges" are required when a 2024 crate imports a 2026 crate.

### 4.3 Container View (C4 Level 2): Playground Architecture

```text
+-------------------+       +---------------------------------------+
|   User Browser    |       |            CDN / Static Host          |
|                   |       |  +-------------+    +---------------+ |
| +---------------+ | HTTP  |  | glyim-play |    | glyim-book    | |
| | Monaco Editor |<---------->| ground.js  |    | (The Book)    | |
| +---------------+ |       |  +-------------+    +---------------+ |
| +---------------+ |       +---------------------------------------+
| | Web Worker    | |
| | +-----------+ | | WASM  
| | | Glyim WASM|<---------+
| | +-----------+ | |
| +---------------+ |
+-------------------+
```
* **Frontend:** React/Next.js application hosting Monaco.
* **Worker:** A dedicated Web Worker thread downloads `glyim_compiler.wasm` (~15MB initial, ~5MB cached/gzipped).
* **State:** Editor text is serialized, sent to Worker via `postMessage`. Worker returns stdout/stderr or Fuel-exceeded error.

## Document 5: Behavioral Specification & Test Verification (Level 4/5)

**Document ID:** GLYIM-V090-BEHAV-001  
**Verification Method:** Automated Test (Playwright for UI, Unit for Compiler, Integration for WASM).

### 5.1 Specification by Example (SbE): Playground Critical Path

**Feature:** Zero-Install Hello World Execution

```gherkin
Scenario: User executes a valid Glyim program successfully
  Given the user opens the Glyim Playground in a modern browser
  And the WASM compiler has finished loading (splash screen hidden)
  When the user types "fn main() { println!("Hello, Glyim!"); }" into the editor
  And the user clicks the "Run" button
  Then the output pane displays "Hello, Glyim!" within 5 seconds
  And no browser alerts or tab freezes occur

Scenario: User executes an infinite loop (Fuel protection)
  Given the Playground is loaded
  When the user types "fn main() { loop {} }" into the editor
  And the user clicks the "Run" button
  Then the output pane displays an error message containing "Execution limit exceeded"
  And the browser tab remains responsive to user input
```

### 5.2 Specification by Example: Edition Migration

**Feature:** Automated Edition Fixing

```gherkin
Scenario: Migrating a project from Glyim 2024 to 2026
  Given a valid Glyim 2024 project directory containing "glyim.toml" with edition = "2024"
  And a file "src/main.gy" containing deprecated 2024 syntax
  When the developer runs "glyim fix --edition 2026"
  Then the CLI exits with code 0
  And "glyim.toml" is updated to edition = "2026"
  And "src/main.gy" is rewritten to use valid 2026 syntax
  And a diff is printed to stdout showing only the required syntactic changes
```

### 5.4 Test Strategy & Living Documentation
* **Stdlib Doc Coverage:** A CI gate will fail if `glyim doc --check` finds any public symbol missing a `///` doc comment containing a code block. This ensures the documentation never rots below 100%.
* **Playground E2E:** Playwright tests run on every PR to the `playground` repo, testing compilation, execution, sharing (URL generation), and cross-browser compatibility (Chrome, Firefox, Safari).
* **Edition Linter:** The `glyim fix` tool is tested via snapshot testing. A corpus of v0.8.0 codebases is stored in `test_fixtures/`; the tool is run, and the resulting AST/diff is compared against checked-in `.expected` files.
